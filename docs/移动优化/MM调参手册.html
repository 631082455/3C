调了半年 Motion Matching，最大的感受不是参数难，是参数散。同一个"起步不对"，病根可能在库的行序、可能在采样范围、可能在驱动变量、也可能在轨迹预测，每次都要重新想一遍去哪找。这本手册就是把这半年翻过的地方一次性收编成查表，以后自己不用再考古，接手的人和 AI 也不用。

写法上有一条底线：不背旧教程。PoseSearch 这套插件在 UE 各版本间变动很大（5.7 删了 MirrorMismatchCostBias、废了 bShouldSearch 引脚、条目结构整个换了），网上资料大多停在 5.3 到 5.5。所以全部参数对着项目引擎的源码逐行核过，项目资产的现值也是编辑器里实读的，每条都标了证据档位，源码、项目、实战、推断分开写，不混。

一次匹配怎么发生

Motion Matching 每帧做一件事：拿"我现在的姿势 + 我接下来要去哪"当查询，去候选库里找 cost 最低的一帧动画跳过去播。所有调参都是在改这句话里的某个环节。项目里这条链共六层：

cost 怎么算【源码】

一帧候选的总代价四项相加（PoseSearchCost.h）：

TotalCost = 特征差异（实时算）+ 帧级烘焙偏置 + 续接偏置 + 交互偏置（未用）

  • 特征差异 = Σ(（候选特征 − 查询特征） × 权重平方根）²。权重在 Schema 层定义，索引期烘焙。
  • 帧级烘焙偏置：起始值 = 库的 BaseCostBias → 若该帧被 ModifyCost 标记覆盖则替换（不是叠加）→ 可循环资产再加 LoopingCostBias。索引期烘进每帧元数据，半精度存储。
  • 续接偏置：只有"继续播当前这条"的那次评估会加 ContinuingPoseCostBias（负值 = 倾向不换）。运行期读取，改了立即生效。

候选在算 cost 之前还要过三道黑名单：BlockTransition 标记（可顺播不可跳入）、同段邻域禁跳（PoseJumpThresholdTime）、近期已选禁重选（PoseReselectHistory）。另有一条整库跳过优化：已有结果的 cost 低于某库的理论下限时，那个库整个不搜。

病在哪层去哪层

选错动画：先查路由（08）和数据（02），再查轨迹（07）和 Schema 权重（03），bias 是最后的微调不是首选。选对了但过渡难看：混合（05）与修饰（06）。选对了但脚下滑：修饰层与数据速度对齐（06/02）。粘住不换或换太勤：bias 与重选抑制（02/05）。别上来就调 bias，它是天平上的小砝码，掩盖问题比解决问题快。

数据库层（PSDB）

数据库是候选帧的容器，也是三个 bias 的家。先记一条纪律再看参数：动画原生速度必须覆盖角色移速档。本项目 Sprint 战役的根因就是角色 650 对动画原生 756【实战】，这种失配任何 bias 都救不了，只能靠数据对齐或 PlayRate 吸收（见 05）。

Schema 层

Schema 定义"像不像"这三个字：拿哪些特征、各占多重。调它之前先吃透权重管线的三条机制【源码】，不然会调出反效果：

权重三机制

一，权重是相对量：Normalize 系预处理会把全向量权重归一，调大轨迹权重等于自动稀释其余所有通道。二，权重是索引期烘焙：改了必须等索引重建才生效。三，权重对该通道 cost 贡献是线性放大（×2 权重 = ×2 误差贡献），且在 PCA 投影前就乘入，同样影响近似搜索阶段。

项目实配（PSS_Default，实读）【项目】

只有两个通道，比想象中朴素得多。SampleRate = 15（非引擎默认 30），预处理 = NormalizeWithCommonSchema（实验模式）。注意一个流传甚广的说法要修正：引擎构造器里 Trajectory 通道默认权重确实是 7（Pose 的 7 倍），但 GASP 把它改回了 1，靠逐样本权重精调：

动画标记层

标记是加在动画资产时间轴上的帧级修饰，索引期被烘进库（除了一个例外）。最常用的几件经常被混为一谈，先看一条动画被标记覆盖后，每一帧到底处于什么状态：

标记级效果典型用途
Exclude From DatabaseP1覆盖段完全不进索引：搜不到，也播不到动画里的废段、起手前摇里不该被匹配的部分
Block Transition InP1帧仍在库里、可以被顺播进入，但搜索永不把它当跳转落点。PCAKDTree 下这些帧构建时就剔出了 kdtree【源码】"这段只能从前面播进来，不许半路空降"：起步后半段、落地恢复段
Override Base Cost Bias（类名 ModifyCost）P1帧级 cost 偏置，默认 -1。【源码】对库级 BaseCostBias 是覆盖不是叠加；烘焙为半精度，0.001 量级的差异会被精度吃掉重点段拉偏好（如折返的关键帧段）

GASP 的六条 MM 曲线

GASP 每条移动动画带一组名字像参数的曲线：FootSpeed_L/R、Phase、MoveData_Speed、Enable_OrientationWarping、Enable_PlayRateWarping（Sprint_Loop_F 实测带前五条【项目】）。它们不进匹配：项目 Schema 没有曲线通道，权重管线碰不到它们；消费方是蓝图侧的 warping 开关与 FootPlacement。项目此前烘的重定向动画六条全缺，经源码逐条核实定性为纯装饰级损失（FootPlacement 退化为近直通、无故障）【实战】；但镜头贴近脚部、上斜坡地形之前要补烘，修复用 GASP 自带的 AnimModifier（注意脚骨名要重指向）。

P3 列名：实验与交互类标记
  • Sampling Event / Sampling Attribute：给通道打事件锚点或替换采样数据源（配 TimeToEvent 通道用，项目未用）。
  • IKWindow：多角色交互 IK 约束窗口（实验）。
  • Pose Search: Event（UAnimNotify_PoseSearchEvent）：GameplayTag 事件直达搜索（实验，EventOnly 模式的唯一入口）。

动画蓝图层

项目的 MM 节点在共享动画蓝图的 AnimGraph 里，结构是"蓝图每帧喂库 + 节点搜索"：节点的 UpdateFunction 指向 Update_MotionMatching 图，那里评估 Root Chooser 后 SetDatabasesToSearch【项目】。节点本身就是一个混合栈（继承 BlendStack），不是外接【源码】。

  • BlendParametersDeltaThreshold / BlendspaceUpdateMode:BlendSpace 条目专用，项目库以 Sequence 为主，列名。
  • 同步组四件套（GroupName/GroupRole/Method/bOverridePositionWhenJoiningSyncGroupAsLeader)：项目 ExclusiveAlwaysLeader + DoNotSync【项目】。
  • 实验列名：EventToSearch（事件搜索）、Stitch 三件（缝合库代替 crossfade）、交互 MM 全家（Availabilities/Warping*Ratio）、PlayRateMultiplier。
  • OnMotionMatchingStateUpdated 回调：项目挂了 OnStateExit【项目】，状态联动的口子。

轨迹层

查询向量的一半是姿势历史，另一半就是这条预测轨迹：它长什么形状，MM 就倾向选什么动画。项目走的是 GASP 标准做法：蓝图里 PoseSearchGenerateTransformTrajectory 生成，喂给 HistoryCollector 节点的轨迹引脚（默认采样：历史 0.04s×10 个、预测 0.2s×8 个，约看未来 1.6 秒）【源码】【项目】。

移动参数就是动画参数

预测积分是 CMC 刹车/加速逻辑的复刻：MaxSpeed（含 buff 与摇杆系数）、BrakingDeceleration、Friction 全部直接进轨迹形状【源码】。改移动组件的手感参数时，MM 的选择倾向会跟着变，反过来"动画怪"也可能是移动参数改了，两个工作台是同一个。

路由与驱动层

严格说这层不是 MM 的参数，是喂给 MM 的变量，但一半的"病"出在这。全是项目 GASP 层的东西，纯【实战】。

Chooser 路由

  • 行序即优先级：从上到下 first-match wins，条件填 Any 的兜底行必须垫底。Sprint 子表"巡航行（Any）排在起步行（True）之前被截胡"是 GASP 原始数据 bug，起步永远进不了起步库，修复只是挪一行【实战】。查路由走没走对，用 Rewind Debugger 的选择器评估轨道，别猜。
  • 层级：Root（按英雄 ID） → LOD 列（Dense/Sparse） → 四列条件（模式/姿态/起停/Gait） → 库。嵌套子表编辑器手动看，自动化工具读不进【实战】。
  • Dense 侧修过的行序，Sparse 侧没同步，切 Sparse 会重现修复前症状，改路由要两档一起【实战】。
