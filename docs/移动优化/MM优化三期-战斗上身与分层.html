总览与阶段索引

战斗姿态 Overlay 体系

MM 只负责"人往哪跑、腿怎么迈"。战斗中角色需要一层持续的姿态：举着枪、端着盾、搭着箭。这一层不能用蒙太奇硬顶（播完就没了），也不能焊死在移动数据里（脱战还得放下来）。三月下旬开始搭的就是这层：roaming 与 combat 两种身体状态，加一条逐英雄的 Overlay 姿态通道。

A1 · roaming-combat 状态定义与过渡（03-22，CL 4973642）

现象：Twinblast 普攻后仍是巡游姿态，举枪与收枪没有状态承载，过渡生硬。

方案：定义 roaming-combat 双状态。从提交文件清单看，这一步同时动了三层：主 ABP ABP_HeroCharacter_MM、Twinblast 的 Overlay ABP 与蒙太奇资产、以及 C++ ability 层四组文件（HeroAbility / HeroAbilityTags / HeroAbilityTypes），即战斗状态的定义进入了代码层而非只在蓝图里。

过程留痕：该 CL 里带着 _BrokenBackup_20260321、_Clean、_EnterTempV2、_ExitTemp 等六个中间资产与一个坏蒙太奇备份，说明进出战过渡当时反复试错、甚至改坏过一版 Overlay ABP 靠备份回退。这些中间产物随提交一并入了库。

机制链路（依据后续机制梳理记录，运行时全链）：技能 HAD 的 SkillModuleMontagePlay 模块配 bChangeCombatStateOnPlay / OnBlendOut（Add/Remove）；保存 HAD 时 PreSave 反写到蒙太奇资产的 UAbilityMontageData；运行时 HeroAbilitySystemComponent 在蒙太奇播放与混出时调 UHeroOverlayPose::Enter/ExitCombatState，脱战有 3 秒延迟（ExitCombatStateDelay），HeroAbility::EndAbility 兜底退出。Steel 一类英雄只在普攻上开进出战，其他技能不开。

A2 · Overlay 基类沉淀（03-25，CL 4997669）

Twinblast 单英雄跑通后，把可复用部分下沉：本次提交除 Twinblast 自身资产外，改动了 GASPALS 插件内的 ABP_OverlayPose_Base。此后逐英雄接入的标准形态是：子 Overlay ABP 继承该基类（原生父类为 C++ UHeroOverlayPose），经 ALI_OverlayPose 接口以 Linked Anim Layer 挂进主 ABP；HeroData.CombatOverlayPose 配上后由 MobaDataSubsystem 在英雄数据加载时自动 Link，主 ABP 层节点不需要逐英雄改。

A3 · Steel 接入与改造（03-31~04-01，CL 5031382 / 5039351 / 5042457）

现象：Steel 还挂着旧动画蓝图，无战斗姿态层。

方案与动作：新增 ABP_Overlay_SteelCombat_New；把 Paragon 的 combat idle 重定向出 rtgSteel_Idle_Combat（这份资产两个月后成为战役一节里 additive 基准动画之争的核心）；删除旧 Steel_AnimBlueprint；次日的 5039351 顺手带出新一批普攻动画 Steel_Attack_Melee_A/B_Normal_V1 及其 Recovery 与蒙太奇（这批资产的 additive 烘焙问题当时无人察觉，五月才现形），同时把 Muriel 的 HeroData / ABP / MCD / RTG 基础盘一并建了出来。

效果：Steel 进出战姿态走通，后续英雄照此模板铺开。

A4 · Muriel 与 Sevarog 铺开（04-03~04-09，CL 5061950 / 5095446）

Muriel 补上普攻姿态与射击资产（UEFN_Primary_Pose、快慢两档 Fire 动画）与飞行 Overlay；Sevarog 新增 ABP_Overlay_SevarogCombat_New 并整理基础移动动画与 AO。至此 Overlay 接入从"单英雄验证"进入"逐英雄例行公事"，每个英雄的接入面收敛为：一个子 Overlay ABP、一组战斗 idle 姿态动画、HeroData 一处引用。

这条线立住的原则：战斗姿态是持续状态，归 Overlay（Linked Anim Layer）管；瞬时动作归 Slot 蒙太奇管。两者语义分开，五月战役里的分层决策全部建立在这个前提上。

上身与下身的边界

Overlay 铺开后，问题转向边界骨骼：spine 让不让下半身参与？瞄准时 root 的朝向归谁？手指算上半身吗？五月上旬四个提交都在回答这类问题。这一段会话考古未覆盖，细节以提交说明与文件清单为准，不作超出证据的展开。

B1 · spine 参与后拧成麻花（05-08，CL 5237275）

现象：为了让上半身 Overlay 更跟手，尝试让下半身 spine 参与联动，结果角色躯干被拧成麻花。

方案：提交同时修改主 ABP ABP_HeroCharacter_MM、Twinblast Overlay ABP、AO 资产 Idle_AO_CC_UEFN 与瞄准移动姿态 Pose_Twinblast_Pistol2H_Stand_Aim_Move，在"更跟手"与"不拧麻花"之间同一天内调平。

效果：提交说明记为两个问题同时解决；这也是后来"上半身跟镜头其实是整身 Strafe ＋ Warping ＋ AO 的合成，不是脊柱独立旋转"这一认知（见战役一节）的前置实践。

B2 · Aiming 移动的 root rotation 释放（05-08，CL 5240454）

现象：Aiming 状态下移动，腿保持进入瞄准时的旧朝向，不随移动方向更新。

方案：释放 Aiming 移动时的 root rotation。文件清单为主 ABP 加 Twinblast Overlay 及其 Enter/Exit 过渡资产，即改动落在状态过渡链路上而非单个姿态。

效果：腿部朝向交还给移动系统。与 B1 合起来，当天定下的边界是：瞄准姿态管上半身与手臂，root 与腿归移动。

B3 · hand Overlay 覆盖到指头（05-13，CL 5267661）

现象：弓箭手（Sparrow）持弓与释放时，hand Overlay 覆盖不到手指骨，指头仍走底层动画。

方案：这是三期单次体量最大的资产提交之一：DrawOne / DrawThree / ShootOne / ShootThree 全套 UEFN 重定向动画，每套配九方向 AO（CC/CD/CU/LC/LD/LU/RC/RD/RU），加 MT 拆段与蒙太奇，同时改主 ABP 与 ABP_Overlay_SparrowBow 把覆盖范围扩到指骨。

效果：持弓、搭箭、释放三态在移动中成立，手型不再穿帮。

B4 · 弓箭手普攻抖动（05-14，CL 5274260）

现象：Sparrow 普攻（放箭）在移动中抖动。

方案：从资产命名判断，做法是在 UEFN_Sparrow_Attack_Release 的 0.36 秒处截出一段过渡动画 UEFN_Sparrow_Attack_Release_0p36_To_Bow_Aim_Move，让放箭动作在该点接回持弓移动姿态，而不是播完整段再硬切；配套改了 ACD / HAD / MCD 三份配置。

效果：提交说明记为抖动优化完成。值得注意的是五天后（05-19）UpperBody 抖动仍以 Sparrow 的 BladeShower_Start_Montage 为复现用例被重新提出，说明这次是治了普攻衔接这一处症状，蒙太奇随下半身晃动的共性根因当时尚未触及。

UpperBody 抖动根因战役

三期的重头戏。起点是一个描述性极强的体验问题，终点是一次架构级定稿加一场连锁事故。会话跨 05-19 至 06-03，394 条有效消息，全程留痕。

问题定义与复现（05-19 上午）

问题定义（整理稿，05-19）：UpperBody Slot 蒙太奇与 MM 移动叠加播放时，移动层的骨盆运动传导进上半身，角色整体晃动。复现示例：BladeShower_Start_Montage、UEFN_Steel_Ability_Shield_Hold_Montage。预期行为：移动中上半身保持稳定。

复现条件：主 ABP ABP_HeroCharacter_MM（GASPALS/GASP 模板迁移而来，Steel / Sevarog / Sparrow / Twinblast / Muriel / Phase / Gideon / Kallari 八英雄子类继承）；MM 移动中播 UpperBody Slot 蒙太奇，"静态 hold"类最明显。pelvis 走 MM 路径，跑步的摇摆与颠簸沿骨链带动 spine，slot 里写死的举盾、拉弓姿势无力补偿。

根因：一个被忽略的默认值（05-19 10:00~10:09）

AI 读 AnimGraph 后锁定 UpperBody 链的 Layered Blend Per Bone 节点实际配置：

  • LayerSetup = ((BranchFilters=((BoneName="spine_01")))），过滤起点 spine_01，pelvis 有意留给 MM，避免髋部僵死；
  • bBlendRootMotionBasedOnRootBone = True；
  • bMeshSpaceRotationBlend 未出现，即为 UE 默认值 False。这就是根因。

推理链：local-space 混合下，spine_01 的世界朝向等于 pelvis 世界朝向乘其局部旋转，LBPB 只覆盖局部旋转，pelvis 一晃整条 spine 链跟着抡；mesh-space 旋转混合则用 slot pose 的 component-space 朝向反算局部旋转，每帧动态抵消 pelvis 晃动，上身钉稳。佐证：Lyra 与 Game Animation Sample 的 UpperBody 层此项默认开启，本项目迁移时正好是关的。次要嫌疑 SpineRotation 的六节 ModifyBone 链被列出但排除：那是全时段影响，与"仅 slot 期间晃"的症状不符。

策划确认修复方向后先要求回滚："你的改动是正确的 不过我想改回去先 录个屏"，改前改后各录一版 A/B 对比再定稿。

体系化扩展：从修一个开关到定一套分层（05-19 11:17~12:45）

开关修完先做负面影响评估：失去上身随髋部甩动的 follow-through；只修旋转不修平移，跑步上下颠簸保留（通常是想要的）；大幅度 spine 动作的攻击蒙太奇可能变硬，处理方向是延长混合时间而不是回退 mesh-space。随后策划提出关键架构判断：

"upperbody Slot的用意就该如此吧，如果想保持一些下半身带来的传导 应该用OVERLAY Pose 或者其它？"
[05-19 11:21 策划原话]

由此定调分层心智模型：Slot 是瞬时强占语义，Overlay 是持续姿态语义，与第二节的 Overlay 体系互为表里。在此之上扩展两件事：

  • BlendMask 双档：BP_UpperBody_Hard（全 1.0，持盾/瞄准/蓄力）与 BP_UpperBody_Soft（spine 链渐变权重，挥砍类保留下身惯性传导）；
  • UpperBodyAdditive Slot：Additive Identity Pose 经 Slot 走 ApplyMeshSpaceAdditive，插在 PostUpperBody 与 DefaultSlot 之间，作受击 flinch、后坐力、呼吸类的叠加通道，与 Lyra/GASP 同构。

途中一次自我纠错：最初方案想靠蒙太奇上配 Blend Profile 换档，AI 随后主动收回该说法，因为蒙太奇上的 Blend Profile 只影响混入混出过渡、不影响稳态权重。改走平行双 Slot 方案：新增 UpperBodySoft slot 加第二条 LBPB 链，蒙太奇作者直接选 slot 名换档，零运行时逻辑。策划拍板"方案 A吧"。

被否与被救回的路径

  • 程序化创建 BlendProfile：失败。BlendProfiles 属性 protected 且无 UFUNCTION，AIK 与 Python 全路径走不通，只能编辑器 UI 手建，由策划手动操作、AI 看截图校对。
  • 删七个"没人用"的 Slot：被策划一句话救回。AI 首轮扫描认为 Curves / Arm R / Arm L / Head / Spine / Pelvis / Legs 七个 slot 无人引用可删，策划质疑"该不会是用到的吧"；补扫发现它们全部被 GASPALS ABP_LayerBlending（OverlaySystem 层混合中心，经 Linked Anim Layer 接入主 ABP）占用。七个全保留。AI 承认首次扫描只查了蒙太奇、没查 AnimBP，是偏见结论。

连锁事故：Steel 普攻不可见（05-19 13:35 → 05-20 03:11）

现象：主线修复入库前后，Steel 普攻动画两次完全消失。第一次是排查"combat 上半身不跟手"时给 rtgSteel_Idle_Combat 加了六条 Layering_* 曲线所致，回滚后恢复；第二次在回退 AO 绑定后再次消失。策划给出关键定位："HAD_SteelAttack 普攻都在这里面。其它半身动作是OK的，普攻完全没动作"。

真根因：Steel 四个普攻动画（Steel_Attack_Melee_A/B_Normal_V1 及 Recovery，即 A3 阶段 3 月底入库的那批）是 AAT_ROTATION_OFFSET_MESH_SPACE additive，但 RefPoseSeq=None。Paragon 重定向历史遗留的坏烘焙，此前一直被当绝对 pose 塞进 UpperBody slot 播。旧的 local-space ＋ BranchFilter 恰好"容忍"了这个破损；开启 mesh-space 后 slot 输出的解释方式变了，接近零的 delta 旋转被直接设为 spine 最终朝向，动作因此不可见。修复暴露了潜伏 bug，根因在数据不在修复。

方案 B（additive 正路）的尝试与失败：策划判断"之前标ADDITIVE似乎是有意为之的"，先走恢复 additive ＋ 蒙太奇 slot 改 UpperBodyAdditive 的正路，其间策划两次补齐 AI 漏掉的配置（BasePose 应选 Overlay 里的 Pose；RefPoseSeq 应指 rtgSteel_Idle_Combat）。showdebug animation 实锤链路百分之百工作（slot weight 与 Alpha 都满）但视觉纹丝不动，卡在两层死结：

  • Python 改 RefPoseSeq 不触发 re-bake（绕过 PostEditChangeProperty），手动 re-bake 后仍不对；
  • runtime base 不等于 bake-time base：Steel 的 pose 缺 Layering_Spine 曲线，overlay 的 combat idle 根本没叠到 spine 上，运行时基座还是 locomotion pose，delta 加在了错误的基上。

终案退回方案 A：四个动画改 AdditiveAnimType=AAT_NONE，两个蒙太奇 slot 回 UpperBody。代价是失去"挥砍随相机俯仰弯曲"。要走通 additive 正路需要美术与程序三步联合（统一 base、补 Layering 曲线、重烘 delta），单做任何一步都不成立。防范建议随之沉淀：批量扫描 AdditiveAnimType != NONE 且 RefPoseSeq == None 的动画列给美术。

顺手修掉的独立 bug：HeroAbility.h 的 OnAbilitySpecPostRelicatedChange 拼写错误在 UE5.7 严格 override 校验下炸编译（策划选择删 override 保留 dead code）；HeroAbility.cpp 的 TargetAbilityReport 空指针崩溃补了 null check。

展开可复用经验（沉淀条目）
  • UpperBody 层 LBPB 必开 Mesh Space Rotation Blend（Lyra/GASP 默认开），从 GASP 类模板迁移时优先核对这个默认值。
  • 切 blend 空间前先扫坏烘焙：AdditiveAnimType != NONE 且 RefPoseSeq == None 的动画在 local-space 下被"宽容"，mesh-space 一开就现形。
  • BlendMask 权重不沿骨链继承，twist 骨（八根）与 neck/head 必须显式填；空 mask 等于 slot 完全失效。
  • Slot 语义三件套定型：UpperBody（硬钉）/ UpperBodySoft（渐变传导，pelvis 0.15 起）/ UpperBodyAdditive（叠加）；换档靠选 slot 名，不靠运行时逻辑。
  • AIK 红线：set_node_property 碰 LBPB 数组属性或 ApplyAdditive 的 alpha bool 必崩编辑器；Python 改 RefPoseSeq 不触发 re-bake；BlendProfile 创建与删除只能 UI。
  • 验收纪律：别用一帧蒙太奇验 slot；蒙太奇编辑器预览不走 ABP；PIE 必须重启；保存后必须核 LogSavePackage，防"内存已改、盘上没改"的假象。
  • "上半身跟镜头"的真实实现是整身 Strafe ＋ Warping ＋ AO 的合成，不是脊柱独立旋转；跟手差异主要来自 AO 资产质量。
