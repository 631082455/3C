上半身 Overlay 体系：持械姿态与 Motion Matching 的叠加

Motion Matching 管腿和位移，但"这个英雄拿着锤子、正在戒备、准备开枪"这些信息不该由移动库承担：每个持械姿态都烘进移动库，量产会当场爆炸。我们的答案是三层叠加：MM 打底、Overlay 姿态层管持械与戒备、Slot 管瞬时动作。本篇写清这套体系的状态语义、驱动链、与 MM 的叠加形式，以及一个字段收口的逐英雄接入配方（10 个英雄在用）。

定位移动文档矩阵的姿态层分册：手册管参数、历程管过程、标准管验收，本篇管"移动之上的表达层怎么搭"。
体系roaming / combat 双态语义 + HAD 四字段驱动链（策划勾两个布尔即接入）+ C++ 基类 HeroOverlayPose 的三态姿势矩阵（Relax / Ready / Aim × 站蹲 × 速度档）。
现状10 个英雄子 Overlay ABP 在用；接入成本 = 一个子 ABP + HeroData 一个字段；姿势资产可跨英雄分级复用（KongfuTea 借 Twinblast 双枪 Aim 套实证）。
血统底层叠加节点位与姿势插槽概念沿用引擎动画方案；状态语义、驱动链、C++ 基类与接入模式为项目自建，本篇只写自建的部分。

如果持械差异直接做进移动库，每增加一种武器，跑、停、转身和冲刺就要跟着扩一套。英雄数量和武器类型一乘，移动资产很快就养不起。这里的取舍是让 Motion Matching 只负责通用腿部移动，武器持握、戒备和瞄准姿态放到上半身 Overlay 里处理。

引擎已经提供了姿势叠加的基础节点，项目补的是量产所需的整条链：roaming、combat、aim 三档状态；技能资产上的进出战配置；保存时反写蒙太奇；运行时由主动画蓝图和英雄 Overlay 接起来。新英雄接入时仍需要准备经过正确骨架处理的姿势资产、一个子 Overlay ABP、HeroData 入口和技能侧进出战配置，并不只是“填一个字段”。

当前资产盘点中有 10 个英雄在使用这套体系。部分姿势可以按武器类型复用，例如 KongfuTea 暂时借用了 Twinblast 的双枪姿势；但它的 Ready 列和瞄准开关仍未配完，Crouch 对应的插槽也只是保留结构。现有结果来自需求方逐英雄 PIE 验收，没有独立量化指标或多人验证。

体系

三层叠加模型

先立分工，再谈机制。角色最终姿势由三层从下往上叠出来，每层只回答一个问题：

分层铁律

每层只许回答自己的问题：移动库里不许出现持械变体（否则库规模 × 武器数），Overlay 层不许产生位移（位移权归底层与技能系统），Slot 播完必须归还（占用有路、归还也要有路）。三条里任何一条破了，都是跨层污染的开始。

血统声明（一句话说清）：中层那个"叠加姿势"的节点位与插槽概念沿用引擎动画方案；本篇之后的全部内容（状态语义、驱动链、C++ 基类、接入模式、打磨）是项目自建，也是这套东西真正能量产的原因。

状态语义：roaming / combat

姿态层的输入是一个问题：这个角色现在算不算"在战斗"。我们把它定义成两态：

状态语义姿态表现进出规则
roaming（游走）脱战移动、逛地图、回城松弛持械或收械，上身自然摆臂，MM 原味为主默认态；combat 计时结束回落
combat（临战）正在或刚刚进行战斗行为武器就位、上身戒备（Ready），可瞄准英雄进 Aim 子态由战斗行为驱动进入（下节驱动链）；退出走计时/行为结束

两条设计判断记录在案：

  • 进出战是表现层状态，不是判定层状态。它只改姿态与镜头语言，不改任何数值与判定，所以它可以由蒙太奇播放这种纯表现事件驱动，不需要过服务器仲裁。
  • 戒备度拆三档（Relax / Ready / Aim），而不是一个布尔。持弓走路（Ready）和拉弓瞄准（Aim）是两种上身，一个布尔装不下；三档对应姿势矩阵的三列，扩容不用改结构。

逐英雄可以收窄进战口径：Steel、Kallari 只在普攻上开进出战标志（技能不进战），因为他们的技能语义偏突进/隐身，举着武器戒备反而违和。口径是逐英雄资产配置，不是全局开关。

驱动链：从 HAD 到姿态

驱动链的设计目标只有一个：策划配技能时顺手决定进出战，运行时零额外配置。链路四步：

字段实读（编辑器实测，Kallari 普攻 HAD 的 MontagePlay 模块）：bChangeCombatStateOnPlay=True、OverrideCombatStateOnPlay=Add、bChangeCombatStateOnBlendOut=True、OverrideCombatStateOnBlendOut=Remove。四个字段就是策划的全部接口。

PreSave 反写这步是链路的关键取舍：备选方案是运行时每次播放查 HAD 配置，被否掉：查表把表现层和技能配置在运行时耦死，且 DS 上多一次无意义查找。代价是"改了 HAD 必须重存一次让反写生效"，这条写进接入配方的检查项。

与 MM 的叠加形式

姿态层的宿主是 C++ 基类 HeroOverlayPose（蓝图基类 ABP_OverlayPose_Base 直接父类，编辑器实读）。它内部是一个三态状态机加一张姿势矩阵：

三态状态机（SM_OverlayPose，实读）

Relaxed ↔ Ready ↔ Aiming 三个状态对应戒备三档。进 Ready 由 CombatState 驱动（上一节的链）；进 Aiming 由 bCanAim 桥控制：主 ABP 把 InputStateWantsToAim 桥进来，只有该英雄的 Overlay 声明了 bCanAim 才响应瞄准输入。Aim 态另配 AimSweepValue 驱动 AO_*_Aim_Sweep 瞄准偏移（俯仰扫掠）。

姿势矩阵（22 个插槽，实读自基类变量表）

维度取值说明
戒备档Relax / Ready / Aim三列；Aim 列另带 AO_Stand/Crouch_Aim_Sweep 两个瞄准偏移插槽
姿态Stand / Crouch项目裁了 Crouch 玩法，Crouch 行保留插槽不填（结构不动，量产不做）
速度档Idle / Walk / Run / Sprint与 MM 的 Gait 对齐；每格一个姿势资产插槽（Pose_Stand_Run_Ready 这样的命名）

叠加的形式：这些 Pose_* 是单帧/短循环的姿势资产，不是完整动画。姿态层拿它对 MM 输出做上身方向的修正叠加，腿部与位移完全归 MM。跑步的摆臂节奏来自 MM 的跑步动画，"举着锤跑"里的锤和上身角度来自 Overlay 姿势，两层在骨骼链上分工（上身权重高、骨盆以下趋零），所以移动库不需要任何持械变体。

与 MM 侧机制的两处衔接（机制细节指路手册 06，不重讲）：分层混合必须 Mesh Space + 显式 BlendMask（否则上身跟骨盆晃，三期根因）；蒙太奇 Slot 层叠在姿态层之上，原地/移动分支切全身透传或遮罩分层。

速度档插槽与 Gait 的对齐是隐性契约：MM 侧裁了 Walk 档（MOBA-Lite），姿态矩阵的 Walk 列实际不会被驱动，但插槽保留，若未来某英雄恢复 Walk 档，姿态层零改动。反过来，动了 E_Gait 枚举会同时打穿两层，这是那条"枚举不要动"红线的另一半理由。

量产10 个英雄在用的接入配方

逐英雄接入配方

新英雄要"进战举起武器"，三步，全在资产层：

#动作说明
1建子 Overlay ABP：继承 ABP_OverlayPose_Base，类默认值里填姿势矩阵只填用得上的格：不许瞄准的英雄 bCanAim=False、Aim 列不填；姿势资产可借现成的（见下方 KongfuTea 实例）。图不用改，逻辑全在基类
2HeroData 一个字段收口：CombatOverlayPose 指向子 ABP实读：HeroData_KongfuTea.CombatOverlayPose = ABP_Overlay_KongfuTea_C。主 ABP 的桥（InputStateWantsToAim 等）自动生效，无需逐英雄接线
3技能侧勾进出战：HAD 的 MontagePlay 模块四字段普攻至少勾一对 Add/Remove；技能是否进战按英雄气质定（Steel/Kallari 只普攻开）。改完重存 HAD 让 PreSave 反写生效

前置红线

姿势资产必须先重定向到 UEFN 驱动骨架再进插槽，直接塞原骨架姿势，骨盆比例差会在原地全身分支现形（重定向治理战役的教训，见六期）。新英雄漏接 bCanAim 桥的症状是"瞄准输入没反应但不报错"，接入自查第一条。

实例：KongfuTea 的省钱接法（编辑器实读）

茶壶英雄的瞄准姿势没有专属资产，直接借 Twinblast 的双枪套：Pose_Stand_Idle_Aim = Pose_Twinblast_Pistol2H_Stand_Aim_Idle，AO_Stand_Aim_Sweep 同套借用；Ready 列暂空、bCanAim=False（瞄准链路配好但未开闸）。这就是姿势矩阵设计想要的量产形态：插槽标准化，资产分级复用：旗舰英雄配专属全套，次级英雄借同武器类型的现成套，先上线再逐步替换专属资产。

变体清单

  • Steel / Kallari：只在普攻开进出战（技能气质不适合戒备姿态）。
  • Muriel：飞行形态单独一套 Overlay（ABP_Overlay_MurielFlight），姿态层按形态切换而非只按战斗状态，是体系支持多套姿态并存的实例。
  • Twinblast：双枪套是最早打样的英雄，目录里留有 Temp / BrokenBackup 一串试错化石（打磨过程见三期文档）。

打磨实录摘要

体系从能用到好用打了四场小战役，完整过程在三期文档，这里只留结论：

战役症状定案
spine 边界上身 Overlay 参与到下半身 spine，扭身拧成麻花上下身在 spine 链上的分工重划，OrientationWarping 分摊 0.5 定案
更跟手举枪姿态跟镜头有迟滞Overlay 姿态响应链路收紧（05-08 批次）
root rotation 释放Aiming 移动时腿保持旧方向Aim 态下 root rotation 释放，腿部允许对齐移动向
手部细化持弓手型覆盖不到手指hand overlay 骨骼链延伸到指骨（弓箭手批次）

验收口径：全部为需求方本人 PIE 实机逐英雄过（n=1），与移动文档矩阵其余各篇一致；姿态层无独立量化指标，挂靠《手感量化标准》巡航组的转向表现项。
