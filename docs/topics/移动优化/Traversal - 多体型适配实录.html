攀爬系统的多体型适配·Steel 深潜实录

同一套 GASPALS 翻越动画，放在 TwinBlast 身上能成立，放到 Steel 这种矮粗、大胶囊、骨骼比例差异明显的角色身上，就会依次暴露手脚约束、落点空间和悬挂高度三类问题。最后的解法不是统一缩放动画，而是把差异拆成骨架与 IK、Motion Warping 空间、动作 × 高度 × 角色配置三层分别收口。

起点标准 Mannequin 动画与大体型角色的骨骼比例、IK 目标、胶囊半径和手搭边缘后的 Root Motion 位移都不一致。
终案手臂回到 FK、脚保留受控 IK；Warp 的 XY 使用角色实际胶囊半径；Z 由 Mantle / Climb × Short / Tall 四格角色配置决定。
实证Steel 当前只有高墙 Climb 额外补偿 30cm；TwinBlast 与 Sparrow 四格均为 0。它是角色差异，不是全局推荐值。

标准体型的攀爬方案放到 Steel 上会同时暴露几类误差：手部约束可能把手臂拉坏，胶囊尺寸会改变角色与墙面的横向距离，骨架比例和动画剩余 Root Motion 又会影响最终高度。它们最后都表现成“攀爬位置不对”，但不是同一个问题，继续只调一个 Z Offset 很容易把错误藏起来。

这次把适配拆成三层处理。先按目标骨架重新决定哪些 IK 约束该保留；横向位置直接读取角色实际胶囊半径；高度补偿则拆成 Mantle、Climb 与高低墙组合，由共享逻辑负责选格，各英雄的 MCD 负责填写自己的值。这样代码不需要知道角色名字，体型差异也不会被一个 Steel 特例带进所有英雄。

当前实读配置中，只有 Steel 的高墙 Climb 使用 30cm 补偿，TwinBlast 和 Sparrow 对应四格仍为 0。旧接入文档中的 Steel 单值 120 已经过时，不能再作为现行参数。这个系统自动化的是差异如何被表达和选择，不会自动算出每个体型的正确结果；IK 取舍和补偿值仍要拿目标骨架与动画逐项验收。

为什么标准方案会失效

GASPALS 的 Traversal 动画以 UEFN Mannequin 为设计基准。常规体型角色通过运行时重定向复用它时，骨盆高度、肢体长度和胶囊尺寸接近，误差通常还能被 Motion Warping 吃掉。Steel 不一样：躯干宽、四肢比例和标准人形差异明显，胶囊横向占用也更大。标准方案里原本互相接近的四个点开始分家。

动画语义

手在第几帧抓住边缘，抓住后 Root Motion 还会上移多少。

骨架语义

root、pelvis、手脚末端在该体型上的真实高度和比例。

碰撞语义

胶囊半径决定身体离墙多远才不会穿墙，不能沿用标准角色常量。

Warp 语义

FrontLedge 是最终落脚点，还是手搭边缘时的悬挂根位置。

判断顺序不能反。如果手臂已经被 FBIK 拉坏，继续调 Z Offset 只会让一个错误姿势落到更准确的位置；如果胶囊离墙距离不对，改动画高度也解决不了穿墙。先形状，再空间，最后才是数值。

证据时间线

提交记录找回后，能看到多体型适配并不是 4 月突然出现的一个字段，而是三个月里逐层积累：先建 Steel 自己的动画数据底座，再让 Climb 真正跑起来，最后才把体型差抽象成共享机制。

三层适配形状 → 空间 → 数值

第一层：骨架与 IK

Steel 当前的 IKRig 和 Retargeter 已经把「大体型不能照搬标准 IK」写进资产本身。编辑器实读不是一套全开的通用解，而是一组有意识的取舍。

约束项Steel 当前状态设计目的
LeftHandIK / RightHandIKGoal 存在，但与 FBIK Solver 断开手臂回到纯 FK，避免大体型手被强拉向 Mannequin 的世界目标而扭曲
LeftFootIK / RightFootIKGoal 绑定 foot_l / foot_r，仍连接 Solver脚保留落地约束，防止体型变化后脚底漂浮或穿地
ik_foot_rootFK Chains 中 EnableFK=false避免额外的脚根链参与重定向，和腿链重复驱动造成抖动
Retarget IK Goals op整层关闭；左右臂链也显式 EnableIK=false大体型上不让标准角色的手脚世界目标直接接管动作
腿部 IK 链设置BlendToSource=1，左右腿各有 ±4cm Local Offset腿保留源目标语义，并对 Steel 的宽体站距作细调
Run IK Rig开启把最终需要的脚部约束放回角色自己的 IKRig 求解

这里的核心不是「IK 越多越好」，而是每个 Goal 是否仍然表达目标角色真实需要的约束。手部目标在 Steel 上带来的误差大于收益，所以断开；脚底接触仍是硬约束，所以保留，但交给 Steel 自己的 IKRig 解。

第二层：Motion Warping 空间

骨架形状稳定后，第二层才是「角色应该被 Warp 到哪里」。现行代码把目标拆成 XY 与 Z 两个独立问题处理：

TargetXY = SurfaceXY - WallNormalXY × CharacterCapsuleRadius

TargetZ = SurfaceZ - CharacterZOffset(ActionType, ObstacleHeight)

XY：必须读角色真实胶囊半径

GetMantleStartLocation() 直接调用 GetScaledCapsuleRadius()。这条看似普通，实际就是横向体型适配：大胶囊要离墙更远，小胶囊可以更贴边。如果写死 Mannequin 的半径，Steel 即使 Z 高度完全正确，胸口和肩仍可能挤进墙体。

Z：先弄清 sync point 说的是什么

现行注释把 FrontLedge 定义为「手抓住边缘时，角色仍在悬挂」的同步点。抓住以后，动画自己的 Root Motion 还会继续把角色向上送一段距离。因此 Warp 目标不能直接落在墙顶，而要从墙顶减去这段剩余上移量。角色体型和动作不同，这段量也不同。

旧系统曾用单一 MantleHeightFromMontage 表达这一差值。它能区分「GASPALS 落脚点式」与「Steel 悬挂式」，但无法同时覆盖 Mantle / Climb、矮墙 / 高墙四种组合，所以后面被四格机制取代。

第三层：动作 × 高度 × 角色 MCD

4 月落地的关键抽象，是把 Z 补偿从一个全局数拆成四格。代码先用 ActionType 选行，再用 ObstacleHeight >= HeightTierThreshold 选列：

动作 \ 高度Short（低于阈值）Tall（达到或高于阈值）
MantleMantleZOffset_ShortMantleZOffset_Tall
ClimbClimbZOffset_ShortClimbZOffset_Tall

Hurdle / Vault 与旧系统仍回退到 MantleHeightFromMontage。也就是说，四格不是把所有动作都复杂化，而是只对「手搭边缘后还要完成上墙」的 Mantle / Climb 开启体型补偿。

Steel 这 30cm 在说什么

只在高墙 Climb 生效。Warp 的悬挂目标比墙顶低 30cm，给 Steel 的高墙攀爬动画留出后续 Root Motion 上移空间。

它不是什么

不是所有大体型的推荐值，不是胶囊高度，也不是角色整体缩放。换一个骨架或一套 Climb 动画，都要重新验。

这套设计把共享逻辑和角色差异分开：C++ 只负责选哪一格，MCD 负责这一格是多少。新增体型不再改代码分支，也不会把 Steel 的补偿污染到所有英雄。

从单角色到全英雄治理

Steel 把多体型问题暴露出来后，系统没有停在「给 Steel 写特殊分支」。4 月底三次提交把共享部分铺到全英雄 MCD：

  1. CL 5202323「攀爬整体流畅度优化」：一次修改 DoublePole、Gideon、Kallari、Muriel、Phase、Riktor、Sevarog、Sparrow、Steel、TwinBlast 等十多个英雄 MCD。
  2. CL 5207085「fix traversal」：继续覆盖同一批角色，并补 Narbash、SkyfireMecha 等配置。
  3. CL 5209320 / 5209620：代码新增 Hurdle / Vault / Mantle / Climb 分动作速率倍率，各英雄 MCD 再批量落值。
适合全英雄共享必须保留角色差异
动作类型判定、Height Tier 选格逻辑、Warp 公式、PlayRate 合成方式、Ability 生命周期胶囊尺寸、IK Goal 取舍、Retarget Chain 设置、四格 Z Offset、动作槽和体感倍率

可扩展性的关键不是「所有英雄配成一样」，而是所有英雄使用同一种配置语言。Steel 可以只改 Climb Tall，标准体型保持全 0；程序逻辑不需要知道哪个角色叫 Steel。

验收矩阵与排错顺序

多体型问题很容易跨层误诊。下面按运行时现象给首查顺序，避免在错误层级里来回调数。

最小验收场景

  • 同一角色至少测 Stand / Walk / Run 三种入墙速度，避免只在单一相位成立。
  • 至少测矮墙 Mantle 与高墙 Climb，两边跨过 HeightTierThreshold。
  • 正面与斜角接近都测，观察手搭点、胶囊离墙距离、最终脚底落点三个时刻。
  • 单机通过后再测 Listen Server 客户端，确认没有服务端位置纠正把角色拉回。
  • 翻越结束后继续移动，确认 MM 能从最终姿势接回步态而不跳帧。

边界、旧口径与可复用结论

旧文档里的 Steel=120 已经过时

早期《新角色攀爬/翻越系统接入指南》记录 Steel 的 MantleHeightFromMontage=120，代表当时用单值描述悬挂动画余下上移量。现行资产已经不是这套口径：Steel 为新系统，MantleHeightFromMontage=-1 只作为非 Mantle/Climb 或旧系统回退；Mantle / Climb 实际走四格，当前只有 ClimbZOffset_Tall=30。

-1 不是当前高墙 Climb 的实际补偿。新系统遇到 Mantle / Climb 会覆盖这个回退值。直接盯着旧字段判断 Steel 落点，会得出错误结论。

多体型适配没有变成自动缩放

系统自动化的是「差异如何被表达和选择」，不是「自动算出每个角色的正确值」。胶囊半径可以运行时直接读取，动作类型与高度档可以由代码选择；但手臂该不该启用 IK、Climb Tall 应补多少，仍需要拿目标骨架与目标动画在编辑器里验。

最后留下的三条方法

  1. 先把问题放回正确坐标系。爆骨是约束问题，穿墙是胶囊空间问题，站位高低才是 Z 补偿问题。
  2. 特例应该变成配置维度，不应该变成角色名分支。动作 × 高度 × 角色 MCD 让 Steel 的差异可表达，也不污染标准体型。
  3. 资产联调要按链路验收。IK Rig、Retargeter、HeroData、Chooser、PSD、MCD 任一层缺失，都可能表现成「攀爬动画不对」。
