原系统不区分箱子、薄墙和高台，所有障碍都走同一套 Mantle。此次接入把翻越拆成 Hurdle、Vault、Mantle、Climb 四类，并根据高度、深度、背面地板和接近速度选择动作，同时保留项目原有的 GAS、CharacterMovement 与 Motion Matching 框架。

真正需要联调的不只是动画。翻越期间角色处于无重力的 Flying 状态，技能无论正常完成、被打断还是启动失败，都必须把 MovementMode 归还；障碍深度探针、Motion Warping 目标名和蒙太奇安全结束点也必须使用同一套语义。最终视觉启动延迟从约 530ms 降到约 67ms，第一天就存在、却因为测试场景全是薄墙而没有暴露的“深度恒为 0”问题也被替换为顶面向下的步进探针。

全文按三场调试、28 个阶段保留了每次判断、验证和回退，参数是当时的实测记录，不是对现行系统的再次追改。当前仍不支持大角度斜向入墙，Walk 档资产没有补齐，Motion Warping Notify 提前播放也只是对 PoseSearch BranchIn 的近似替代。要复用这份记录，应先看问题发生在哪一层，不要直接照抄其中的阈值。

背景

工程背景与系统全貌

项目是 UE5 多人英雄射击，英雄移动动画统一跑在 Motion Matching 上。翻越这件事在此之前只有一个 GA_Mantle：单一蒙太奇、固定高度假设、Motion Warping 目标手动配置。GASPALS 提供了完整的四类翻越动画资产与分选设计，本次工程的目标就是把这套东西移植到自有英雄身上，同时不动现有 GAS 框架与 MM 框架。

涉及模块：UHeroAbility_Traverse（新建）、UHeroCharacterMovementComponent::CanTraverse()、FMantleSetting（扩展）、HeroCharacter::CheckHeroInput()、Motion Warping Warp Targets、AnimBP 与 GASPALS 蒙太奇适配。

一次翻越的完整链路

按住空格到翻越完成，系统按这条链走：

  1. 玩家按住空格，bHeroInputJumpHeld 每帧为 true（持续窗口，不是单帧脉冲）
  2. CheckHeroInput() 检测：基础距离 80cm + 速度加成（最大扩到 450cm）
  3. CanTraverse() 探针：FrontHit → 高度探针 → 顶面向下的深度步进探针
  4. 按高度 + 深度 + 背面地板分类出动作类型（见下方决策表）
  5. StartTraversal()：设 MOVE_Flying，注册 FrontLedge / BackLedge Warp Target
  6. GA_Traverse 激活：按速度与深度分选蒙太奇，StartTime = MotionWarping notify 时间提前 2 帧
  7. Motion Warping 把角色平滑推到目标点，安全结束点到达后 Montage_Stop(0.25f) → EndAbility()
  8. SetMovementMode(MOVE_Falling)，物理接管落地，自然切回 Walking

MOVE_Flying 是全篇最大的雷区

翻越期间角色处于 MOVE_Flying（无重力，位移全靠 Root Motion / Motion Warping 驱动）。任何一条 Ability 路径（正常完成、被打断、Task 为 null）只要漏掉 EndAbility，这个状态就会泄露：角色永久悬浮、滑步、且 CanTraverse() 检测到 Flying 直接拒绝，系统整体挂死。后面的实录里，这个模式反复出现了四次。

翻越类型决策

每次触发按以下优先级判断，第一个满足的分支生效：

判定条件动作类型说明

高度 80~160cm 且深度 ≥ 59cmMantle厚墙，爬上去站立

实录

第一场：从接入到能玩

第一场从零搭到基本可玩，13 个阶段。前半是骨架与蒙太奇的架构问题，后半是手感与时序，其中视觉延迟一战打了三次败仗。

阶段 1：搭建，看起来顺利，实际埋了三颗雷

扩展 FMantleSetting、新建 UHeroAbility_Traverse（GAS Tag Event.Input.Traverse 触发）、接入输入逻辑、批量重定向 GASPALS 蒙太奇。当时以为一切正常，实际埋了三颗雷，后面逐个爆：重定向方向搞反了（把 UEFN Mannequin 动画重定向到 TwinBlast 骨骼，而 TwinBlast 的 Primary Mesh 本来就用 UEFN Mannequin 骨骼）；Warp Target 名字沿用了旧系统的 MantleTarget；FMantleSetting 改结构后热重载会丢 DataAsset 字段。

阶段 2：按空格完全没反应

根因是委托绑定顺序的竞争条件：OnCancelled / OnInterrupted 绑定在 ReadyForActivation() 之后。蒙太奇若在第一帧就被取消，委托还没挂上，回调永不执行，EndAbility 永不调用，角色永久停在 MOVE_Flying。而 CanTraverse() 检测到 Flying 直接返回 false，于是空格再按也没用，整个 Ability 挂死。修复：所有委托绑定移到 ReadyForActivation() 之前，补齐全部四个回调。

阶段 3：T-Pose 加空中悬浮

误判 1　怀疑 ABP 里 BlendListByBool 没激活 DefaultSlot 链路，花了较长时间分析节点图，最后确认这条路是错的，DefaultSlot 一直正常。

真根因：双 Mesh 架构没弄清。播放的蒙太奇是重定向出来的 TwinBlast 骨骼版本，而 Primary Mesh 用的是 UEFN Mannequin 骨骼。骨骼不匹配，UE 拒绝播放，触发 OnCancelled，角色回到无动画驱动的 T-Pose。

误判 2　弄清双 Mesh 后又试了一条歪路：改用 MovementModeChangedDelegate 监听状态变化，删掉了蒙太奇播放代码。结果角色完全不动。GASPALS 的翻越必须由 GAS 主动播蒙太奇驱动，ABP 不会自己进入翻越状态。

最终修复：直接用 GASPALS 原生 UEFN Mannequin 骨骼的蒙太奇，不再要重定向版本。教训一句话：T-Pose 加 Ability 秒取消，先查骨骼匹配性，别先跳进 AnimBP 节点图。

阶段 4：动画播了，角色原地不动

又先后怀疑了 LayeredBoneBlend 权重、PoseSearch 的取消日志，都不是。真根因很朴素：Warp Target 名字对不上。GASPALS 全套蒙太奇里 Motion Warping Notify 的目标名是 FrontLedge，代码注册的却是旧系统的 MantleTarget。找不到目标，Motion Warping 不执行，角色原地播完整段动画。修复：全部注册改为 FrontLedge，Hurdle / Vault 的后向落点改为 BackLedge。

阶段 5：能动了，但飞到墙顶上方

FrontLedge 位置公式沿用旧系统，假设根骨骼在髋部，所以给落点加了 HalfHeight 偏移。GASPALS 骨骼的根在脚底，FrontLedge 本身就是落脚点。修复后公式一行：SurfaceHit.Location - FrontHit.Normal × CapsuleRadius，不加任何 Z 偏移。

阶段 6：高度分类阈值混乱

各种高度的墙触发的动作类型随机感很强。原始阈值全是直觉估的，而 GASPALS 动画是按设计高度命名的：_1_0 系列对应 100cm，_2_5 系列对应 250cm。旧配置 MantleMin=150cm 配 100cm 设计高度的动画，Motion Warping 硬拉伸 1.5 倍，动作严重扭曲。按动画设计高度重新校准后的分类见 01 章决策表。

阶段 7：薄墙厚墙不区分

同一高度段里，薄墙应该翻过去（Vault），厚墙应该爬上去站住（Mantle），原逻辑不看深度一律同一动作。修复：80~160cm 高度段增加障碍物深度检测，按深度阈值分流。

阶段 8：站着不动也播跑步版蒙太奇

Motion Matching 只管平地移动动画，翻越蒙太奇是 GAS 指定播放的，MM 不会替你选版本。修复：FMantleSetting 每种动作新增 Run / Stand 两档蒙太奇字段，按速度阈值选择。

阶段 9：跑向墙时按空格没反应

两个问题叠加：bHeroInputJump 只在按下的第一帧为 true，同帧末清零；检测距离固定 80cm。以 300 cm/s 跑速计算，从按下到进入检测范围要 0.27 秒，标志早归零了。修复：新增 bHeroInputJumpHeld 持续窗口标志（按住每帧为 true）；检测距离动态扩展，SpeedBonus = Min(Speed2D × 0.5, 300)，高速时最大扩到 450cm。修复后主观手感从「贴墙才有反应」变成「快接近就响应」。

阶段 10：视觉延迟 0.5 秒，全程最硬的一仗

现象：按下空格后约 0.5 秒才看到翻越动作，GASPALS 原版几乎瞬发。这一仗打了两轮诊断加三次实现失败。

次要因素　第一轮诊断指向 OffsetRootBone：奔跑时骨骼相对胶囊有累积偏移，翻越开始时要 0.3~0.6 秒衰减回中心。在 Get_OffsetRootTranslationHalfLife 里让 DefaultSlot 激活时返回 0.001，偏移 5 帧内清零。延迟少了约 0.1 秒，主延迟还在。这里误判了主因，耗掉不少时间。

根因确认　分析 GASPALS 原版 CBP_SandboxCharacter::PerformTraversalAction()：原版靠 PoseSearch 在 PoseSearchBranchIn notify 范围内（帧 0~16，约 0.53 秒）搜一个与当前跑姿最匹配的帧，直接从那帧起播，把前摇整段跳掉。我们的系统始终从第 0 帧播，0.53 秒前摇一帧不少地走完。

  • 失败 1　用 NotifyStateClass != nullptr 找 BranchIn 的结束时间。PoseSearch 插件类在当前模块未被解析，指针恒为 null，条件永远不成立。
  • 失败 2　改成 GetTriggerTime() < KINDA_SMALL_NUMBER 判断。结果 BranchIn 的触发时间恰好精确等于 KINDA_SMALL_NUMBER（0.0001），0.0001 < 0.0001 恒为 false，条件再次失效。
  • 失败 3　逻辑终于改对了，但用 unreal.reload() 当作重编译。它只 reload Python 模块，完全不碰 C++。测试的一直是旧代码，「修复 → 测试 → 没变化 → 继续找 bug」空转了好几轮。

全程单一最大的效率陷阱就是失败 3。改 C++ 后必须点编辑器 Compile 按钮（或走 Hot Reload），unreal.reload() 无效。这个模式在整场调试里出现了 4~5 次。

最终修复：放弃从 BranchIn 找结束时间，改为扫描蒙太奇里 MotionWarping notify（ActionWarpIn）的触发时间（通常 0.4~0.6 秒），从该时间提前 2 帧（67ms）开始播放。Motion Warping 在这 67ms 里已经开始推角色，主观上接近瞬发。视觉延迟从约 530ms 降到约 67ms。编译后当场验收通过：「延迟感没了，ok 了。」

阶段 11：翻完不落地，空中滑步

重启编辑器后暴露：翻越完成后角色空中滑步，日志持续 Mode=5（MOVE_Flying）。EndAbility 没有任何机制触发。修复：新增 OnWarpPhaseComplete 回调加 UAbilityTask_WaitDelay，在蒙太奇安全结束点依次执行 Montage_Stop(0.25f)、恢复 MovementMode、EndAbility()。

阶段 12：结束太早，翻不过去

修完滑步又出新问题：角色爬到墙顶就直接坠落。第一次选的安全点是 MotionWarping notify 结束时间（0.835s）加 50ms，但此刻 Root Motion 还在把角色往前推，提前 Stop 打断了它。修复：改用 PlayMontageNotifyWindow（TraversalBlendOut）的结束时间（1.192s），此时角色已完全越过墙体开始下落，才是真正安全的结束点。

阶段 13：Climb 还是翻不过去

Vault / Hurdle 都好了，Climb 独独不行。分层拆开看：Climb 实际播的是 MM 短片段（AM_M_Neutral_Traversal_Climb_Start_2_5_run_F_Lfoot），它没有 PlayMontageNotifyWindow，BlendOut 通知用的类名是 BP_NotifyState_TraversalBlendOut_C。代码用 Contains("MontageBlendOut") 匹配，永远匹配不到，fallback 到 0.9 秒，而 Climb 的安全点在 2.1 秒。修复：匹配条件放宽为 Contains("BlendOut")，两种命名风格都覆盖，自动算出 Climb run ≈ 1.70 秒。

第二场：CHT 对齐与深度探针

第一场结束时系统「能玩」，第二场解决「选得对不对」：蒙太奇分选规则与 GASPALS 原版对齐，顺带挖出一个从第一天就存在的底层 Bug。

阶段 14：起步/折返加速从 ABP 迁到 C++

原版起步加速逻辑写在 AnimBP 状态机里，参数硬编码，运行时不可调，角色间无法独立配置。迁移到 UHeroCharacterMovementComponent 作纯运动层逻辑：起步检测（静止转有输入）和折返检测（速度方向与输入方向的点积低于 PivotDotThreshold）触发时把 StartBoostTimer 重置为 BoostDuration，每帧在 GetMaxSpeed() 里按剩余比例线性叠加峰值速度。三个参数（BoostMaxSpeed / BoostDuration / PivotDotThreshold）全部进 DataAsset，按角色独立配置。迁移后 AnimBP 只读速度播动画，调参不再重编蓝图。

阶段 15~17：V2 蒙太奇始终被选中

现象：不管速度和墙的深度，永远播 V2 版本（run_F_V2_Lfoot）。初始误判把 V2 和非 V2 当成互斥替换关系，试图直接换掉。正确做法是回头读懂 GASPALS 原版 CHT（Chooser Table）Hurdles 子表的分选规则，逐格核对截图后确认了完整规则：

ObstacleDepth速度档选用蒙太奇
[0, 25) cm 薄< 100 站立stand_F_V2_Lfoot
100~250 行走walk_F_Lfoot / Rfoot
≥ 250 奔跑run_F_Lfoot / Rfoot
[25, 60) cm 厚< 100 站立stand_F_V2_Rfoot
100~250 行走walk_F_V2_Lfoot / Rfoot
≥ 250 奔跑run_F_V2_Lfoot / Rfoot

三个颠覆原认知的发现：V2 不是厚墙专属（薄墙站立也用 V2）；速度是三档不是两档（行走 100~250 是独立一档）；V2 分界在 25cm，与 60cm 的 Mantle 分界、与高度分类都无关。据此把 Hurdle 蒙太奇槽从 4 个扩到 12 个（3 速度档 × 2 深度档 × 左右脚），选择逻辑带完整 fallback 链（某档未配则回退上一档）。

阶段 18：ObstacleDepth 恒为 0，从第一天就错的底层 Bug

对齐 CHT 后发现：明明是厚墙，日志里 ObstacleDepth = 0.0，分选永远走薄墙路径。根因：原深度探针从 FrontHit 碰撞点向障碍物背面做水平反向 Line Trace，而测试场景的 LevelBlock_Traversable 背面不响应 MantleCheck 碰撞通道，探针永远不命中。

这个 Bug 从系统搭建第一天就存在，却活到了第二场。因为早期测试场景恰好全是薄墙，深度 0 的结果凑巧「正确」，系统勉强能跑，把底层错误盖住了。影响面：深度分选、厚墙分类、地板探针位置、BackLedge 目标计算，全部建立在一个恒为 0 的输入上。

修复方案换了探测思路：顶面向下步进探针。从障碍物顶面沿前进方向每 5cm 一步、向下打短射线（最大 70cm），命中则更新深度，落空即为背边缘。不依赖背面碰撞响应，还顺带拿到背边缘精确位置供 BackLedge 使用。

第三场：打磨与旧系统连修

第三场做两件事：把新系统的边角磨平（阶段 19~24），以及收拾一个连带后果——新系统改动的底层逻辑把旧版 GA_Mantle 弄出了一串新 Bug，Sparrow 和 Steel 两个角色逐一排修（阶段 25~28）。

阶段 19：PlayRate 加速播放后角色多悬浮一截

SafeEndTime 是蒙太奇内部时间，WaitDelay 等的是真实时间。PlayRate=1.2 时蒙太奇 1.0 秒处实际只要 0.833 秒，差值就是角色在 MOVE_Flying 里多挂的时间。修复一行：所有等待统一用 WarpPhaseDelay ÷ PlayRate。

阶段 20：MaxDepthScan 不随配置

DataAsset 里调大 MinMantleDepth，深度扫描上限却写死不动，超过旧上限的厚墙识别不出来。修复：MaxDepthScan = Max(100, MinMantleDepth × 1.5)，探针范围随配置自动扩展。

阶段 21：Vault 补 Walk 档

Vault 只有 Stand / Run 两档，与 Hurdle 的三档不一致，行走速度贴近障碍时播 Run 版突兀。新增 Walk 档蒙太奇槽，选择逻辑与 Hurdle 完全对齐，Walk 未配置时回退 Run 保持兼容。

阶段 22：低帧率下连续触发两次翻越

PostTraversalCooldown = 0.1s，帧率低于 30fps 时一帧的 deltaTime 就可能吃掉整个冷却，落地前检测又通过了一次。提到 0.25 秒，20fps 下也有足够窗口。同期做了日志分级整理：流程确认降 Verbose，非预期状态保持 Warning，系统性错误升 Error。

阶段 23：落地后步态跳变

翻越结束落地，步态循环和翻越末姿势接不上，有明显错位。先排除了 Foot IK / Foot Placement（它们只管踝关节贴地，不管步频相位）。真根因在 MM 节点属性：bResetOnBecomingRelevant = true，蒙太奇结束、MM 重新激活时从零开始搜索，与当前姿势差距大，直接跳变。改为 false 后，MM 基于蒙太奇结束那一帧的姿势就近搜索，衔接平滑。

阶段 24：楼梯误触发翻越

初次误判想改楼梯的碰撞通道，那只是绕过症状。真正的洞在 CanTraverse() 的 step-up 分支：没检查背面下落高度。楼梯台阶背面下落通常不到 22cm，本该由 UE 原生 StepUp 处理。修复：增加 MantleMinStepUpDrop（默认 22cm）阈值检查。同时移除了旧的 IsWalkable 法向量检查，它的 0.71 阈值会误拒法向量偏斜的普通箱体（实测 NormalDotUp=0.39 被拒）。

阶段 25~28：旧版 GA_Mantle 四连修

新系统开发中动过的底层（MovementMode 管理、EndAbility、移动组件接口）波及了旧攀爬。四个问题按角色逐一排掉：

问题角色根因修复
无法触发，日志刷屏SteelLineTraceMulti 返回 true 但无有效 blocking hit，OutSurfaceHit 保持 Reset 状态（Z=0），算出负高度还意外通过了单边检查高度探针后补 IsValidBlockingHit() 有效性检查
触发后永久悬浮Steel蒙太奇缺失时 AbilityTask 为 null，无回调来源，EndAbility 永不触发；且旧代码只绑了 OnBlendOut 一个回调null 检查（null 则立即 CancelAbility 恢复状态），补全四个回调绑定
攀爬完成后空中溜冰Sparrow依赖的 UpdateRootMotionTransitionAfterMovement() 只在有 AnimRootMotion 时才恢复移动模式，旧蒙太奇没有 Root Motion 数据，恢复永不触发override EndAbility，检测到 MOVE_Flying 时清速度并显式切 MOVE_Falling，让物理接管落地
Warp 进墙体内部或严重偏低Steel两个角色的蒙太奇 sync point 含义不同：Sparrow 是最终落脚点（偏移 0），Steel 是悬挂时的根位置（需减去手搭点到站立点的 Z 差）新增 MantleHeightFromMontage DataAsset 字段按角色配置，默认 0 兼容 Sparrow 风格

四连修共用同一条结论

不能把 MovementMode 的恢复寄托在任何「正常路径」上。EndAbility 必须显式检查并恢复 MOVE_Flying，且恢复目标选 MOVE_Falling 而不是 Walking：让物理引擎接管落地检测，避免空中直接切 Walking 的位置突变。

整场约 13 个主要阶段，每个阶段内含 1~3 次失败尝试，明显「走错方向再回头」的有三次，编译未生效导致的无效循环至少 4~5 次。误判全录：

三条带走的教训

一：T-Pose 加 Ability 秒取消，先查骨骼匹配，别先钻节点图。二：「系统勉强能跑」是最危险的状态，恒为 0 的深度探针靠巧合活过了一整场。三：改 C++ 必须确认编译真的发生了，这一条的损耗超过所有技术误判之和。

研发链路的变化

这一场同时是一次工作流实验：3C 体验问题的传统链路是策划描述问题、程序定位修改、TA 调蒙太奇、策划再验证，每轮都是跨角色的沟通与排期。本次全程没走程序排期，策划确认体验问题后当场分析、当场修改、当场验证。逐环节对比：

需要人来补的部分同样真实：AI 感知不到编译状态，看不到角色在场景里的实际表现，长对话后上下文会丢失早期决策，也不了解项目与 GASPALS 原版之间已有的 diff。最顺的协作模式固定下来是：人描述期望表现与实际现象（附 log），AI 分析根因并改代码，人编译测试再回报现象，如此循环。附 OutputLog 片段能显著提高定位精度。

比省下的排期更值钱的是认知边界的扩展：过完这一场，蒙太奇 Notify 的时序分布、GAS Ability 生命周期、MovementMode 状态机语义，都从「间接听说」变成「亲手读过」。之后对翻越类动作的体验判断、问题描述、调试收敛速度，全都受益于此。

遗留问题

  1. Steel 多体型适配：单值 MantleHeightFromMontage 已演进为 Mantle / Climb × Short / Tall 四格角色配置，Steel 当前仅高墙 Climb 使用 30cm 补偿。完整证据链与现行参数见《多体型适配 · Steel 深潜实录》。
  2. 翻越角度判断：目前只支持正面朝墙。斜角接近（与墙面法线夹角大于 30°）时 Motion Warping 会过度旋转角色，与 GASPALS 原版有差距。
  3. Walk 档蒙太奇资产：代码层已支持，DataAsset 里 Walk 槽尚未指定资产，当前回退 Run 档。纯编辑器可解。
  4. 左右脚接缝跳变：L/R 版本切换在边缘情况下仍有姿势跳变。GASPALS 原版靠 PoseSearch 自动选最匹配帧回避了此问题。
  5. PoseSearch BranchIn 完整移植：当前用 MotionWarping notify 时间近似模拟，不是等价方案。完整接入 PoseSearch 后应替换为真正的 BranchIn 搜索，可同时消除前摇残余与左右脚接缝。
