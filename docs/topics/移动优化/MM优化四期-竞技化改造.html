移动优化四期：MM 竞技化手感改造 V1（2026-06）

把 GASP 展示级的 Motion Matching 移动整套搬进 MOBA 之后要算两笔账：手感病灶（冲刺打摆子进不了巡航、起步误选反身动画）要治到根因层；量产成本要从每英雄 624 条移动动画裁到 87 条，并立下一条以后每次取舍都用的保留判据。本篇是这两笔账的完整过程，验证到 11 个英雄局内一键 AB。

课题展示级移动库搬进 MOBA 后两头不对：冲刺在过渡动画里打摆子、起步误选反身动画；量产一个英雄要 624 条移动动画，几十个英雄摊不起。
终案裁出 MOBA-Lite 档（8 库 87 条，判据：能感知、做得出、影响命中判定，三选一才留）；Gait 档位改由移速属性驱动（含 buff，迟滞 560/480）；局内一键 Dense/Sparse 实机 AB。
结果11 英雄生效，起步与巡航稳定；移动动画量产账 624 → 87；起步偶发误选 Reface 未根治，写在遗留区。
分工作者（战斗 3C 策划）定原则、否方案、逐轮实机验收；AI agent 做编辑器实测、资产修改、引擎源码查证并代写唯一一处 C++；程序本轮零投入。

GASP 的 Motion Matching 接进项目后，Sprint 档首先暴露出两个问题：650 移速下，角色会在过渡动画之间反复切换，难以稳定进入巡航；正向起步还会偶发选到反身 180° 的 Reface 动画。与此同时，官方 Dense 库的移动四族共有 624 个数据库候选条目，直接按这个密度扩到几十个英雄，维护成本也不可接受。

手感侧已经落地的处理包括：用角色的最大移速属性驱动 Gait，加入升降档迟滞；修正 Chooser 中起步行被 Any 兜底行截断的问题；把抢占巡航的过渡动画采样范围限制在前半段。Dense 档的起步到巡航因此稳定下来，并做了局内 Dense、Sparse 切换工具，方便后续按实机结果比较动画库。

量产侧则定下了 MOBA-Lite 的目标结构：从现有候选库中收敛到 8 个库、约 87 个数据库 Entry，并用“玩家能感知、玩家做得出、影响命中判定”作为保留条件。这里的 624 和 87 都是数据库候选口径，不等于动画师的源 Clip 交付量；MOBA-Lite 的最终路由也还没有完整铺开。Reface 误选仍未根治，Sparse 档尚未同步 Dense 的全部修复，现有手感结论主要来自单人 PIE 验收。

背景

体验目标与课题

项目是 UE5 的多人 MOBA 射击游戏，移动层整套采用 GASPALS（项目对 Epic Game Animation Sample 与 ALS Overlay 体系的整合插件），全部英雄共享同一个 Motion Matching 动画蓝图。做这件事时移动系统已经上线跑了半年，处在"能用，但要过竞技这关"的阶段。

竞技移动对动画层的要求其实就三条：按下去就动、松开就停、转身干脆；播出来的动作和角色实际运动一致，不骗人；这套东西要养得起几十个英雄。前两条是手感，第三条是账。

现状背离

手感这头有两个病灶，都发生在冲刺（Sprint）档：

现在我的 MCD 把角色设置到了 650，一直按住前进键，但是角色始终在这些动画里切，实际预期应该是一个 Loop 了。
需求方（作者本人），2026-06-02。MCD 即 MovementConfigDataAsset，逐英雄的移动配置资产
我明明是在只往前跑 为什么会被这个 reface 给干掉啊
同上，2026-06-24。Reface 指反身 180° 转身起步动画，正前直跑时被它抢中

账这头：GASP 的 Dense 全量档，仅移动四族（巡航/起步/急停/折返）就 624 条动画条目。照这个标准量产，每个新英雄都要交付一套 624，项目排不起这个产能。

课题定性：打磨型为主（病灶要治到根因层，不打补丁），兼一点建设型（裁库标准从无到有）。这里没有两个正确目标打架的矛盾，就是系统从展示级转竞技级欠下的两笔账。

核心课题

一，冲刺档的两个病灶治到根因层，修复不许以砍动画为代价；二，移动库裁到 MOBA 品类值得付的量，并且立下判据，管住以后每一次"这条动画留不留"的争论。

分工与工作方式

作者（战斗 3C 策划，C++ 让 AI 狠狠写、由我拍板提交）：出课题、定设计原则、否决方案、每一轮实机验收。AI agent（经 AIK 编辑器自动化桥）：库数据实测、资产修改、引擎源码查证，本案唯一一处 C++（控制台变量注册）也由 AI 代写、作者提交。程序：本轮零投入。

工作方式上有一条贯穿全程：单变量实验加明确证伪。AI 的每个结论都要过作者实机复核，实录里 AI 被驳回、被打脸的地方全部保留原样，那正是定位问题最快的几步。

过程

业界调研

先回答必答题：处理"动画库太大养不起"，业界最普遍的做法是动画数据分档，同一套匹配框架，配不同保真度的候选库。这个先例不用出门找，GASP 官方自带 Dense/Sparse 两档。我们与它的差异只有一步：顺着它的砍法再切一档品类特化档，并把"砍什么"从官方的经验做法整理成可执行的判据。

对象它的做法借了什么 / 为什么不走核实方式
GASP Sparse（官方精简档）高频移动四族砍 66%（624 → 215），低频（跳跃/待机/落地/原地转身）不做精简版，路由直接指回 Dense砍法策略全部照搬（见下表）；"精简高频、复用低频"的结构照搬AIK 编辑器实测，16 对同名库逐条 entry 对比
Fortnite与 GASP 同款 PoseSearch 框架，上线跑竞技框架层竞技可用的信心，框架一行不重写Epic 公开资料
Marvel Rivalslocomotion 全部原地动画（拆包实测 12 片段 root 位移 = 0）加 6 向 blendspace，胶囊驱动，完全不用 MM竞技射击的另一极参照：响应优先可以彻底不要 MM。没走这条：已有 MM 框架与资产的沉没成本，以及转向/起步的表现上限项目内拆包资产实测。注意边界：其状态机结构文档是由命名反推的，非真实 ABP 解析
ALS（社区）上一代事实标准：状态机 + blendspace + 分层 Overlay无直接借用，作为技术代际坐标公开仓库

从 Sparse 逆向出的五条砍法

对 16 对同名库逐条对比后，官方的砍法可以归纳成五条（这是我们自己的归纳，不是官方文档写的）：

#策略实例
1Sprint 系不砍Loops 7 / Starts 6 / Stops 2，两档同份（本就极简）
2方向矩阵砍 45° 对角，留正向与 90° 侧向Run_Stops 20 → 12，对角靠惯性混合在 0.2s 内插出来
3折返库砍复合过渡Run_Pivots 136 → 34（−75%），Diamond/Hourglass/Prism 全砍
4Reface 反身起步全留急转后起步是玩家最敏感的戏剧瞬间，8 条一条不砍
5巡航砍微变体微偏角、相位偏移、单脚 mirror，靠轨迹插值补回

这一节的谱系实测多于文献。"竞技游戏怎么裁动画库"这种行业常识很少有人专门写文章，检索只能找到被写下来的东西（路灯效应），所以先例主要靠拆手边的资产。谱系有没有漏掉更近的同类解，入库前要请作者本人过目。

方案排定

方案是四刀切出来的，每一刀都有被否掉的对手。

第一刀：框架和资产分开判

裁库之前先回答一个更根本的问题，作者的原话问得很直接：

实际上 如果我们是一个上线项目 直接采用 GASP 这套移动资产 会不会太挫了？ 但感觉自己做 意义又不大
作者，2026-06-22

判断是把这个问题拆成两层。框架层（MM 匹配 + PoseSearch + Chooser 路由 + Warping 变形）直接吃，一行不重写，Fortnite 同款没什么可自研的；资产层分级换：旗舰英雄专属全套，普通英雄共享下半身加专属上半身 Overlay，杂兵全共享。"挫不挫"只发生在资产层，而资产层是可以逐英雄逐步替换的。

第二刀：裁多少

让 AI 分别站动画设计师、程序、3C 策划三个立场各给一份保留清单，三个视角的结论收敛成一句判据（见下卡）。然后算两档账：只砍 Walk，省 29%；Walk 和 Crouch 全砍，省 58%，落到 8 库 87 条。作者拍板走激进档："没有 Crouch 也没有 Walk"。MOBA 视角拉得远，走与跑的姿态差玩家几乎不可感知；下蹲这个动词在本项目玩法里根本不存在。低速走位（补刀、卡位）靠 Stride Warping 压步幅防脚滑兜底。

保留判据（本案最重要的沉淀）

一条动画候选留不留，三问：玩家能感知吗；玩家做得出这个操作吗；它影响命中判定吗。三选一就留，三个都不沾就砍。程序侧另有一条硬指标兜底：单库候选 ≤35 条是搜索开销与匹配干净度的舒适区。

第三刀：Gait 档位的驱动依据

修冲刺病灶时 AI 先后提过两个方案：锁档（起步保护期强制 Run 1.3 秒）和瞬时速度驱动。都被否了，否决原话在第 04 节实录里。定下来的原则：Gait 由"角色能跑多快"（移速属性，运行时含 buff 的 GetMaxSpeed）决定，不由"此刻跑多快"（瞬时速度）决定。加速期是暂态，档位是身份，身份不该随暂态漂移。配迟滞双阈值（升 560 / 降 480）防临界抖动。

第四刀：验证工具先行

裁库是数据决策，数据决策要能实机 AB 才敢下刀。所以先做局内切换开关再动库：控制台变量方案（作弊位，不写配置不持久化，重启回 Dense），挂在 F1 GM 面板上一个勾选框。这一步顺序是有意的，工具先于决策。

回退条款

裁库用软改：只删 Chooser 路由行，不删插件共享库的任何资产。路由不到等于不存在，效果与删库相同，但随时可以把行加回来；共享库牵一发动全身（多英雄引用、归一化集合引用 Crouch 库），物理删除不可轻回滚。所有匹配参数（采样范围、cost bias）的调整，没调稳的一律回退不提交。

实验与落地实录

按时间序五场战役。执行者均为 AI agent，每轮结论由作者实机复核。

战役一：冲刺打摆子，三层根因（06-02）

650 移速按住前进进不了巡航。根因链逐层挖出，一层比一层深：

  1. Gait 变量全项目无驱动来源。GASP 原版在它自己的角色蓝图里 set Gait，本项目角色是 C++ 类不继承它，转型失败后 Gait 永远卡在动画蓝图默认值 Run（C++ 全局检索 0 处写入实证）。
  2. Chooser 路由只认 Gait 枚举不看速度，于是永远只把 Run 巡航库喂给 MM。
  3. Run 巡航库里藏着两条"Sprint 转 Run"的过渡动画，原生速度约 650 与查询轨迹匹配度最高，MM 反复选它们，循环不止。

当天先落地 Gait 驱动（临时用瞬时速度版）并断开原版每帧把 Gait 覆盖回 Run 的原始 bug。之后 AI 开始连环调参（重选开关、cost bias 一路加码到 −0.5、直接禁用条目），越调越乱，被作者叫停：

还是会混 你先别急 这样打补丁修改肯定是不合理的
作者，2026-06-02

撤回补丁时发现一个大乌龙：前几轮所有库改动改的都是 Steel 目录下的副本，而角色实际走的是插件目录的原版库，等于全部白改。教训：改共享资产前先摸清引用链，"我以为在改的库"和"角色在用的库"可以不是同一个。

战役二：Dense 和 Sparse 到底差在哪（06-15 ~ 06-16）

为裁库做的全量实测：16 对同名库逐条对比，移动四族 624 → 215（−66%），折返库砍得最狠（Run_Pivots 136 → 34），Sprint 系一条不砍。两档引擎参数完全一致（同 Schema、同 KD 树配置），Sparse 只减候选不动算法。中途 AI 断言"Sparse 档不能跳跃"，被作者实机打脸：

我切到 SPARSE 后 是有跳跃的呀
作者，2026-06-15

修正后的认知成了裁库结构的地基：Sparse 只精简巡航/起步/急停/折返四类高频族，跳跃、待机、落地、原地转身这些低频库不做精简版，路由直接指回 Dense。五条砍法的归纳见第 02 节。

战役三：冲刺卡过渡，速度失配（06-24）

裁库方案定稿后回头治病灶。先确认与精简无关（作者："不是 我是说 dense 的 psd 里 依然会这样"）。第一个假设"重选开关导致粘滞"被单变量实验干净证伪。决定性证据来自 Blend Stack 调试面板：巡航动画的播放速率显示 0.86，反推动画原生速度 650 / 0.86 ≈ 756，正是 GASP 标准三档 200/500/750 里的 750。根因就此清楚：角色 650 与动画 756 的速度失配。"Run 转 Sprint"的过渡动画中段必然扫过 650，那一帧的匹配代价天然最低，MM 死守它不放，动画层没有坏，是数据和配置的错配。

修法是把过渡动画的采样范围限到前半段 [0, 0.5]，稳了；试着放宽到 [0, 0.8]，又乱（作者："又开始乱切了 而且乱七八糟的"），回到 0.5。结论记录在案：在采样这个维度上，"起步动画完整"和"巡航无波浪"天生冲突，没有甜点值，停止调参。

这场战役里 AI 有一次自我收回值得记录：曾推荐用 Stride Warping 解决速度失配，查证后收回，因为匹配代价用的是烘进 KD 树的原始轨迹，Warping 是选中之后的变形，改的是"选中之后脚怎么摆"，改不了"MM 选谁"。

Gait 驱动依据的三连纠错也发生在这天，最后一问 AI 被引擎源码打脸：

这么设计好像不对吧，假如我角色的移速属性已经达到 650 了，当前速度还没达到，我这时候也不应该锁在 run 的 start 呀
作者，2026-06-24，否决"起步保护期锁 Run"方案，确立移速属性驱动原则
那个值确实不是那里，我所说的 是游戏实际运行时生效的速度属性。我现在进入之后会有一个加速 buff
作者，2026-06-24，指出真实移速要含 buff
你确定是吧 现在没办法拿到真实移速？ 你确认下
作者，2026-06-24。AI 此前断言 GetMaxSpeed 不可蓝图调用、必须加 C++ 接口，查引擎源码后认错：它本就是 BlueprintCallable

落地方案：事件图（游戏线程）每帧把 GetMaxSpeed 缓存进 EffectiveMaxSpeed 变量，动画线程读变量算 Gait，绕开线程安全限制（GetMaxSpeed 节点由作者在编辑器手连，AIK 生成不了引擎继承函数的本地化节点）。

三值全对之后起步还是不进起步库，最后一环是 GASP 原始数据自己的 bug：Sprint 子表里"巡航行（条件填 Any）"排在"起步行（条件 = True）"之前，Any 兜底行截胡了一切；Run 子表行序本来就对，所以 Run 从来没犯病。挪一行修复。作者当天验收："现在对了 还有什么茬子不你感觉"。

战役四：起步误选 Reface，七轮证伪，未根治（06-24 晚 ~ 06-25）

冲刺稳了之后，起步问题浮出：正前直跑，起步却播反身 180° 转身动画，还在左右两个 180 之间抽搐。七轮假设全部证伪：

轮假设验证结果
1上一战役补进起步库的过渡动画扰动了匹配空间禁用后重建索引证伪 Reface 照旧
2特定角色骨骼问题换 Sparrow 对照证伪 同骨骼同表现
3strafe 朝向模式（面朝 ≠ 移动向）运行时切朝移动方向否决 Reface 消失但违背设计意图：镜头朝后时就该有反身转跑

5朝向模式在 C++ 层永久生效再测编译实测，运行时确认 flag 生效证伪 仍 Reface，彻底排除朝向路线
6Gait 改瞬时速度驱动实测证伪 更糟：临界抖动，且暴露 Run 起步同样有 Reface（全向库把它盖住了）

方向的每一次扭转都来自作者的追问。第一问挖出数据面：Run 起步库 28 条全向覆盖，Sprint 起步库只有 8 条（正前 2 + Reface 4 + 补的过渡 2），侧向中间角度全缺，GASP 的设计里 Sprint 是"Run 的加速态"，起步库天然残缺。第三问破了题：

跑起来了，不过你得想一个问题 为什么 run 没有这个问题 sprint 才有

方法论沉淀

展开沉淀条目

候选元规则（M），入库时并入 references 并与本案建双向引用：

  • 三选一才留：表现资产裁减的保留判据，能感知、做得出、影响命中判定。适用边界：任何动画/特效/表现类候选库的量产裁减；不适用于逻辑资产（逻辑缺一条就是功能缺失，不是保真度问题）。
  • 档位由属性驱动，不由瞬时值驱动：Gait、姿态档、任何"角色处于哪一档"的判断，依据是角色的能力属性（含 buff 的运行时值），不是当下的瞬时状态；暂态用迟滞吸收。适用边界：档位语义是"身份"时成立；档位语义本身就是"当前物理状态"（如空中/落地）时不适用。
  • 验证工具先于数据决策：要动数据档，先做实机 AB 开关。适用边界：一切可分档、可开关的数据资产调优；开关本身要做状态回读，否则开关会误导排查。
  • 选后变形改不了选谁：匹配代价用的是烘焙进索引的原始数据，Warping 类节点是选中之后的修饰。适用边界：一切"检索 + 修饰"结构的系统（MM、寻路平滑、目标选取后的表现修正），治选择问题要去数据和查询侧。

候选反模式（AP）：

  • 无证据补库：往候选库里加条目是回归源，不是安全操作（本案"进不了巡航"的元凶正是自己补进起步库的过渡动画）。补库前先证明缺的是数据，而不是路由或驱动。
  • 连环调参代替单变量证伪：cost bias 一路加码到 −0.5 的那一小时什么都没定位到；三次干净的单变量证伪（重选开关、朝向、转向速率）各自省了一条弯路。
  • 修共享资产不摸引用链："我以为在改的库"和"角色在用的库"可以不是同一个，改错目标白干一轮。
