1. 适用范围

本文面向当前项目内“新增一个英雄角色”的接入工作，重点覆盖以下系统：

  • HeroData 角色主配置
  • 小白人骨骼（UEFN_Mannequin）驱动 + 目标角色网格体重定向
  • IK Rig / IK Retargeter
  • ABP_TwinblastCharacter_MM 主角色动画蓝图
  • ABP_Overlay_TwinblastPistol Overlay 动画蓝图
  • CHT_PoseSearchDatabases_Root Motion Matching 选库链路
  • MCD_Twinblast 移动 / Traversal 配置
  • 技能 Montage 的 UAbilityMontageData
  • 武器挂件 / MobaUnit 绑定

本文以 Twinblast 当前项目内的实际生效链路为准，不以旧说明或注释为准。

2. Twinblast 当前接入架构

Twinblast 当前不是“直接让 Paragon 网格体自己跑完整 AnimBP”，而是下面这套双网格体结构：

隐藏驱动网格体（Source）
/GASPALS/Characters/UEFN_Mannequin/Meshes/SKM_UEFN_Mannequin
    -> ABP_TwinblastCharacter_MM
    -> Link Overlay: ABP_Overlay_TwinblastPistol
    -> Motion Matching / Aim / Traversal / 技能逻辑都跑在这层

可见表现网格体（Target）
/Game/Characters_Test/Paragon/Heroes/TwinBlast/Meshes/SKM_TwinBlast_ActionHero
    -> RetargetMeshComponent
    -> ABP_GenericRetarget
    ->； 通过 RTG_UEFN_to_TwinBlast 实时跟随隐藏驱动网格体

这意味着：

  • 英雄的“行为逻辑骨架”是小白人骨骼。
  • Paragon 角色网格体只是可见表现层。
  • 主 AnimBP、Overlay、Motion Matching、Traversal、Ability Montage 绝大多数逻辑都仍然围绕小白人这一层构建。

4. 当前实际运行时链路

4.1 HeroData 初始化

UMobaDataSubsystem::OnHeroDataLoaded(） 会按 HeroData 完成下面几件事：

  1. 给主 Mesh 设置 HeroMesh 和 HeroAnimClass
  2. 如果同时配置了 RetargetMesh 和 RetargetAnimClass
  3. 隐藏主 Mesh
  4. 启用 RetargetMeshComponent
  5. 给 RetargetMeshComponent 设置目标角色网格体和 ABP_GenericRetarget
  6. 把 HeroData.IkRetargeterAsset 写入 RetargetAnim->HeroIkRetargeter
  7. 给主 Mesh 执行 LinkAnimClassLayers(CombatOverlayPose）

因此，新英雄最核心的入口就是 HeroData_[HeroName]。

4.2 主 ABP 与 Overlay ABP 的分工

Twinblast 当前并没有大改父类 ABP_HeroCharacter_MM 的整套 AnimGraph，而是在子类里只补“角色特有输入”：

  • ABP_TwinblastCharacter_MM
  • 父类：ABP_HeroCharacter_MM

  • EventGraph 逻辑：

    • 调用父类 BlueprintUpdateAnimation
    • 获取已链接的 ABP_Overlay_TwinblastPistol
    • Cast 到 HeroOverlayPose
    • 读取 bCombatState
    • 写回本体的 InputStateWantsToAim

  • ABP_Overlay_TwinblastPistol

  • 父类：ABP_OverlayPose_Base
  • EventGraph 逻辑：
    • 每帧把 bCombatState 写入 bCanAim

这两个子蓝图本质上只是在给父类图提供“这个英雄现在是否处于瞄准 / 战斗态”的输入。

5. 新英雄接入时必须理解的双网格体模型

如果你准备新增一个 Paragon 英雄，且仍然走“小白人驱动，目标网格体重定向”的路线，则必须遵守下面这个模型：

  • HeroMesh
  • 必须是小白人或兼容小白人动作栈的驱动骨架

  • 负责：

    • Motion Matching
    • Overlay
    • Traversal
    • 技能蒙太奇
    • GameplayTag 驱动的动画状态

  • RetargetMesh

  • 是最终玩家看到的英雄网格体
  • 负责：
    • 跟随 Source 层实时重定向表现

如果你只改 RetargetMesh 那一层，而没有把 Source 层配置好，角色通常会出现：

  • 有模型，但没有正确动作
  • Motion Matching 不工作
  • Overlay 不生效
  • 技能蒙太奇逻辑和表现不同步

6.3 新英雄的推荐做法

如果新英雄仍然走 Paragon 风格角色网格体：

  1. 为目标角色创建 IK_[HeroName]_Retarget
  2. retarget_root 设为 pelvis
  3. 补齐与小白人一致的主要 chains / goals
  4. 创建 RTG_UEFN_to_[HeroName]
  5. Source 选 IK_UEFN_Mannequin
  6. Target 选 IK_[HeroName]_Retarget
  7. 先做 Auto Map
  8. 再手工检查手臂、腿、脊柱、脚趾、手指
  9. 最后把 HeroData 的这 3 个字段配齐： - RetargetMesh - RetargetAnimClass - IkRetargeterAsset

6.4 一个非常重要的当前项目坑

项目内还存在另一个同名资产：

  • /Game/Heroes/Twinblast/RTG_UEFN_to_TwinBlast

它不是 HeroData_Twinblast 当前实际使用的那个，并且它的 Target IKRig 指向的是：

  • /Game/ParagonCountess/Characters/Heroes/Countess/Skins/Tier1/Count_Carnivale/Meshes/IK_SM_Countess_Carnivale

也就是说：

  • 不要因为名字一样就直接复制 /Game/Heroes/Twinblast/RTG_UEFN_to_TwinBlast
  • 参考 Twinblast 时，应该以 HeroData_Twinblast 里真正引用的 RTG 为准

7. ABP_GenericRetarget 的实际意义

ABP_GenericRetarget 是可见目标网格体的通用重定向 ABP。

当前有效链路是：

HeroData.IkRetargeterAsset
-> UMobaDataSubsystem::OnHeroDataLoaded
-> RetargetAnim->HeroIkRetargeter
-> ABP_GenericRetarget.EventGraph
-> Set IKRetargeter
-> Copy Retarget Profile from Retarget Asset
->； 更新 RetargetProfile

7.1 当前有效逻辑

ABP_GenericRetarget 当前真正使用的是变量：

  • HeroIkRetargeter

而它的值来自 C++ 注入，不是运行时自己猜出来的。

7.2 旧注释 / 旧逻辑不要照搬

ABP_GenericRetarget 里仍然保留了：

  • IKRetargeter_Map
  • 通过 ComponentTags 查找 RTG 的注释和节点

但按当前蓝图连线看，这条 Tag + Map 路径不是当前主链路。
当前项目里新增英雄时，应优先采用：

  • HeroData.IkRetargeterAsset 直接指定 RTG

不要把“必须给 Mesh Tag 才会重定向”当成现状结论。

8. 主角色 ABP 的意义与构建方式

8.1 必须继承 ABP_HeroCharacter_MM

Twinblast 的主角色 ABP：

  • /Game/Heroes/Twinblast/ABP_TwinblastCharacter_MM
  • 父类：ABP_HeroCharacter_MM

推荐新增英雄也沿用这个模式：

  • ABP_[HeroName]Character_MM : ABP_HeroCharacter_MM

原因：

  • 父类已经内置了 Motion Matching、Overlay、Trajectory、Traversal、AimOffset 等主流程
  • ABP_OverlayPose_Base 在 SetReferences 里会拿“主 AnimInstance”并 Cast 到 ABP_HeroCharacter_MM
  • 你的子类只要继承自它，这条链就能继续工作

8.2 Twinblast 子类当前只做什么

Twinblast 子类当前只补了一件事：

  • 读取链接的 HeroOverlayPose.bCombatState
  • 写入本体 InputStateWantsToAim

这意味着新英雄如果也是“战斗态时开启瞄准驱动”的模型，通常只需要：

  1. 继承 ABP_HeroCharacter_MM
  2. 保留父类所有主图
  3. 在子类 EventGraph 中补自己的状态输入桥接

而不是复制整套父类大图重做。

9. Overlay ABP 的意义与构建方式

9.1 Twinblast Overlay 的作用

Twinblast 当前 Overlay ABP：

  • /Game/Heroes/Twinblast/ABP_Overlay_TwinblastPistol
  • 父类：/GASPALS/OverlaySystem/Overlays/Poses/ABP_OverlayPose_Base

父类 ABP_OverlayPose_Base 已经有：

  • bCanAim
  • AimSweepValue
  • 一整套 Stand / Crouch / Relax / Ready / Aim pose 槽位

Twinblast 子类当前只补：

  • bCanAim = bCombatState

9.2 Overlay 的运行时挂载方式

当前运行时不是靠 DataAsset 间接找 Overlay，而是：

  • HeroData.CombatOverlayPose
  • -> LinkAnimClassLayers(CombatOverlayPose）

所以对新增英雄来说，真正的必配项是：

  • HeroData_[HeroName].CombatOverlayPose

9.3 DA_Overlay_TwinblastPistol 当前状态

/Game/Heroes/Twinblast/DA_Overlay_TwinblastPistol 当前包含：

  • Overlay Animation Blueprint
  • Static Mesh
  • Skeletal Mesh
  • Weapon Animation Blueprint
  • Left Hand
  • Socket Name

但当前项目内 get_referencers(） 结果为 0，说明它不是当前运行时主链路必须资产。
可把它视为：

  • 备用编辑器数据
  • 未来扩展用配置
  • 或历史遗留资产

新增英雄时不要把它当成 HeroData.CombatOverlayPose 的替代品。

10. CombatState 与技能 Montage 用户数据

10.1 CombatState 的真正来源

Twinblast 当前的战斗态，不是单纯在 ABP 里判断“正在放技能”。

真实链路是：

AnimMontage 上的 UAbilityMontageData
-> UHeroAbilitySystemComponent::PlayMontage / OnMontageStartBlendOut
-> UHeroOverlayPose::EnterCombatState / ExitCombatState
-> Overlay 内部维护 bCombatState / CombatAbilityTags
->； 主 ABP / Overlay ABP 读取 bCombatState

10.2 UHeroOverlayPose 当前逻辑

UHeroOverlayPose 当前会：

  • EnterCombatState(AbilityTag）
  • 把技能 Tag 记入 CombatAbilityTags
  • 立即置 bCombatState = true

  • 给 ASC 增加 Hero.Animation.CombatState

  • ExitCombatState(AbilityTag）

  • 从 CombatAbilityTags 移除该技能
  • 当数组清空后，延迟 ExitCombatStateDelay
  • 默认延迟为 3.0f
  • 延迟到期再把 bCombatState 置回 false

10.3 Twinblast 当前 Montage 示例

已确认几个 Twinblast Montage 上的 UAbilityMontageData：

  • 普攻示例：
  • DoubleShot_Fire_Lft_Additive_Montage1
  • AbilityTag = Hero.Ability.Skill.Twinblast.Attack
  • bChangeCombatStateOnPlay = true
  • OverrideCombatStateOnPlay = Add
  • bChangeCombatStateOnBlendOut = true

  • OverrideCombatStateOnBlendOut = Remove

  • 炸弹释放示例：

  • AM_TwinblastBomb_ChargeBlastRelease
  • AbilityTag = Hero.Ability.Skill.Twinblast.Bomb
  • bEndAbilityOnMontageEnd = true
  • bUpperBodyStateActive = true
  • bChangeCombatStateOnPlay = true
  • bChangeCombatStateOnBlendOut = true

新增英雄时，如果技能希望驱动战斗姿态或瞄准姿态，不能只配 Ability，不配 Montage UserData。

10.4 一个当前项目里的隐藏坑

UHeroAnimInstanceBase::AddOverrideOverlayPose(） 和 RemoveOverrideOverlayPose(） 当前 C++ 里是空实现。

这意味着：

  • UAbilityMontageData.bOverrideOverlay
  • OverrideOverlayPoseClass

虽然在接口上存在，但当前项目里不要把它当作已经可用的正式能力，除非你先补完这条 C++ 实现。

Twinblast 当前主链并不依赖这两个空函数。

11. CHT_PoseSearchDatabases_Root 的配置意义

11.1 Root chooser 当前结构

/Game/Heroes/Steel/MotionMatchingData/CHT_PoseSearchDatabases_Root

当前是一个按 GameplayTag 过滤的 Root chooser：

  • Hero.ID.Steel
  • Hero.ID.Twinblast
  • Hero.ID.GaspALS
  • Hero.ID.DoublePole
  • Hero.ID.Kallari
  • Hero.ID.Gideon
  • Hero.ID.Muriel
  • Hero.ID.Sevarog

Twinblast 当前已经有：

  • Hero.ID.Twinblast 对应的一行

11.2 Root chooser 的调用位置

ABP_HeroCharacter_MM 的 Update_MotionMatching 函数里会直接：

  • EvaluateChooser: CHT_PoseSearchDatabases_Root
  • 把返回的 PoseSearchDatabase 数组喂给 Motion Matching 节点

11.3 Root chooser 的上下文来源

ABP_HeroCharacter_MM::SetReferences 会：

  • 从 HeroCharacter 读出 HeroId
  • 写入变量 HeroIdTag

Root chooser 的上下文对象是：

  • ABP_HeroCharacter_MM self

所以新英雄只要 HeroData.HeroID 正确，且 Root chooser 有这一行，Motion Matching 才能走到正确分支。

11.4 Root 之后的 chooser 链

Twinblast 当前 Root row 实际会继续走通用 chooser：

  • /Game/Heroes/Steel/MotionMatchingData/NormalCHT/CHT_PoseSearchDatabases

它再按 MMDatabaseLOD 分流到：

  • CHT_PoseSearchDatabases_Dense
  • CHT_PoseSearchDatabases_Sparse

Dense / Sparse 再根据以下维度选库：

  • MovementMode
  • Stance
  • MovementState
  • Gait

例如：

  • Stand Idles
  • Stand Walks
  • Stand Runs
  • Stand Sprint
  • InAir
  • Crouch Idle
  • Crouch Moving

11.5 新英雄该怎么接

如果新英雄与 Twinblast 一样，仍然使用通用小白人 Motion Matching 数据：

  1. 在 CHT_PoseSearchDatabases_Root 中新增 Hero.ID.[HeroName]
  2. 新行的 Result 类型必须选择"计算选择器（EvaluateChooser）"，不能是直接资产引用
  3. 在 EvaluateChooser 的"选择器"字段中指向通用 CHT_PoseSearchDatabases

关键坑：新增行时编辑器默认的 Result 类型可能不是 EvaluateChooser，如果 Result 设置为直接对象引用而非 EvaluateChooser，Motion Matching 将无法正确选库，角色仍会 T-Pose。必须在行属性面板中手动将 Result 切换为"计算选择器"。

如果新英雄需要自己的 PoseSearch 数据：

  1. 复制一套 CHT_PoseSearchDatabases_[HeroName]
  2. 再拆成： - CHT_PoseSearchDatabases_Dense_[HeroName] - CHT_PoseSearchDatabases_Sparse_[HeroName]
  3. 最后让 Root row 指向你自己的 Hero chooser

11.6 当前项目里的一个值得注意的现象

项目中存在：

  • CHT_PoseSearchDatabases_Steel
  • CHT_PoseSearchDatabases_Dense_Steel
  • CHT_PoseSearchDatabases_Sparse_Steel

但当前 Root chooser 并没有引用它们，而是仍然引用通用 NormalCHT 版本。
所以如果你要做英雄专属 PoseSearch 数据，不要假设现有 *_Steel 已经是主链路模板，必须自己再检查 Root 引用是否切过去了。

12. MCD_Twinblast 的意义

Twinblast 的 MovementConfig：

  • /Game/Heroes/Twinblast/MCD_Twinblast

当前关键值：

  • MaxWalkSpeed = 425
  • MaxWalkSpeedCrouched = 300
  • GravityScale = 1.6
  • JumpZVelocity = 900
  • AirControl = 1.0
  • bUseNewTraversalSystem = true
  • MantleReachHeight = 130 ~ 160
  • MantleCheckDistance = 200
  • MinMantleDepth = 59
  • Traversal 使用的是 GASPALS 小白人那套 Montage 资源

对新增英雄的建议：

  • 如果体型和 Twinblast 类似，先复制 MCD_Twinblast
  • 如果体型差异大，再单独调：
  • Traversal 高度阈值
  • 对齐角度
  • Catch / Vault / Climb 阈值
  • 移动与跳跃参数

13. 武器 / MobaUnit 绑定

Twinblast 当前 HeroData.AbilityMobaUnits 会生成两个：

  • BP_TwinblastWeapon 挂到 hand_l
  • BP_TwinblastWeapon 挂到 hand_r

并带各自 RelativeTransform。

所以新增英雄如果也有：

  • 武器 Actor
  • 枪口 FX
  • Socket 发射点

则必须在 HeroData.AbilityMobaUnits 里把这些附属单元也接起来。
否则常见结果是：

  • 技能逻辑能放
  • 武器模型不出现
  • FX / Muzzle Socket 不对

14. 新增英雄的推荐最小接入步骤

14.1 必配资产

至少准备下面这些：

  1. HeroData_[HeroName]
  2. ABP_[HeroName]Character_MM
  3. ABP_Overlay_[HeroName]...
  4. IK_[HeroName]_Retarget

16. 当前项目中最容易踩的坑

16.1 复制了错误的 RTG

同名 RTG_UEFN_to_TwinBlast 有两份，只有 HeroData 引用的那份是当前真正生效的。

16.2 误以为 DA_Overlay_* 是运行时必配

Twinblast 当前运行时真正使用的是：

  • HeroData.CombatOverlayPose

而不是 DA_Overlay_TwinblastPistol。

16.3 误以为 ABP_GenericRetarget 仍然依赖 Mesh Tag

当前主链路是 C++ 直接注入 HeroIkRetargeter，不要照着旧注释只配 Tag 不配 HeroData.IkRetargeterAsset。

16.4 误以为 bOverrideOverlay 已经可用

AddOverrideOverlayPose(） / RemoveOverrideOverlayPose(） 当前是空实现，新增英雄不要把关键功能压在这条路上。

16.5 Root chooser 行的 Result 类型必须是 EvaluateChooser

HeroData.HeroID 正确不等于 Motion Matching 就能自动工作。
在 CHT_PoseSearchDatabases_Root 中新增行后，必须将 Result 类型切换为"计算选择器（EvaluateChooser）"，并在其"选择器"字段中指向 CHT_PoseSearchDatabases。

如果 Result 保留为默认的直接资产引用类型，Chooser 无法正确向下传递选库，角色依然 T-Pose，且不会有任何报错提示。

16.6 忘了 Source 层才是“主逻辑层”

如果你只盯着目标英雄网格体改 ABP，而没管小白人那层：

  • Motion Matching
  • Overlay
  • Traversal
  • 技能蒙太奇驱动

大概率都不会真正接通。

17. 建议的新增英雄模板

对于一个“体型接近 Twinblast / 接近小白人”的新英雄，建议直接按下面模板开工：

  1. 复制 HeroData_Twinblast
  2. 复制 ABP_TwinblastCharacter_MM
  3. 复制 ABP_Overlay_TwinblastPistol
  4. 复制 MCD_Twinblast
  5. 新建 IK_[HeroName]_Retarget
  6. 新建 RTG_UEFN_to_[HeroName]
  7. 把 HeroData 中 RetargetMesh / RTG / Overlay / MCD / Ability / Weapon 指向新资产
  8. 在 CHT_PoseSearchDatabases_Root 增加 Hero.ID.[HeroName]，Result 类型必须选"计算选择器（EvaluateChooser）"，选择器指向 CHT_PoseSearchDatabases
  9. 为关键 Montage 补 UAbilityMontageData
  10. 最后再做专项微调，而不是一开始重做父类大图
