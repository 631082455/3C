镜头动效层·二期需求（程序对齐）

2026-07-10 · 提出人:战斗3C · 背景:一期(CL 5628029/5628058)已上线,Kallari CrossSlash 全链路实配完成(CL 5635204 待提交);以下需求来自实配过程中撞到的边界 + 竞品调研定案(实证与出处见《战斗镜头动效层设计稿》附录与本页各条)。每条独立可做,按优先级排序。

一页读懂:一期解决了"演绎不污染弹道"(GameplayPOV 快照,已验证);二期解决三件事——①反馈链的最后一拍(命中位点),②演绎与瞄准的视觉矛盾(FOV 补偿 / Look-at 模式 / AimSafe 分类),③技能对 UI 的控制权(准心开关)。

P0-1 技能命中镜头通道:Cue→镜头映射表 【设计升级 2026-07-12:取代原"弹道命中触发位点"——受益面从弹道技能扩成一切命中,且不再逐管线开钩子】

现状技能命中(弹道/近战/AOE)在施法者与受击者两端都没有镜头反馈位点;普攻命中确认已上线(伤害回执通道,见附注分工)。技能伤害落地统一走 Buff 的 ExecuteCueTag → 受击方 GameplayCue(命中特效即此路),cue 参数自带施法者与受击者。
需求在项目的 cue 执行集中处(OnExecuteSkillParticles 一族)加一个镜头监听:查Cue→镜头映射表(镜头域新配置资产,行 = CueTag → {施法者档 FCameraMotionEffectRef, 受击者档 FCameraMotionEffectRef});按 cue 参数判"施法者/受击者是否本地被观察者",命中哪端推哪端。
约定推入走 PushMotionEffect(预算/重触发/全清全继承);映射按行选择加入——DoT 每跳的 cue 不加行即永远安静,默认安全;暴击分档不在此通道(cue 无暴击信息,那是回执通道的能力);cue 为不可靠多播,极端丢包丢一跳反馈可接受(只做反馈不做判定)。通道分工:底噪走结算(回执),签名走表现(cue),施法演绎走位点(模块/窗口)。
验收映射表加一行 CrossSlash 命中 cue → 施法者短促确认:刀波命中敌方英雄,施法者镜头与命中特效同帧反馈;同一行配受击者档,受击方有对应受击踢;未加行的 DoT/辅助 cue 零触发;团战多命中不叠爆(带内软拐点)。

P0-2 FOV 灵敏度补偿

现状动效层含 FOV 通道且允许在可瞄准窗口使用(蓄力 FOV 微缩、出手 FOV 冲已实配)。FOV 变化时未做输入补偿 = 演绎期间偷改玩家手感。行业标配:BF4 Uniform Soldier Aiming(系数 1.33)、OW zoom sens 37.89。
需求视角输入(yaw/pitch 增量)按 tan(渲染FOV/2) / tan(基准FOV/2) 缩放(焦距匹配,即 monitor-distance 0%;系数可做成后续可调项)。渲染 FOV 取动效后的最终 POV(GameMatchCameraComponent 写入 DesiredView 的值)——玩家看的是渲染画面。
验收蓄力(FOV -1.7%)与出手 FOV 冲期间做匀速追踪转身,屏幕上的角速度体感与常态一致;关闭动效后行为不变。

P1-1 准心显隐的 GameplayTag 开关

现状准心 = HAD 的 WidgetClasses 配的 per-技能 widget(18 个英雄共用 W_SparrowAttack,继承 HeroCrossHairWidget),由 HeroAbility.cpp:147 → MobaHUD::AddHeroAbilityUniqueWidget 按技能 tag 托管;无运行时隐藏/恢复接口。
需求注册 tag UI.HideCrosshair;HeroCrossHairWidget::NativeTick 查 Owner ASC 的该 tag → SetVisibility(Collapsed/自恢复)。技能侧用现有 Buff granted tags 开关,无需新配置字段。
约定tag 不存在时行为与现状完全一致(零回归);开关语义跟随 GAS 生命周期(打断/死亡 Buff 掉落即恢复,不会出现准心永久消失)。注意 DefaultGameplayTags.ini 当前在 CL 5600239 checkout 中,注册时协调。
验收任意技能挂 tag 的 Buff 期间准心消失;Buff 结束/技能打断/死亡重生后恢复;观战切换视角无残留。

P1-2 Look-at 反解合成模式(OrbitAimLocked)——演绎自由度的钥匙 【已实装 · CL 5643871,2026-07-11 Kallari 实测通过:"准星完全没有晃动感"】

现状可瞄准窗口内的轨道旋转必然挪动屏幕中心与真实瞄准点的映射(几何必然,Kallari 蓄力 yaw 实测确认)。当前只能在"环绕感"与"准星诚实"间取舍。
需求ECameraMotionRotationSpace 增加第三值 OrbitAimLocked:该空间下动效只驱动机位位置(沿轨道弧线平移),旋转逐帧由 look-at 反解——看向 GameplayPOV 瞄准射线上的参考点(取 min(技能射程, 固定截断距离) 处,防空旷方向视差不稳)。锁定镜头(魂系/塞尔达)的通用骨架,调研范围内唯一兼得"环绕演出 + 中心诚实"的方案。
约定实现位置在 GameMatchCameraComponent::ApplyMotionEffects 的 Orbit 分支旁;环绕弧度建议钳 ≤10°(角色挡准线风险,Nesky:近目标滑移勿旋转);玩家转向输入改 GameplayPOV → look-at 参考点跟着走,自洽。
验收配一个 8° yaw 的 OrbitAimLocked 蓄力动效:准星压住的世界点全程不动(camera_trace 可量化验证),机位绕行视差明显;与 ViewKick/FOV 通道叠加无打架。

P2-1 CME 资产 AimSafe 分类 + 校验器把关

现状"哪些动效能进可瞄准窗口"目前是人脑纪律(FOV/roll/沿轴推拉/平移=安全;旋转/非轴向位移=挪准星),没有制度化。
需求CME 加枚举 AimSafety { AimSafe, AimDisplacing };IsDataValid 校验:含非零 ViewKick/Orbit 旋转或非视轴平移的资产必须标 AimDisplacing,标错拒绝存盘。库存资产的标注由策划侧补。

P2-2 瞄准态 Band 抑制(运行时兜底)

需求处于"可瞄准且操作权满"状态时(操作权档位从 HAD 推导,一期 config-grill 已定此原则),对 AimDisplacing 条目的权重乘全局衰减(可配 0~0.3)或直接 blend out。任何人配错资产,瞄准一开演出自动淡出——纪律的保险丝。依赖 P2-1。

储备(暂不排期)· WoT 式真值准星

大幅度演绎技能(大招级)启用:主准星淡化为"意图锚点",另画一个半透明真值标记投影在 GameplayPOV 射线落点(ProjectWorldToScreen 全套模式项目内已有先例:HeroDamageNumberCanvas.cpp:236-243,注意 DPI 除法与 UMG 上帧 POV 拖尾两个坑);分离-收敛全程平滑,0.15~0.3s 缓动回中。实证:WoT/War Thunder 双准星(官方为第三人称视差而做)、MH Wilds Focus Mode 准星归属设置项、CS2 Follow Recoil。成功律:偏心必须钉住看得懂的真值,忌无语义漂移。

验证工具已就绪:以上每条的验收都可用 GameProject/Scripts/camera_trace.py(逐帧 POV/蒙太奇进度/投射物首现记录仪)出客观数据,不靠肉眼争论。
