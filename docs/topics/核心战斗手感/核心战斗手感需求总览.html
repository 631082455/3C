效果提升

把“新技能遇到旧技能时发生什么”变成可配置、可验收的公共语义，降低后续技能扩展成本。

1. 背景与目标

当前项目已经有 GAS 的标签阻挡、取消能力，也有 HeroAbility 层的输入、动画、RootMotion、镜头等自定义逻辑。随着角色技能增多，如果继续靠单个技能特判处理“谁能打断谁、谁能并行、谁互斥”，策划会很难预测最终表现，程序也会很难维护。

本版目标：
  • 统一技能相遇时的策划语义，避免“互斥”“打断”“阻挡”混用。
  • 策划能配置技能默认关系，也能为少数关键窗口配置例外。
  • 输入缓存作为全局输入策略管理，技能卡只声明少量例外。
  • 程序能按固定顺序判定，不允许单个技能绕过公共规则。
  • 逻辑允许并行时，必须同步检查动画、移动、镜头等表现层是否成立。

2. 本版范围

类型 内容
本版支持 阻挡、打断、并行、全局输入缓存策略、不可打断窗口；默认关系；少量窗口和缓存例外；表现层并行验收。
本版不做 全技能全窗口的 N × N × M 完整矩阵；技能对技能缓存矩阵；所有角色历史技能一次性重配；完整编辑器工具链；AI 自动平衡技能关系。
首批样例 普通攻击与技能的双向关系；技能期间是否阻挡开火；开火期间是否阻挡技能；持续开火类技能与 Dash 并行；强技能起手不可打断；Holding / 蓄力技能的取消与缓存例外。
本版先解决“语义和配置口径”，不是要求一次性重构底层实现。能映射到现有 GAS 字段的先映射，不能映射的放入窗口规则或程序备注。
配置方向必须明确： “技能期间不能开火”和“开火期间不能放技能”是两个方向。前者应让开火技能在启动时被技能 Tag 阻挡；后者才是让技能在启动时被开火 Tag 阻挡。

3. 术语与规则

策划可以使用“互斥”这个词讨论需求，但正式配置和验收时必须拆成具体结果：是新技能被阻挡，还是新技能启动并打断旧技能。

术语 类型 策划定义 验收口径
阻挡 结果 技能 A 正在释放时，技能 B 输入进来，但 B 不能启动。 A 继续，B 不启动；是否缓存由全局输入策略决定。
打断 结果 B 可以启动，并结束当前正在释放的 A。 B 成功激活，A 被取消或结束。
并行 结果 B 可以启动，A 继续存在。 A 和 B 同时有效，表现层也不能冲突。
缓存 全局输入策略 B 因阻挡不能启动时，系统短时间保留这次输入。 阻挡原因解除后重新尝试激活 B；技能只配置例外，不配置完整关系矩阵。
不可打断 窗口属性 A 当前窗口不接受普通打断。 普通打断规则不能结束 A；B 是否被阻挡由关系判定，缓存由全局策略判定。
互斥 策划讨论词 A 和 B 不能稳定共存。 正式配置时必须落为“阻挡 B”或“B 打断 A”。

5. 配置需求

推荐使用“全局策略 + 技能卡 + 窗口规则 + 表现协议”的配置方式。这里先说明各层各管什么，具体怎么填看后面的“策划最终配置口径”。

静态 GAS 字段的配置方向：
  • ActivationBlockedTags 配在“来袭技能 B”上，含义是：B 启动时，如果角色身上已有 A 的 Tag，则 B 被阻挡。
  • CancelAbilitiesWithTag 也配在“来袭技能 B”上，含义是：B 启动成功后，取消身上带 A Tag 的旧技能。
  • 因此“技能期间不能开火”应配置在开火技能上：Attack.ActivationBlockedTags += 技能Tag。

配置分层

配置层 负责内容 不负责内容
全局策略 统一默认值，例如缓存时长、输入槽规则、判定顺序。 不按技能对技能维护特殊表。
技能卡 每个技能自己的默认关系：谁阻挡我、我打断谁、我和谁并行。 不写模糊的“互斥”，不假定并行表现一定成立。
窗口规则 Startup / Active / Recovery 中少量不同于默认关系的例外。 不替代整段 Ability 的默认关系。
缓存例外 默认走全局策略；只有特殊技能或窗口声明能缓存、不能缓存、改写时长。 不作为阻挡、打断、并行之外的第四种关系。
表现协议 并行成立后的动画、移动、镜头、武器和特效验收。 不反过来决定逻辑关系；表现不稳时回退到阻挡或打断。

Montage 式窗口关系图

技能关系的细颗粒度不是“技能 A 和技能 B 是否互斥”这么简单，而是“技能 A 当前处在哪个时间窗口时，来袭技能 B 应该得到什么结果”。

静态 GAS 字段只能表达“整段 Ability Active 期间”的默认关系。如果同一个来袭输入在 Startup、Active、Recovery 的结果不同，就需要窗口 Tag、AnimNotifyState 或统一求值器支持窗口级覆盖。

全局缓存策略

缓存不是第四种技能关系，也不需要做成“技能 A 对技能 B 是否缓存”的完整矩阵。关系求值只回答阻挡、打断、并行；当结果是阻挡时，系统再按全局输入缓存策略处理这次输入。技能卡只声明少量例外。

策略项 建议口径
触发前提 来袭技能被阻挡，且输入来源属于可缓存输入。
默认窗口 系统层统一短窗口，例如 100-180ms；首版建议先定一个全局值。
缓存槽 默认每个输入通道只保留最后一次有效输入，避免旧输入连锁触发。
重试时机 阻挡 Tag 或窗口解除后，在统一输入刷新点重新尝试激活。
重试校验 必须重新检查冷却、消耗、目标、距离、状态和服务器权限。
例外来源 技能卡或窗口规则只声明：能缓存、不能缓存，或覆盖缓存时长。
不要做成关系矩阵： “Dash 被 Bomb 阻挡时能缓存、Attack 被 Bomb 阻挡时不能缓存”这类需求，落点应是 Dash / Attack 自己的缓存例外，或 Bomb 某个窗口的缓存例外，而不是给每一对技能新增一个缓存关系。

策划最终配置口径

策划最终不是直接写“互斥”或“并行感觉可以”，而是按“来袭技能 B 怎么处理当前技能 A”的方向，给每个技能填三类静态关系；缓存另按全局策略处理，只在技能或窗口上记录少量例外。

策划要填的内容 问题口径 程序落点 例子
阻挡我启动的技能 我这个技能 B 输入时，如果 A 正在释放，我是否不能启动？ B.ActivationBlockedTags += ATag 技能期间不能开火：在 Attack 的阻挡列表加入该技能 Tag。
我会打断的技能 我这个技能 B 成功启动后，是否要结束当前 A？ B.CancelAbilitiesWithTag += ATag Dash 打断当前开火：在 Dash 的打断列表加入 Attack Tag。
我允许并行的技能 我这个技能 B 和当前 A 是否允许同时 active？ 不互相配置 Block / Cancel，并登记表现层验收。 FastFire Active + Dash 并行：逻辑允许后，还要验 Overlay、武器和移动。
缓存例外声明 我这个技能是否需要覆盖全局缓存默认策略？ 技能卡缓存例外 / 窗口缓存例外。 全局能缓存 120ms；Dash 可声明能缓存；Attack 可声明不能缓存；特殊窗口可改写缓存时长。
窗口例外 A 在 Startup / Active / Recovery 时，对 B 是否和默认关系不同？ 窗口 Tag / AnimNotifyState / 统一求值器。 RadicalFire Startup 阻挡 Attack，Recovery 可允许 Attack 接续。
推荐填写顺序：
  1. 先按整段 Ability 填默认关系：谁阻挡我、我打断谁、我和谁并行。
  2. 再只挑关键手感补窗口例外：Startup 保护、Active 并行、Recovery 可接续。
  3. 最后配置缓存全局默认值，并补技能或窗口例外：能缓存、不能缓存、改写时长。
  4. 所有“并行”关系单独进入《技能并行表现层协议》验收，不在主关系表里假定成立。
最小交付标准： 每个技能至少要能回答三类关系和一类例外：谁会阻挡我启动；我启动后会打断谁；我和谁允许并行；我是否有缓存例外。缓存没有例外时走全局默认，答不出来的关系默认不要开放并行，也不要写成模糊的“互斥”。

7. 表现与交互要求

技能语义不是只影响能不能激活，也会影响输入响应、动画连续性、移动控制权和镜头稳定性。

输入手感

  • 缓存不是逐对技能关系，而是所有阻挡结果共享的输入策略。
  • 位移、反制、普攻等高频输入需要确认全局默认和少量例外。
  • 缓存命中时必须重新校验冷却、消耗、目标、距离、状态。

表现层并行

  • 逻辑并行不等于动作一定正确。
  • 正式配置为并行的组合，需要验证动画、Overlay、RootMotion、Camera。
  • 表现层不稳定时，应回退为阻挡或打断，不让程序临场兜底。
PVP 手感优先级建议：先保证输入响应一致，再保证强技能起手保护，最后再开放并行和高级穿插。

8. 兼容与迁移

当前项目能力 迁移要求

CancelAbilitiesWithTag 继续用于“新技能启动后打断旧技能”的关系。
HeroAbility 层输入与特判 需要把历史特判列入技能卡，逐步迁移到统一语义。

Montage / Overlay / Movement / Camera 模块 所有“并行”组合必须经过表现层验收，不能只看 GAS 是否允许。
Holding / 蓄力 / 持续技能 先记录现有取消逻辑，再决定哪些关系进入窗口规则。

9. 优先级与验收

优先级 需求 验收标准
P0 统一术语和判定顺序。 任意两个技能相遇时，能明确回答关系结果是阻挡、打断或并行；阻挡后的输入缓存由全局策略处理。
P0 静态关系接入现有 GAS 字段。 阻挡和打断能通过现有配置稳定生效，不依赖新增特判。
P1 关键窗口支持不可打断和并行例外。 强技能 Startup 可保护；持续开火 Active 可与 Dash 并行。
P1 输入缓存统一策略。 短窗口内输入不丢失，过期输入不会乱触发；缓存例外不需要维护技能对技能矩阵。
P2 表现层并行标准。 正式并行组合不出现动作瞬断、根运动冲突、镜头突跳、武器状态错乱。
暂不做 全技能全窗口矩阵、技能对技能缓存矩阵和完整编辑器工具。 后续版本再评估。

Skill - 单体施法选取 V2

降低单体技能选人负担，统一预选、锁定、释放决议和失效反馈。

回到首页 打开原文件
效果提升

让点人技能在移动、瞄准、目标丢失、服务器校验下都保持同一套选择逻辑。

1. 背景与目标

单体目标技能既要顺手，也要可控。系统可以在前段持续找候选目标、给 UI 提示、减少空放；但进入施法决议后，不能把玩家已表达的目标意图改成另一种语义。

核心规则

系统可以帮助玩家稳定锁住“已经想选的人”，不能在目标失效时自动发明一个新目标或自动切成自施法。

2. 本版范围

本版要做

  • 常驻软锁定：空闲时维护最优候选目标，用于 UI 和点按快放。
  • 武装目标：技能按下进入 Holding 时锁存当前目标。
  • 释放决议：Release 时优先读取武装目标，不重新按最近目标计算。
  • 宽容时间：处理短暂出框、遮挡、目标边缘抖动。
  • 技能级配置：不同技能通过 Profile 与 ResolvePolicy 表达差异。
  • UI 与网络验收：目标状态、失效原因、服务器校验口径明确。

本版不做

  • 不把普通施法默认等价为自动自施法。
  • 不允许每个技能在蓝图里重复写一套目标搜索逻辑。
  • 不把宽容时间用于重新创造新目标。
  • 不覆盖范围技能、地面点选、锁敌相机、辅助瞄准完整方案。

3. 术语

术语 定义 使用边界
候选目标 被采集到、可能成为技能目标的 Actor。 进入评分前必须先过硬过滤。
硬过滤 阵营、死亡、距离、不可选中、LOS、技能目标类型等硬条件。 不满足则直接出局，不参与评分。
Soft Target 空闲或非施法阶段的当前推荐目标。 用于提示和点按快放，不代表最终施法一定成功。
Armed Target 技能按下或进入 Holding 后锁存的目标。 Release 时优先读取；目标失效后按 ResolvePolicy 处理。
LockProtectTime 新锁定目标的短暂保护时间。 防止刚锁上就被相邻目标抢走。
LostGraceTime 目标短暂丢失后的保持窗口。 保 UI 和锁定感，不生成新目标。
SwitchCooldown 自动切换目标后的冷却窗口。 防止 A/B 抖动；手动切换可覆盖。
ResolvePolicy Release 时目标无效的收口规则。 必须技能级显式配置。

4. 行为规则

阶段 系统行为 禁止行为
空闲 持续采集候选，经过硬过滤和评分后更新 Soft Target。 把 Soft Target 当成已经完成施法决议。
点按快放 按下时读取当前 Soft Target，并立即进入 Resolve。 Release 或执行帧再重算最近目标。
按住 / 蓄力 进入 Holding 时锁存 Armed Target；Holding 期间显示更强锁定反馈。 目标短暂丢失时偷偷换成另一个目标。
手动切换 玩家明确输入切换目标时，可覆盖 SwitchCooldown。 自动切换压过明确手动输入。
目标短暂丢失 LostGraceTime 内保持 UI 与 Armed 状态，并标记为 Grace。 Grace 结束后自动跳到下一个目标或自动自施法。
Release 读取 Armed Target，做服务器可验证的合法性检查，再按 ResolvePolicy 处理。 用“最近友军 / 最近敌人”替代已锁存目标。

默认红线

普通施法路径下，无合法外部目标不能默认转成给自己。自施法必须来自明确输入、modifier、双击或技能级例外配置。

5. 运行流程

目标选择分成两段：前段负责“当前推荐谁”，后段负责“这次施法按什么目标结算”。两段共用 Profile，但职责不同。

6. 配置需求

目标规则统一放入 TargetSelectionProfile，技能只引用 Profile，并选择自己的 ResolvePolicy。

6.1 TargetSelectionProfile

字段 用途 备注
AllowedTargetTeam / Type 允许友军、敌人、自身、召唤物、建筑等目标类型。 不满足则硬过滤。
Range / ScreenCone 世界距离和屏幕选取区域。 第三人称技能优先同时考虑屏幕与距离。
LineOfSightRule 是否需要视线、允许短暂遮挡的时间。 遮挡宽容只保锁定，不创造目标。
PriorityBuckets 按英雄、小兵、召唤物等分桶。 先比桶优先级，再在桶内评分。
ScoreWeights 屏幕中心、朝向、世界距离、范围余量、当前目标粘滞。 权重需可配置、可 Debug。
LockRetainRadius 已锁定目标维持半径。 避免边界抖动导致频繁脱锁。
LockProtectTime 新目标锁定保护时间。 建议默认 0.15s - 0.3s。
LostGraceTime 目标短暂出框 / 遮挡后的保留时间。 建议按技能类型配置。
SwitchCooldown 自动切换后的冷却。 手动切换输入可覆盖。
ResolvePolicy Release 时目标无效的处理方式。 技能级必填。

6.2 技能卡模板

字段 填写内容
SkillId 技能 ID，例如 Skill_Muriel_Shield。
SelectionProfile 引用目标选择 Profile。
InputMode 点按、按住、蓄力、二段释放。
SelfCastInput 是否存在明确自施法输入，以及输入方式。
ResolvePolicy 目标无效时失败、取消、自施法例外或同目标重校验。
UIStyle Soft、Armed、Grace、Invalid 的显示样式。

7. ResolvePolicy

Release 时目标无效必须走显式策略。策略由技能选择，中层系统不自行猜测。

策略 行为 适用场景
FailOnInvalid 目标无效则释放失败，给失败反馈。 PVP 默认策略，输入语义最稳定。
CancelCast 取消本次技能，不进入消耗或冷却，按技能配置处理。 蓄力、按住类辅助技能。
RevalidateSameTarget 只重新验证同一个 Armed Target 是否恢复合法。 短暂遮挡或出框后可能恢复的技能。
ExplicitSelfCastOnly 只有明确自施法输入时才能以自己为目标。 友军保护、治疗、护盾类技能的默认口径。
AllowSelfFallback 无目标时允许自动转为自己。 技能级特例；必须在技能卡中标记并给 UI 提示。
默认建议：友军目标技能使用 ExplicitSelfCastOnly 或 FailOnInvalid。自动自施法只能作为明确技能例外。

8. UI 与网络

8.1 UI 状态

状态 表现要求 目的
Soft 轻量框、描边或准星提示。 告诉玩家当前推荐目标。
Armed 更强锁定框、连线、目标高亮。 告诉玩家 Release 将优先读取该目标。
Grace 目标框变淡、倒计时或丢失态提示。 告诉玩家目标处于宽容保持期。
Invalid 红色短反馈、音效或 HUD 文案。 说明失败原因，避免误以为系统没响应。
SelfCast 角色脚下或 HUD 显示自施法态。 避免玩家误以为仍在指向外部目标。

8.2 网络与权威

事项 要求
客户端 可以预测 Soft / Armed UI，并随输入上报技能、ArmedTarget、SelfCastFlag。
服务器 验证目标合法性、技能范围、阵营、死亡、不可选中和 ResolvePolicy。
目标无效 服务器返回失败原因，不自动替换为其他目标。
回放 / 观战 记录最终决议目标、失败原因和自施法标记。

9. 技能示例：Muriel Shield

护盾类技能需要同时支持低输入负担和明确自施法。默认流程优先保护玩家已选友军目标，目标无效时不自动改成给自己。

项 建议配置
AllowedTargetTeam FriendlyHero，可按需求包含 FriendlySummon。
Soft Target 空闲时持续预选屏幕中心附近、距离合法、LOS 合法的友军。
Armed Target 按下技能时锁存当前友军目标。
LostGraceTime 短暂出框可保留 UI 和 Armed 状态。
ResolvePolicy ExplicitSelfCastOnly 或 FailOnInvalid，不默认自施法。
SelfCastInput 独立按键、modifier 或双击；触发时 UI 明确显示自施法态。

10. 验收

优先级 验收项 通过标准
P0 Soft / Armed / Resolve 三阶段语义一致。 Holding 后 Release 不重新按最近目标计算。
P0 普通施法不自动自施法。 友军目标失效时，不会默认把技能放给自己。
P0 LostGrace 只保锁定表现。 Grace 期间 UI 可保持，Grace 结束不自动跳新目标。
P0 服务器权威校验。 客户端预测与服务器失败原因一致，目标无效不会被服务器替换。
P1 Profile 数据化。 至少两个技能可复用同一套目标选择逻辑，只改配置。
P1 Debug 可解释。 能看到候选列表、过滤原因、评分、SoftTarget、ArmedTarget、ResolvePolicy。

03 · 技能释放

Skill - 范围施法落点

统一范围技能落点、吸附、合法性和服务器复检，提升预览可信度。

效果提升

减少“预览能放但释放失败”“边缘吸到另一层”“抛物线落点不准”等空间施法问题。

1. 背景与目标

范围施法落点如果每个技能各写一套规则，最容易出现三类问题：预览能放但释放失败、释放位置和预览位置不一致、边缘或高低差场景被吸到玩家没想选的位置。这里统一落点决议口径，让策划配置、客户端预览、服务器复检和 QA 验收说同一种语言。

一句话规则

系统可以把玩家的原始落点修正到“同一语义位置”的合法点；不能把楼上修到楼下、墙前修到墙后、目标地面修成另一个平台，也不能在失败时悄悄消耗技能。

需要解决

  • 地面范围圈、召唤物、陷阱、投射落点使用统一判定流程。
  • 预览表现和正式生效位置一致，修正后的最终落点必须可见。
  • 非法原因可枚举，失败时能给 UI、网络和 QA 明确结果。

交付结果

  • PlacementProfile：技能落点配置。
  • PlacementResolver：客户端和服务端共用的决议函数。
  • ResolvePolicy 与 FailureReason：修正规则和失败原因。

边缘部署

同一桥面内可以贴边修正；桥下是另一个语义位置，不能自动掉层。

抛物线落点

Reticle、范围圈和服务器命中点必须是一回事，不能只让轨迹看起来对。

预览承诺

玩家松手前看到的状态要可信：合法就释放，修正就显示修正点，非法就明确失败。

2. 本版范围

范围 本版要求 说明
地面范围技能 必须接入统一落点流程。 圆形、扇形、矩形等范围先得到 FinalPlacement，再进入伤害或区域系统。
召唤 / 陷阱 / 放置物 必须校验占地、阻挡、导航和高度。 不能只判断中心点合法，必须判断实际 Footprint。
投射落点 / 抛物线终点 预览终点和服务器终点需要共享落点口径。 飞行过程的命中仍由投射物系统负责，本稿只定义“落到哪里”。
墙面 / 天花板 / 任意 3D 放置 不进入 V1。 需要单独的表面朝向、附着、遮挡和动画规则，避免拖慢地面技能落地。
自动替玩家选目标 不进入本系统。 锁敌、辅助瞄准、目标选择归目标选择系统；落点系统只处理空间位置是否合法。
V1 先把“地面落点可信”做稳。特殊技能可以先不接入，但接入后必须遵守同一套预览、释放和服务器复检口径。

3. 术语

术语 含义 验收口径
AimRay 玩家当前瞄准方向，通常来自镜头或准星。 客户端预览和服务端复检应能得到同类输入。
IntentPoint 由瞄准射线或最大距离投影得到的原始意图点。 用于表达玩家想点哪里，不一定合法。
SupportSurface 承载落点的地面或平台。 同一支撑面内可以修正，跨支撑面默认失败。
Footprint 技能实际占地形状，可能是点、圆、盒、胶囊或自定义形状。 放置物和范围技能必须检查完整占地。
PlacementProfile 技能落点配置，包括形状、距离、表面、修正策略和失败反馈。 策划配置入口，不在技能蓝图里散写找点逻辑。
FinalPlacement 最终用于释放、生成、伤害范围、VFX 和网络同步的位置。 正式表现必须以它为准。
FailureReason 落点失败原因。 UI、日志、QA 用同一份枚举，不用猜失败原因。

4. 落点流程

所有接入技能按同一条流水线求值。客户端预览可以高频调用，服务器释放时必须用同一套配置重新求值。

步骤 输入 输出 要求
1. 构建上下文 施法者、瞄准、技能配置、当前移动状态 PlacementInput 预览和释放使用同一类输入结构。
2. 求原始意图点 AimRay、最大距离、准星命中 IntentPoint 只表达玩家想点哪里，不在这一步做吸附。
3. 找候选面 IntentPoint 附近的地形 / Nav / 碰撞 SupportSurface 默认优先同一支撑面，不跨平台找点。
4. 校验占地 候选面、技能形状、阻挡信息 合法 / 非法原因 范围和放置物都按完整 Footprint 判断。
5. 执行修正策略 非法原因、容差配置、候选点 FinalPlacement 或 FailureReason 修正结果必须能反馈给玩家。

5. 合法性规则

6. 宽容与修正

宽容只解决“差一点”的手感问题，不改变玩家的目标语义。所有修正都需要落在同一支撑面或同一可理解区域内。

策略 行为 适用场景
Reject 非法即失败，不修正。 精准放置、强策略技能、会造成不公平误放的技能。
ClampToRange 超出距离时沿瞄准方向夹到最大距离。 普通地面范围技能。
SnapSameSurface 在同一支撑面内找最近合法点，距离不超过配置容差。 召唤、陷阱、地面圈在边缘附近的手感优化。
UseLastValidPreviewOnly 拖拽预览时短暂保留上一个合法点，释放时仍重新判定。 鼠标或摇杆抖动导致的预览闪烁。
禁止行为：跨层吸附、穿墙吸附、绕过遮挡吸附、把失败改成自动自施法、释放失败仍消耗资源或进入 CD。

8. 预览与反馈

玩家看到的位置必须接近最终结果。发生修正时，预览显示修正后的落点；发生失败时，预览进入非法状态，释放不应产生隐藏的成功或隐藏的消耗。

合法

正常颜色显示完整 footprint，释放时直接使用当前 FinalPlacement。

非法

非法状态要明确，不释放、不消耗；例外必须由技能单独配置。

已修正

吸附不能只移动中心点，还要显示修正后的完整占地范围。

状态 显示 释放结果
合法 使用正常范围圈、召唤轮廓或落点标记。 释放到当前 FinalPlacement。
已修正 显示修正后的最终落点；必要时用细线或边缘提示原始意图和最终点有偏移。 释放到修正后的 FinalPlacement。
非法 范围圈变为非法色，召唤轮廓隐藏或变红，失败原因可供 UI 文案读取。 默认不释放、不消耗、不进入 CD。
预览抖动 拖拽时可短暂保留上一个合法点，避免边缘闪烁。 释放瞬间仍按服务器规则复检。
预览可以更平滑，但不能骗人。玩家松手时看到的是哪里，正式结果就应该尽量落在哪里；如果服务器拒绝，需要返回明确原因。

9. 网络与权威

客户端负责高频预览和输入提交，服务器负责最终落点权威。客户端提交的坐标可以作为预测参考，但不能直接作为最终结果。

数据 要求
客户端提交 AimRay、施法者状态、PlacementProfileId、可选预测落点。
服务器复检 用同一 PlacementProfile 调用 PlacementResolver，输出权威结果。
成功同步 实体生成、范围判定、VFX、音效和日志都使用服务器 FinalPlacement。
失败同步 返回 FailureReason，客户端播放失败反馈，不消耗或按技能规则回滚。

10. 边界案例

边界案例用来验收“系统是否尊重玩家意图”。这些场景不要求每个技能都能成功，但要求成功、修正、失败的口径稳定。

案例 默认结果 验收重点
最大距离外一点点 可沿瞄准方向夹取到最大距离。 预览圈显示在夹取后位置，不在原始非法点闪烁。
同平台边缘轻微越界 可在容差内向平台内侧收缩。 完整 footprint 合法，而不是只有中心点合法。
桥面瞄到边缘，桥下也可放 桥面非法就失败，不自动落到桥下。 不能把同屏可见的另一个合法点当成玩家意图。
楼梯 / 斜坡 坡度和高度差在配置内则成功，否则失败。 召唤物不倾斜穿地，范围圈不漂浮。
墙角 / 门框旁放置 地面技能只认地面；墙面技能需独立 Profile。 不要从一个面偷偷转到另一个面。
抛物线落点被遮挡 按弹道解得到真实终点；如果和预览不一致，需要修正预览或失败。 reticle、落点圈、真实命中点一致。

有限内缩

轻微越界可以收回，同面内缩距离超过阈值后必须失败。

面语义一致

V1 不做墙面放置，但这个案例可以提醒后续扩展时不要提交瞬间换面。

状态变化

角色、召唤物或机关进入占地区时，服务端复检可以失败并回传原因。

11. 验收

优先级 验收项 标准
P0 预览和正式结果一致 平地、边缘、斜坡、高低差场景中，释放结果与玩家松手时看到的最终落点一致。
P0 失败不隐藏消耗 非法落点默认不生成实体、不造成伤害、不进入 CD、不消耗资源；例外必须由技能单独配置。

P1 网络校正可接受 延迟下客户端预测和服务器结果偏差可控；差异较大时能做失败或校正反馈。

当前项目已有技能模块、预览数据和范围系统，落点规则建议作为中间层接入。不要让每个技能自己重复写射线、Nav、碰撞和吸附规则。

模块 接入要求

SkillModuleSpawnMobaUnit 服务端复检后再 Spawn，生成位置以服务器 FinalPlacement 为准。

  1. 先建立 PlacementProfile、PlacementResolver、ResolvePolicy、FailureReason。

  3. 把客户端预览和服务器释放改成调用同一套 resolver。

效果提升

把移动、瞄准、技能、受击等镜头需求分层处理，保证可读性和手感一致。

1. 背景与目标

当前战斗镜头已经能支撑基础玩法，下一步重点是把镜头需求拆成可配置、可叠加、可验收的规则：什么场景切镜头，镜头变化是否影响判定，技能镜头如何和基础镜头共存。

一句话目标

玩家能稳定看清战场，输入方向可信；技能镜头有表现力，但不抢走操作权，也不把判定结果变得不可预期。

实现可以沿用当前主战斗相机链路；本轮先约定玩家体验、配置边界和验收口径。

2. 当前实现理解

当前工程已经有可用的战斗相机骨架，问题不在“从零做相机”，而在配置表达、叠加规则、技能镜头清理和调试口径还不够产品化。

模块 当前已经具备 需求层面的判断 迭代方向
主战斗镜头 有自定义 CameraManager、CameraComponent、CameraModeStack，第三人称镜头可稳定输出。 可以继续作为主骨架，先不重写。 补“角色镜头卡”和状态选择规则，让策划能知道基础参数在哪里改。
基础镜头参数 已有 FOV、Pitch 范围、TargetOffset、碰撞避让、蹲伏偏移等能力。 能力够用，但分散在不同资产和代码概念里。 收敛成 CameraProfile：基础距离、肩位、视野、Pitch、碰撞、Lag。
技能镜头 技能模块可提交 AbilityCameraData，支持 Offset、FOV、Lag、Blend、Shake。 已经能做表现，但更像“覆盖最终镜头”，规则不够显性。 定义为 AbilityCameraModifier，并补优先级、叠加策略、清除条件。
镜头反馈 CameraShake、死亡后处理已经接入本地表现。 适合做表现层，不能污染技能判定方向。 区分 GameplayView 和 PresentationView。
观战 / 回放 当前不是本版完整目标。 可以先不做，但状态边界现在要留对。 后续基于 CameraState 重建镜头，避免同步每帧最终相机。

3. 需求模型

每次镜头请求都归到一个明确层级：基础镜头负责常态视野，状态镜头负责瞄准、蹲伏、死亡等持续状态，技能镜头负责短时修饰，反馈镜头负责震屏和后处理。

核心判断

移动、瞄准、投射物、近战朝向等和判定相关的系统，优先读 GameplayView；技能镜头、震屏、死亡滤镜等表现变化，优先落在 PresentationView。两者可以相同，但不能默认混在一起。

4. 本版范围

本版要做

  • 定义基础镜头、状态镜头、技能镜头、反馈镜头的分层。
  • 补角色镜头卡、状态规则表、技能镜头卡的配置需求。
  • 明确技能镜头的叠加、优先级、自动清理和打断清理。
  • 明确 GameplayView / PresentationView 的判定边界。
  • 补调试显示与配置错误提示，方便策划验收。

本版不做

  • 不做完整电影镜头、Sequence 演出系统。
  • 不重写所有 CameraMode，当前主相机仍作为默认运行链路。
  • 不展开完整观战 / 回放协议，只预留 CameraState 边界。
  • 具体动画资产、蒙太奇资产和美术镜头曲线由对应技能或表现需求单独列出。
  • 镜头参数以策划可读、可复查、可回滚为准。

5. 场景需求

镜头需求先按场景写。每个场景都要能回答：玩家应该看见什么、是否影响操作方向、策划要配什么、程序怎么验收。

场景 玩家感受 策划配置 程序验收
普通移动 角色、前方威胁和准星区域稳定可见，转向不拖泥带水。 默认 FOV、Pitch 上下限、目标偏移、肩位、碰撞避让。 连续转身、跳跃、贴墙时无明显抖动和穿墙。
瞄准 / 精确射击 视野更集中，准星更可信，但不遮挡角色关键动作。 瞄准 FOV、肩位偏移、进入 / 退出混合时间、灵敏度联动标记。 开镜和关镜期间，投射物方向与 GameplayView 一致。
锁敌 / 横移 敌人保持在可读区域，玩家仍能感知侧向空间。 锁敌目标优先级、水平偏移、目标丢失后的恢复时间。 目标切换、丢失、死亡时镜头能稳定回退。
突进 / 翻越 / 位移 速度感增强，但落点和朝向不能让玩家迷失。 短时 FOV、拉远 / 拉近、Lag、Blend、是否跟随位移方向。 高速位移中镜头不穿墙，技能结束后能回到正确基础状态。
技能释放 重招、蓄力、近战命中有镜头强调，但不打断玩家读场。 技能镜头卡：触发时点、Offset、FOV、Lag、Shake、持续时间、清除条件。 技能正常结束、被打断、角色死亡时，镜头请求都会清理。
命中 / 受击反馈 有冲击感，但不影响下一次瞄准和移动判断。 Shake 强度、方向、衰减，是否叠加屏幕后处理。 Shake 只影响 PresentationView，不改变核心判定方向。
贴墙 / 狭窄空间 镜头自动收近或避让，角色和威胁仍可读。 碰撞半径、预测避让、收近 / 放回时间。 墙角、门框、坡面连续测试无大幅跳变。
死亡 / 复活 死亡状态明确，复活后迅速回到可操作镜头。 死亡后处理、FOV、镜头是否锁定目标、复活恢复时间。 死亡和复活不会残留技能镜头或 Shake 状态。

6. 镜头语义

后续配置和验收统一使用下面这些词，避免“切镜头”“加镜头”“震一下”在不同人嘴里指向不同实现。

术语 定义 策划写法 程序关注点
基础镜头 角色没有特殊状态时的第三人称镜头。 默认距离、FOV、Pitch、肩位。 永远有回退值，不允许空状态。
状态镜头 由瞄准、蹲伏、锁敌、死亡等状态触发的镜头规则。 进入状态 A 时切到 Profile B，退出时回退。 需要状态优先级和互斥关系。
技能镜头修饰 技能期间短时改变 Offset、FOV、Lag 或 Shake。 技能某窗口内加一个 modifier。 必须有持续时间、清除条件、叠加策略。
反馈镜头 命中、受击、死亡等带来的 Shake、后处理、短时冲击。 本地表现反馈，不参与判定。 默认 owner-only，不能污染 GameplayView。
GameplayView 投射物、近战朝向、辅助瞄准等使用的可信视角基准。 需要稳定，不被震屏和演出镜头带偏。 要和 PresentationView 分离或至少明确采样时机。
PresentationView 玩家最终看到的画面，可以叠加表现层。 可以服务手感和冲击，但不能制造误判。 Shake、后处理、技能镜头大多落在这一层。
清除条件 镜头请求什么时候结束。 技能结束、窗口结束、被打断、死亡、换英雄。 需要兜底，避免 modifier 残留。

7. 配置需求

镜头配置按“角色、状态、技能”三种粒度组织。角色决定默认视野，状态决定持续变化，技能决定短时修饰。

7.1 角色镜头卡

字段 用途 备注
角色 / 相机模板 指定该角色默认使用哪套基础镜头。 英雄之间可以继承同一模板，再局部覆盖。
默认 FOV / Pitch 决定默认视野和上下视角限制。 需要 Clamp，非法值给 warning。
目标偏移 / 肩位 决定角色在屏幕中的位置。 射击角色通常需要稳定准星空间。
碰撞避让 处理贴墙、角落、狭窄空间。 需要测试收近和恢复是否平滑。

状态 触发条件 镜头变化 优先级 / 退出
Normal 无特殊状态。 使用默认 Profile。 最低优先级，所有状态退出后回到这里。

字段 用途 示例

8. 叠加规则

镜头最怕“看起来能配，其实互相覆盖”。本版需要把叠加顺序写死成规则，策划只在规则内填参数。

规则 需求说明 验收方式
基础镜头永远兜底 任何状态退出、技能结束、配置缺失，都必须回到基础镜头。 删除技能镜头配置后，角色仍能正常游戏。
状态镜头先于技能镜头 瞄准、锁敌等状态先确定当前镜头基准，技能只在其上做短时修饰。 瞄准中释放技能，技能结束后仍回到瞄准镜头。
技能镜头必须可清理 每个技能镜头请求都要有 owner、持续时间和清除条件。 技能被打断或角色死亡时，不残留 Offset / FOV / Lag。
同层冲突按优先级 同一层多个请求同时存在时，先比优先级，再按策略处理。 两个技能镜头重叠时，结果能从 debug 信息解释。
Shake 不参与判定 Shake 和后处理只影响最终表现，不改变投射物、近战朝向、移动输入。 强 Shake 下连续射击，命中方向不随画面抖动漂移。
非法配置要可见 Blend 为 0、FOV 越界、Duration 不合法、引用为空，都要有明确处理。 PIE 或日志能看到配置 warning，且不会出现 NaN 或瞬移。

9. 优先级验收

优先级 需求 验收标准
P0 明确基础镜头、状态镜头、技能镜头、反馈镜头的边界。 策划能按角色镜头卡和技能镜头卡补配置；程序能按分层判断落点。
P0 补技能镜头清理规则和非法配置保护。 技能结束、打断、死亡、超时都不会残留镜头；Blend / FOV 非法值不会导致异常画面。
P0 确认 Shake、后处理、死亡滤镜属于表现层。 投射物、近战方向、移动输入不受表现层扰动。
P1 补 CameraProfile、StateRule、AbilityModifier 三类配置资产或等价数据结构。 不改 C++ 代码即可调整普通态、瞄准态、一个技能镜头表现。
P1 实现清晰的 modifier stack 和 debug 显示。 能看到当前生效的 Profile、State、Modifier、优先级、剩余时间。
P2 为观战 / 回放预留 CameraState。 旁观端未来能基于状态重建基础镜头，不依赖同步每帧最终 transform。

至少完成 P0：需求语义统一、技能镜头不残留、表现层不污染判定、非法配置可见。P1 再做数据驱动和堆栈化，P2 留给观战和回放。

效果提升

让角色在移动中开火、施法、转向时不出现骨盆漂、上半身抢动、下半身割裂。

1. 目标

角色在移动、转向、瞄准、普攻、蓄力或释放上半身技能时，下半身应继续保持 Motion Matching 的移动稳定性，上半身则表达技能意图。

一句话目标

玩家看到的是“移动稳定 + 技能动作清晰”，系统内部仍然让 Motion Matching 使用原始可靠的移动数据做选帧。

2. 策划功能描述

场景 期望表现 不接受表现
移动中普攻 角色保持当前步态和移动方向，上半身完成开火、挥击、瞄准等表现。 骨盆被攻击动作带偏，腿部滑步，跑动节奏突然中断。
移动中释放技能 技能动作可以明显影响胸腔、肩膀、手臂和武器，但下半身继续表达移动。 上半身动作污染大腿或脚，导致身体撕裂、异常扭转。
蓄力 / 持续施法 上半身保持施法姿态，下半身根据输入继续移动、减速或停步。 持续动作循环时骨盆抖动，BlendIn / BlendOut 时姿态弹跳。
技能结束 平滑回到普通移动或上半身待机状态。 技能结束瞬间上身或骨盆明显弹回。
优先级：战斗可读性 > 移动稳定性 > 技能表现强度 > 原始动画还原度。

3. 配置需求

技能或动作资产需要能声明自己的混合语义，避免所有技能共用同一套上半身规则。

动作类型

  • 全身动作
  • 上半身动作
  • 上半身强表现动作
  • 瞄准 / 持续施法动作
  • 禁止移动中播放的动作

关键参数

  • 混合权重
  • BlendIn / BlendOut 时间
  • 起始骨骼，如 spine_01
  • 是否允许影响 pelvis
  • 是否启用移动中视觉修正

4. 技术方案

本方案不改变 Motion Matching 的移动选帧逻辑，而是在上半身混合之后，对最终显示姿态做一层受控修正。

Locomotion Motion Matching
  -> PoseSearchHistoryCollector
  -> AimOffset / Overlay / UpperBody Slot
  -> Layered Blend Per Bone
  -> UpperBody Visual Correction
  -> IK / Final Output
步骤 处理内容 说明
缓存移动姿态 在上半身混合前保留 Motion Matching 输出的 locomotion pose。 作为下半身稳定基准。
播放上半身动作 继续使用现有 UpperBody Slot、Overlay、Layered Bone Blend。 优先复用当前动画蓝图管线。
视觉修正 对 pelvis / thigh 的异常偏移做小幅还原，必要时把差异分摊到 spine 链。 目标是减少扭腰、滑步和撕裂感。
IK 收尾 在修正后再进行脚部、手部或武器对齐处理。 保证最终接地和持武表现稳定。

5. Motion Matching 边界

关键风险：如果修正后的 pelvis、thigh、foot 进入 PoseHistory 或 Motion Matching Query，下一帧可能会影响选帧，造成步态跳变、匹配漂移或移动预测不稳定。

禁止影响

  • Capsule 位移
  • CharacterMovement 速度与加速度
  • RootMotionSource
  • RootYawOffset 决策
  • Motion Matching Query
  • PoseSearchHistory / PoseHistory
  • Trajectory 预测输入

允许影响

  • 最终输出给 SkeletalMesh 的视觉姿态
  • 上半身与下半身混合后的显示结果
  • Spine / Chest / Clavicle / Arm 等技能表现骨骼
  • 仅用于视觉稳定的小幅 pelvis 修正

落地原则

Motion Matching 看到原始 locomotion pose，玩家看到修正后的最终 pose。修正结果不回写、不参与下一帧搜索、不改变真实移动。

6. 验收标准

  • 至少选择一名普通人形英雄和一名体型或武器差异较大的英雄验证。
  • 覆盖原地、前进、后退、横移、转向、连续普攻、技能打断、技能 BlendOut 等场景。
  • 移动中释放上半身动作时，无明显骨盆异常、腿部错位、滑步或身体撕裂感。
  • 技能方向、武器朝向和命中表现保持一致。
  • 开启修正前后，Motion Matching 选帧稳定性无明显变差。
  • Debug 模式下可以确认修正节点位于 PoseHistory 采集之后，且不改变移动预测输入。

效果提升

明确视觉起点、逻辑起点和瞄准意图的优先级，提升射击可信度。

1. 背景

当前 Twinblast 普攻的即时子弹在逻辑上更接近从相机射线发出，枪口 socket 主要用于视觉偏移。这样能保证准星命中感，但会带来 TPS 中常见的公平性问题：角色或武器仍在掩体后，子弹却从镜头线命中目标。

本需求的核心结论：
  • 玩家的瞄准意图由相机准星决定。
  • 实体子弹的真实出生点由武器口或逻辑枪口决定。
  • 子弹方向由枪口指向相机解析出的 AimPoint。
  • 需要支持旧逻辑兼容，不能一次性改变全项目所有投射物手感。
不要把本需求理解成“所有技能都从枪口发射”。 这是一个可配置的发射策略需求，优先服务 Twinblast 普攻和同类 TPS 实体子弹。

2. 范围

本文档只定义实体子弹发射起点与准星意图的需求，不替代动画、武器绑定、AO、IK、命中伤害和网络预测文档。

LifeTime Projectile P1 逻辑相似，但生命周期、预测和碰撞成本更高，建议第二阶段接入。

效果提升

1. 背景与目标

当前命中特效配置分散在不同技能数据里。随着敌我、墙体、近战、护盾、爆头等差异增多，继续增加平铺字段会让策划配置和程序维护都变得困难。

命中对象 打到敌人、友军、墙体，还是护盾 / 部署物。 敌方单位、友方单位、世界对象、护盾或部署物。
单位类型 命中的单位具体是什么。 英雄、小兵、防御塔、召唤物、投射物。V1 可选配置。
结算结果 这次命中最终造成了什么效果。 伤害、治疗、护盾吸收、免疫、格挡。
命中部位 是否命中头部。 仅支持敌方单位的头部 / 默认两档。
表现通道 谁能看到这次表现。 本机表现、世界表现、目标身上表现。

配置回退

策划不需要填满所有组合。找不到精确规则时，程序按更通用的规则回退。

敌方 + 头部 + 伤害
  -> 敌方 + 伤害
  -> 敌方
  -> 默认命中表现
配置要求：每个命中特效配置必须有默认表现。特殊规则只配置真正需要差异的场景。

3. 支持场景

以下场景是第一版需要覆盖的设计范围。

场景 需求 备注
激光命中敌人 可配置敌方命中特效、命中音效、本机反馈。 兼容现有 LaserSweep 敌方特效字段。
激光命中友军 可配置友军命中特效，适用于治疗、护盾、增益等技能。 是否能命中友军由技能碰撞和过滤规则决定。
瞬时投射物命中敌人 可配置敌方命中特效。本机可以播放命中确认反馈。 兼容现有单一 HitParticle。
瞬时投射物命中世界 可配置墙体 / 地面命中特效。 V1 不区分金属、石头、木头。
近战窗口命中敌人 可像投射物一样配置命中特效、音效和本机反馈。 近战属于“技能命中窗口”的一种，不单独建系统。
护盾吸收 / 免疫 / 格挡 需要能配置不同于普通伤害的反馈。 第一版先保留配置能力，具体接入可分阶段。

Steel 普攻样例

  • HAD_SteelAttack 的 A / B 段普攻属于近战命中窗口。
  • 命中敌方时播放近战命中特效；挥空时不播放命中反馈。
  • 本机可以立即播放轻量命中反馈，但伤害数字以服务器结算结果为准。
  • 后续需要支持 A / B 段配置不同命中特效。

4. 表现要求

同一次命中需要区分本机看到的反馈、其他人看到的反馈，以及目标身上的反馈。

表现通道 用途 例子
本机表现 给释放者看的即时反馈，优先保证手感和清晰度。 准星反馈、轻量命中粒子、相机轻微震动、命中音效。
世界表现 其他玩家在场景中能看到的命中效果。 受击火花、能量爆点、墙体命中粒子、命中音效。
目标身上表现 挂在被命中目标上的确认反馈。 受击闪光、护盾波纹、免疫 / 格挡提示表现。
UI 反馈 和结算结果相关的反馈。 伤害数字、治疗数字、暴击、击杀、护盾吸收。
GameplayCue 使用方式： 策划可以把 GameplayCue 作为一种表现资源配置。程序需要保证同一套配置能区分本机表现和世界表现，并避免本机预测与服务器确认重复播放。

5. 兼容与迁移

当前内容 迁移要求

6. 优先级与验收

优先级 需求 验收标准
P0 统一命中特效配置入口和默认回退。 缺少特殊规则时稳定播放默认命中特效，不报错、不丢表现。
P0 兼容 LaserSweep 和 InstantProjectile 旧配置。 现有激光和瞬时投射物命中特效不退化。
P1 接入 SkillEffectWindow，并以 Steel 普攻验证近战命中。 Steel 普攻击中敌方播放命中特效；挥空不播放命中反馈。
P1 支持本机表现和世界表现分流。 释放者能看到本机反馈，其他客户端能看到世界表现，且不重复播放。
P2 支持护盾吸收、免疫、格挡等差异反馈。 可配置不同于普通伤害的表现。
暂不做 复杂表面材质和细分身体部位。 后续版本再评估。
