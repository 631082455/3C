1. 设计取向：为什么是 Deadlock 派

维度Deadlock 派（选）Valorant 派（不选）
落点不合法时Clamp 到最近合法点红色预览，按键无效
资源/CD总是消耗失败时不消耗
LineOfSightemergent（由相机射线几何自然涌现）显式 LOS trace
失败原因 / Reject不需要需要 9 值枚举
GAS Commit 时序拦截不需要（无失败路径）需要（失败不消耗）

选 Deadlock 派的 4 个理由:

  • 项目当前真实行为已经是 Deadlock 派：所有 SkillModule 服务端裸信任客户端，SpawnCollisionHandlingOverride = AlwaysSpawn。V1.2 只是把这个隐性行为显化为规范
  • 实现工作量小：省掉"失败原因 + GAS Commit 时序集成 + 客户端预测回滚"整条链路
  • 手感更稳定：玩家不会被"明明能瞄到但不让放"惊讶
  • 对应项目调性：参照 Deadlock / Paragon 风格的 TPS MOBA，技能设计本身避免了"必须精准放在某处"的范式

4. 与 V2.1 TSC 系统的边界

本系统（范围施法 V1）与已上线的 V2.1 单体施法选取（TSC）是正交关系，互不影响。 证据：SkillModuleApplyBuffToTarget.cpp:45-73 的 ArmedTarget 分支完全不读 AreaPreviewData, 直接 TSC->GetCurrentTarget()。

  • TSC 管：锁敌、辅助瞄准、单体目标合法性校验（team / dead / range / LOS，见 ADR-0003）
  • 本系统管：空间位置（地面落点）
  • 3 个已迁 TSC 的技能（KallariMark / MurielShield / TwinblastBomb）对本 V1 完全免疫

5. 风险点

R1:TargetData 协议加 AimRay 字段的兼容性
FSpawnMobaUnitTargetData / FSkillAreaSpawnTargetData / FTeleportToTargetData 都要加 FVector_NetQuantize AimOrigin + FVector_NetQuantizeNormal AimDirection 字段。 UE FStruct NetSerialize 在末尾加字段一般兼容，但客户端服务端必须同版本部署（混版本会出错）。 如果有热更新需求需要先确认。
R2:SummonPreview pick-further bug 修复 = 行为变化
修复后水平瞄墙不再穿过去。7 个使用 SummonPreview 的技能(BladeShower / StormArrow / GideonMeteor / PhaseThunderCloud / MurielBomb / SkyfireMechaUlt / BladeShower_Test)需要 QA 重点回归。 如果某个技能策划故意依赖"穿墙放置"行为，需要单独决定是否给 Profile 加 bAllowAimThroughObstacles（V1 不主动加，V2+ 再说）。
R3：阈值 200cm 写死代码全局常量
放在 UPlacementResolverLibrary::MaxServerClientDeviation。如果未来某个技能（比如远距离召唤） 需要更大阈值，需要先重构成 Profile 字段。V1 不主动加，等真实需求出现。
R4:AimRay 严格反作弊漏洞（V1 接受，V2 补）
V1 服务端复算用客户端发来的 AimRay。cheat 工具如果同时伪造 AimRay + 坐标（让两者"自洽"）可以绕过阈值。 V2 需要改用服务端 ControlRotation 或加 AimRay 合理性校验。
