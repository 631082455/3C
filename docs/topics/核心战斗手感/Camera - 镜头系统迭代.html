Planning Requirement

镜头迭代需求（战斗与3C）

面向第三人称 MOBA / 射击战斗的镜头需求。核心目标是保证移动、瞄准、技能、受击、死亡等战斗场景下的视野稳定、输入可信、表现可控，并为程序提供明确的配置口、叠加规则和验收标准。

需求方：3C / 战斗策划 实现方：战斗程序 / 客户端表现 项目基线：当前自定义战斗相机 重点：视野信息、输入可信、技能表现

1. 背景与目标

当前战斗镜头已经能支撑基础玩法，下一步重点是把镜头需求拆成可配置、可叠加、可验收的规则：什么场景切镜头，镜头变化是否影响判定，技能镜头如何和基础镜头共存。

一句话目标

玩家能稳定看清战场，输入方向可信；技能镜头有表现力，但不抢走操作权，也不把判定结果变得不可预期。

实现可以沿用当前主战斗相机链路；本轮先约定玩家体验、配置边界和验收口径。

2. 当前实现理解

当前工程已经有可用的战斗相机骨架，问题不在“从零做相机”，而在配置表达、叠加规则、技能镜头清理和调试口径还不够产品化。

模块 当前已经具备 需求层面的判断 迭代方向
主战斗镜头 有自定义 CameraManager、CameraComponent、CameraModeStack，第三人称镜头可稳定输出。 可以继续作为主骨架，先不重写。 补“角色镜头卡”和状态选择规则，让策划能知道基础参数在哪里改。
基础镜头参数 已有 FOV、Pitch 范围、TargetOffset、碰撞避让、蹲伏偏移等能力。 能力够用，但分散在不同资产和代码概念里。 收敛成 CameraProfile：基础距离、肩位、视野、Pitch、碰撞、Lag。
技能镜头 技能模块可提交 AbilityCameraData，支持 Offset、FOV、Lag、Blend、Shake。 已经能做表现，但更像“覆盖最终镜头”，规则不够显性。 定义为 AbilityCameraModifier，并补优先级、叠加策略、清除条件。
镜头反馈 CameraShake、死亡后处理已经接入本地表现。 适合做表现层，不能污染技能判定方向。 区分 GameplayView 和 PresentationView。
观战 / 回放 当前不是本版完整目标。 可以先不做，但状态边界现在要留对。 后续基于 CameraState 重建镜头，避免同步每帧最终相机。

3. 需求模型

每次镜头请求都归到一个明确层级：基础镜头负责常态视野，状态镜头负责瞄准、蹲伏、死亡等持续状态，技能镜头负责短时修饰，反馈镜头负责震屏和后处理。

核心判断

移动、瞄准、投射物、近战朝向等和判定相关的系统，优先读 GameplayView；技能镜头、震屏、死亡滤镜等表现变化，优先落在 PresentationView。两者可以相同，但不能默认混在一起。

4. 本版范围

本版要做

  • 定义基础镜头、状态镜头、技能镜头、反馈镜头的分层。
  • 补角色镜头卡、状态规则表、技能镜头卡的配置需求。
  • 明确技能镜头的叠加、优先级、自动清理和打断清理。
  • 明确 GameplayView / PresentationView 的判定边界。
  • 补调试显示与配置错误提示，方便策划验收。

本版不做

  • 不做完整电影镜头、Sequence 演出系统。
  • 不重写所有 CameraMode，当前主相机仍作为默认运行链路。
  • 不展开完整观战 / 回放协议，只预留 CameraState 边界。
  • 具体动画资产、蒙太奇资产和美术镜头曲线由对应技能或表现需求单独列出。
  • 镜头参数以策划可读、可复查、可回滚为准。

5. 场景需求

镜头需求先按场景写。每个场景都要能回答：玩家应该看见什么、是否影响操作方向、策划要配什么、程序怎么验收。

场景 玩家感受 策划配置 程序验收
普通移动 角色、前方威胁和准星区域稳定可见，转向不拖泥带水。 默认 FOV、Pitch 上下限、目标偏移、肩位、碰撞避让。 连续转身、跳跃、贴墙时无明显抖动和穿墙。
瞄准 / 精确射击 视野更集中，准星更可信，但不遮挡角色关键动作。 瞄准 FOV、肩位偏移、进入 / 退出混合时间、灵敏度联动标记。 开镜和关镜期间，投射物方向与 GameplayView 一致。
锁敌 / 横移 敌人保持在可读区域，玩家仍能感知侧向空间。 锁敌目标优先级、水平偏移、目标丢失后的恢复时间。 目标切换、丢失、死亡时镜头能稳定回退。
突进 / 翻越 / 位移 速度感增强，但落点和朝向不能让玩家迷失。 短时 FOV、拉远 / 拉近、Lag、Blend、是否跟随位移方向。 高速位移中镜头不穿墙，技能结束后能回到正确基础状态。
技能释放 重招、蓄力、近战命中有镜头强调，但不打断玩家读场。 技能镜头卡：触发时点、Offset、FOV、Lag、Shake、持续时间、清除条件。 技能正常结束、被打断、角色死亡时，镜头请求都会清理。
命中 / 受击反馈 有冲击感，但不影响下一次瞄准和移动判断。 Shake 强度、方向、衰减，是否叠加屏幕后处理。 Shake 只影响 PresentationView，不改变核心判定方向。
贴墙 / 狭窄空间 镜头自动收近或避让，角色和威胁仍可读。 碰撞半径、预测避让、收近 / 放回时间。 墙角、门框、坡面连续测试无大幅跳变。
死亡 / 复活 死亡状态明确，复活后迅速回到可操作镜头。 死亡后处理、FOV、镜头是否锁定目标、复活恢复时间。 死亡和复活不会残留技能镜头或 Shake 状态。

6. 镜头语义

后续配置和验收统一使用下面这些词，避免“切镜头”“加镜头”“震一下”在不同人嘴里指向不同实现。

术语 定义 策划写法 程序关注点
基础镜头 角色没有特殊状态时的第三人称镜头。 默认距离、FOV、Pitch、肩位。 永远有回退值，不允许空状态。
状态镜头 由瞄准、蹲伏、锁敌、死亡等状态触发的镜头规则。 进入状态 A 时切到 Profile B，退出时回退。 需要状态优先级和互斥关系。
技能镜头修饰 技能期间短时改变 Offset、FOV、Lag 或 Shake。 技能某窗口内加一个 modifier。 必须有持续时间、清除条件、叠加策略。
反馈镜头 命中、受击、死亡等带来的 Shake、后处理、短时冲击。 本地表现反馈，不参与判定。 默认 owner-only，不能污染 GameplayView。
GameplayView 投射物、近战朝向、辅助瞄准等使用的可信视角基准。 需要稳定，不被震屏和演出镜头带偏。 要和 PresentationView 分离或至少明确采样时机。
PresentationView 玩家最终看到的画面，可以叠加表现层。 可以服务手感和冲击，但不能制造误判。 Shake、后处理、技能镜头大多落在这一层。
清除条件 镜头请求什么时候结束。 技能结束、窗口结束、被打断、死亡、换英雄。 需要兜底，避免 modifier 残留。

7. 配置需求

镜头配置按“角色、状态、技能”三种粒度组织。角色决定默认视野，状态决定持续变化，技能决定短时修饰。

7.1 角色镜头卡

字段 用途 备注
角色 / 相机模板 指定该角色默认使用哪套基础镜头。 英雄之间可以继承同一模板，再局部覆盖。
默认 FOV / Pitch 决定默认视野和上下视角限制。 需要 Clamp，非法值给 warning。
目标偏移 / 肩位 决定角色在屏幕中的位置。 射击角色通常需要稳定准星空间。
碰撞避让 处理贴墙、角落、狭窄空间。 需要测试收近和恢复是否平滑。
基础 Lag 控制跟随柔和度。 竞技操作优先低延迟，表现型状态再单独加。

7.2 状态规则表

状态 触发条件 镜头变化 优先级 / 退出
Normal 无特殊状态。 使用默认 Profile。 最低优先级，所有状态退出后回到这里。
Aim 按住瞄准、进入精确射击。 FOV 收窄，肩位微调，降低表现 Lag。 高于 Normal，低于 Death / 强制演出。
LockTarget 锁定有效目标。 保持目标读场，可给水平偏移。 目标无效后按恢复时间退出。
Death 角色死亡或等待复活。 死亡后处理、可能切观察目标。 最高优先级，并清理技能镜头。

7.3 技能镜头卡

字段 用途 示例
技能 ID / 窗口 说明哪个技能、哪个时段触发镜头。 HAD_SteelAttack：出手前拉近，命中帧 Shake。
Offset / FOV 短时调整构图和速度感。 重击可轻微拉近并降低 FOV；位移可短时拉远。
Lag 控制技能期间镜头跟随柔和度。 突进可加位置 Lag，但精确射击不建议加。
Shake / 后处理 只表达冲击反馈。 命中帧 0.12s 轻震，强度按技能等级或距离缩放。
Blend / Duration 进入、退出和最长持续时间。 所有 0 值和负值都要有明确含义或报错。
清除条件 防止技能镜头残留。 技能结束、被打断、死亡、切换 Pawn、超时。
叠加策略 多个镜头请求同时存在时如何处理。 Override、Additive、Max、Min、IgnoreLowerPriority。

8. 叠加规则

镜头最怕“看起来能配，其实互相覆盖”。本版需要把叠加顺序写死成规则，策划只在规则内填参数。

规则 需求说明 验收方式
基础镜头永远兜底 任何状态退出、技能结束、配置缺失，都必须回到基础镜头。 删除技能镜头配置后，角色仍能正常游戏。
状态镜头先于技能镜头 瞄准、锁敌等状态先确定当前镜头基准，技能只在其上做短时修饰。 瞄准中释放技能，技能结束后仍回到瞄准镜头。
技能镜头必须可清理 每个技能镜头请求都要有 owner、持续时间和清除条件。 技能被打断或角色死亡时，不残留 Offset / FOV / Lag。
同层冲突按优先级 同一层多个请求同时存在时，先比优先级，再按策略处理。 两个技能镜头重叠时，结果能从 debug 信息解释。
Shake 不参与判定 Shake 和后处理只影响最终表现，不改变投射物、近战朝向、移动输入。 强 Shake 下连续射击，命中方向不随画面抖动漂移。
非法配置要可见 Blend 为 0、FOV 越界、Duration 不合法、引用为空，都要有明确处理。 PIE 或日志能看到配置 warning，且不会出现 NaN 或瞬移。

9. 优先级验收

优先级 需求 验收标准
P0 明确基础镜头、状态镜头、技能镜头、反馈镜头的边界。 策划能按角色镜头卡和技能镜头卡补配置；程序能按分层判断落点。
P0 补技能镜头清理规则和非法配置保护。 技能结束、打断、死亡、超时都不会残留镜头；Blend / FOV 非法值不会导致异常画面。
P0 确认 Shake、后处理、死亡滤镜属于表现层。 投射物、近战方向、移动输入不受表现层扰动。
P1 补 CameraProfile、StateRule、AbilityModifier 三类配置资产或等价数据结构。 不改 C++ 代码即可调整普通态、瞄准态、一个技能镜头表现。
P1 实现清晰的 modifier stack 和 debug 显示。 能看到当前生效的 Profile、State、Modifier、优先级、剩余时间。
P2 为观战 / 回放预留 CameraState。 旁观端未来能基于状态重建基础镜头，不依赖同步每帧最终 transform。

最小可交付版本

至少完成 P0：需求语义统一、技能镜头不残留、表现层不污染判定、非法配置可见。P1 再做数据驱动和堆栈化，P2 留给观战和回放。

10. 程序备注

当前代码可承接以下实现点，后续实现时按需求分层逐步收敛。

当前代码 可承接的需求
AGameMatchCameraManager 仍作为主相机管理入口，负责输出最终 ViewTarget。
UGameMatchCameraComponent 承接 AbilityCameraData、死亡后处理、最终视图输出，可扩展 modifier stack。
UGameCameraModeStack 继续负责基础 CameraMode 的求值和混合。
UHeroCameraMode_ThirdPerson 承接第三人称基础镜头、碰撞避让、Pitch 曲线和基础偏移。
FSkillModuleCamera / UAbilityCameraData 短期可继续作为技能镜头入口；中期建议升级为 AbilityCameraModifier 请求。 
建议求值顺序：
CameraState
-> BaseProfile
-> StateRule
-> AbilityModifierStack
-> FeedbackLayer
-> GameplayView / PresentationView
实现节奏以低风险迭代为主：先补需求语义、配置边界和清理规则，再逐步整理数据结构和调试工具。
