1. 系统目标

为第三人称 3D MOBA 的"单体施法"提供一套统一的目标选取管线：

  • Idle 时持续维护一个"软目标"(Soft target）给玩家看（像 League/Smite 的命中体提示）
  • Holding 类技能蓄力中维护“已锁目标”（Armed target)，释放时用它结算 （结算走 SkillModuleApplyBuffToTarget / LaunchProjectile / TeleportToTarget / RootMotionSource 的 ArmedTarget 模式，见 §3.4）
  • Holding 锁敌类“无 Armed target 不放、不进 CD”由确认门 FAbilityHasTSCTargetTrigger 保证（见 §7.1)，与 tap 类的 ResolvePolicy 是两套机制
  • 支持 Alt 强制自施法（SelfCast modifier）绕过当前 Soft/Armed
  • 支持"无候选时按键 → 自动选 self"(AllowSelfFallback）语义

设计倾向：预览给玩家积极引导（粘滞、Grace、抗 jitter)，最终裁决保守 （不替玩家改变 EventData 里的 target 意图）。同 range-ground-targeting-spec.md 的取向。

2. 三态状态机

UTargetSelectionComponent（挂在本地玩家的 AMatchPlayerController 上） 维护一个三态机：

状态 进入 API 退出 API 评分 Profile 来源 含义
Idle （默认） BeginHolding / BeginSelfCast IdleDriveProfile（自动 discovery） 持续 20Hz 评分，胜出者 = Soft target
Holding BeginHolding(profile） EndHolding(） 入参的 InHoldingProfile 蓄力中，胜出者 = Armed target
SelfCast BeginSelfCast(） EndSelfCast(） （无评分） CurrentTarget = owner pawn，锁死

状态互斥 — 同时只有一个；SelfCast 优先级最高（详见 §6 时序与 §8 约束 3）。

2.1 IdleDriveProfile 怎么来

UTargetSelectionComponent::TickComponent 每帧会做一次 TryDiscoverIdleDriveProfile：

  1. 遍历 owner pawn 的 GAS 已 grant 的所有 UHeroAbility
  2. 找 AbilityDataAsset->bDrivesIdleSoftTarget == true 那一个
  3. 取它的 TargetSelectionProfile 作为 Idle 期间的评分 profile

约定：每英雄至多一个驱动源技能。资产层校验保证唯一（目前是规约， 后续可加 editor validator）。

3. 核心数据资产

3.1 UTargetSelectionPreviewDataAsset（TSP_*）

单个技能的"找谁、怎么排序"配置。完整字段表（默认值在头文件 L52–128)：

类别 字段 说明
硬过滤 AllowedTeams（bitflag） 友/敌/中立/自身，可叠加
硬过滤 AllowedUnitTypeTags（GameplayTag） 限制 unit 类型（如 Unit.Type.Hero）
硬过滤 RangeCm 距离上限
硬过滤 ScreenConeRadius（0–1） 屏幕中心半径百分比（NDC）
硬过滤 LineOfSightRule None / Required / RequiredWithGrace
评分 ScreenCenterWeight / FacingWeight / WorldDistanceWeight 主要评分项权重
评分 LockRetainBonus / LockProtectBonus / GraceBonus 粘滞类加成
粘滞 LockRetainRadius（NDC) / LockProtectTime（s) / LostGraceTime（s） 抗 jitter 节流
粘滞 SwitchCooldown（s) / SwitchScoreThreshold 主动甩枪短路
Resolve ResolvePolicy NoCastOnEmpty / AllowSelfFallback，见 §7
UI TargetIndicatorWidgetClass（soft class） 当前 target 身上挂的指示 widget
UI TargetIndicatorOffset widget 局部偏移

注： UTargetSelectionPreviewDataAsset 继承 UPreviewDataAsset 是为了进 Preview 资产家族，但不使用 FPreviewDataInstance 模型（CreatePreviewInstance(） 直接返回 nullptr）— 运行时状态全部在 UTargetSelectionComponent 上。

3.2 UHeroAbilityDataAsset（HAD_*）里的 V2.1 字段

字段 含义
TargetSelectionProfile（TObjectPtr<TSP_*>；） 引用 TSP，空 = 该技能不走单体管线
bDrivesIdleSoftTarget（bool） 标记本技能作为该英雄的 Idle 驱动源
bHoldingAbility（bool，V1 就有） 是否走 Holding 链（蓄力技能）
ConfirmHoldingSkillModules Holding 确认（释放）时跑的 module 链
CustomizeEventSkillModules（map） 由释放 montage 的 GameplayEvent notify 触发的 module 链（锁敌追踪弹常放这里，见 §5.3）
CanConfirmHoldingSkillModule（FInstancedStruct of FCanDoAbilityTriggerModule） Holding 释放时的确认门；填 FAbilityHasTSCTargetTrigger = 无 Armed target 时不确认（不放、不进 CD、不耗蓝），见 §3.4 / §7.1

3.3 UGameInputDataAsset（GameInputData_Global）里的 V2.1 字段

字段 含义 默认绑定
SelfCastModifierInputAction（UInputAction*） Alt 修饰键（全局 modifier，见 §6） IA_SelfCastModifier → LeftAlt(IMC_Default + IMC_Default_AltKeys 各一条）
CancelHoldingInputAction（UInputAction*） Holding 中取消（右键） IA_CancelHolding → RMB

注： IA_SelfCastModifier 自身 ValueType=Boolean，Triggers/Modifiers 都为空； Started/Completed 由 MatchPlayerController::Input_SelfCastModifier 维护 bSelfCastModifierActive 状态。

3.4 Armed target 结算模块（Holding 锁敌技能用）

Holding 蓄力锁定的 Armed target，最终由下面任一（可组合）结算模块消费。它们的 UpdateTargetData(） 在构建 EventData 时读 TSC.GetCurrentTarget(） 快照进 EventData.TargetData（时序见 §5.3)，Execute(） 只认快照、不再 query TSC。

结算模块 锁敌配置项 效果 范例
SkillModuleApplyBuffToTarget 目标类型 = ArmedTarget 给 Armed target 挂 buff / 伤害 Muriel 大盾（给友方）
SkillModuleLaunchProjectile ProjectileHomingType = EProjectileHomingType::BasedOnArmedTarget 子弹追踪 Armed target(Instant 投掷物每帧朝目标修正，到点强制 hit） Twinblast 手炮
SkillModuleTeleportToTarget TeleportType = ESkillTeleportToTargetType::BasedOnArmedTarget 瞬移到 Armed target 附近（可配 RelativeLocation 偏移、bFaceToTargetAfterTeleport） Kallari Mark
SkillModuleRootMotionSource RootMotionForceType = ERootMotionForceType::BasedOnArmedTarget RootMotion 冲向 Armed target（可配 DistantceToTargetToEndRootMotion 提前结束） 冲刺类位移

命名不一致（注意）： ApplyBuffToTarget 用的枚举值叫 ArmedTarget，另外三个叫 BasedOnArmedTarget，语义相同（都读 TSC.CurrentTarget）。

落点 AOE 与瞄准解耦： 追踪弹命中后的范围伤害 / buff（如手炮落点 AOE)baked 在 InstantProjectileData 的 HitSkillAreaWithBuffs 里，命中自动触发，与“怎么瞄”无关 —— 从老的区域瞄准（SAPDA）迁到锁敌时这部分不用动。

4. 接入流程（新单体技能怎么挂上 V2.1）

按技能形态分两类：4.1 tap / 自施法类（Idle Soft + ResolvePolicy）、4.2 Holding + 锁敌结算类（投掷弹 / 位移 / 冲刺）。两类都只动数据，不动代码。

4.1 tap / 自施法类（Muriel 大盾范式）

步骤：

  1. 建 TSP_<Hero><Skill>；（UTargetSelectionPreviewDataAsset 实例） - 填硬过滤、评分权重、粘滞参数（可拷贝 TSP_MurielShield 作为模板） - ResolvePolicy 选 NoCastOnEmpty 还是 AllowSelfFallback（见 §7) - TargetIndicatorWidgetClass 选通用 W_MarkTargetIndicator 或做技能专属版 （专属版必须跟 TSP 同 CL 提交，提交顺序错会引用断裂）

  2. 改 HAD_<Hero><Skill>； - TargetSelectionProfile → 上面那个 TSP - bDrivesIdleSoftTarget = true（如果是该英雄的主驱动源； 否则只在 Holding 中评分，Idle 时不显 Soft 框）

  3. GA 本身一般不动，除非： - 原本是纯 Holding 技能，想支持 Alt+键"即按即放"(tap-class 入口） → 拆出 tap 子能力（参考 GA_MurielShield 的拆分）

  4. Alt 是全局的，不需要每个 GA 配 — 只要走 UHeroAbility::ActivateAbility 就会自动消费 IsSelfCastModifierActive(）。

  5. 验证：跑 §9 的验收清单。

4.2 Holding + 锁敌结算类（Kallari Mark / Twinblast 手炮范式）

蓄力锁敌、释放结算的技能（突进、追踪弹等）。在 4.1 的基础上：

  1. 建 TSP_<Hero><Skill>； — 锁敌通常 AllowedTeams = Hostile、ResolvePolicy = NoCastOnEmpty（无目标交给确认门拦，见第 4 步）。可拷 TSP_KallariMark 作模板。

  2. 改 HAD_<Hero><Skill>； - TargetSelectionProfile → 上面的 TSP - bHoldingAbility = true - bDrivesIdleSoftTarget = false —— 按住才出敌方框，平时不常驻 （Mark / 手炮都是 false；要平时也显 Soft 框才设 true）

  3. 把结算模块切到 ArmedTarget 模式（§3.4 选一个）： - 追踪弹：ConfirmHoldingSkillModules 或 CustomizeEventSkillModules 里的 SkillModuleLaunchProjectile.ProjectileHomingType = BasedOnArmedTarget - 突进：SkillModuleTeleportToTarget.TeleportType = BasedOnArmedTarget - 冲刺：SkillModuleRootMotionSource.RootMotionForceType = BasedOnArmedTarget

  4. 挂确认门：CanConfirmHoldingSkillModule = FAbilityHasTSCTargetTrigger(） —— 无 Armed target 时释放被拦，不放、不进 CD、不耗蓝（见 §7.1）。

  5. 摘掉老的区域瞄准残留（若从 SAPDA 迁来）：删 ActivateSkillModules / ConfirmHoldingSkillModules 里的 SkillModulePreviewData（Add/Remove SAPDA）模块， 并把确认门从 FAbilityPreviewTrigger(SAPDA） 换成 FAbilityHasTSCTargetTrigger。 否则老区域 decal 会和新 soft-frame 双重 UI。SAPDA 资产本身不必删（留作回滚锚点）。

  6. 指示器：TSP.TargetIndicatorWidgetClass 选通用 W_MarkTargetIndicator，或复制 技能专属版（如手炮 W_BombTargetIndicator）—— 专属 widget 必须跟 TSP 同 CL 提交，提交顺序错会引用断裂。

  7. 验证：跑 §9 验收清单（含 Holding 专项）。

⚠ 当前已知行为：Holding 类释放后，选择框会停留到 EndAbility（释放 montage 结束，约 1s）才消失 —— indicator 隐藏目前绑在 EndHolding @ EndAbility。 “释放即隐藏”的解耦优化在排期中（只动指示器可见性，不能提前清 Armed target，原因见 §5.3 关键推论）。

5. 数据流（输入 → EventData → ActivateAbility → Execute）

下面这条时序决定了所有关键约束：

AHeroCharacter::Input_AbilityPressed
  ├─ ★ if (Alt 按下） → TSC->BeginSelfCast()      <-- 必须在这里
  ├─ FGameplayEventData ED = AbilityInstance->GetAbilityInputActivateEventData()
  │     └─ 内部跑所有 module 的 UpdateTargetData()
  │           └─ SkillModuleApplyBuffToTarget 读 TSC.CurrentTarget
  │              → 写进 ED.TargetData  (★从此定型，后续只读不写）
  │              → 如果 CurrentTarget 为空 + ResolvePolicy=AllowSelfFallback
  │                → 把 caster 喂进 TargetActors(SkillModuleApplyBuffToTarget L60-67)
  └─ ASC->TriggerAbilityFromInput(handle, &ED, idx)
        └─ replicate to server + 触发 ActivateAbility(client + server)
              ├─ 兜底再 BeginSelfCast / BeginHolding（网络/AI 路径专用，
              │   见 HeroAbility::ActivateAbility 的 module loop 前）
              └─ Module loop:buff / projectile 全部取 ED.TargetData

5.1 EventData 不可改

Execute 路径只 Get(...） InEventData.TargetData，绝不会再 query TSC 一次。 所有"修饰键改 target 选择"的逻辑必须发生在 GetAbilityInputActivateEventData(） 之前。详见 §8 约束 3。

5.2 ActivateAbility 内的兜底

HeroAbility::ActivateAbility 入口处的 SelfCast/Holding 通知块不是为了影响当帧 EventData （已经定型），而是为了： - 给生命周期延续的逻辑用（Holding 类的 next-tick ConfirmHolding 调度） - 给非 Input_AbilityPressed 触发的激活路径兜底（GameplayEvent / AI 决策）

5.3 Holding 类时序（Armed target 何时被读）

按住技能键
  └─ ActivateAbility → TSC->BeginHolding(TSP)      // 进 Holding 态，开始评分 Armed
        └─ 蓄力中：TSC 每帧维护 CurrentTarget = Armed target，指示器挂其身上
松开技能键
  └─ HeroCharacter 释放分支 → ShouldConfirmHolding?
        ├─ 确认门 CanConfirmHoldingSkillModule(FAbilityHasTSCTargetTrigger):
        │    本地端看 TSC.CurrentTarget 是否非空；空 = 不确认（不放、不进 CD)
        └─ ConfirmHoldingAbility
             ├─ 构建 ConfirmEventData:ConfirmHoldingSkillModules 各模块 UpdateTargetData
             │    └─ 结算模块读 TSC.CurrentTarget → 快照进 EventData  (★ 突进/位移在此定型）
             └─ ConfirmHoldingSkillModules.Execute（只认快照）
释放 montage 的 GameplayEvent notify（若有）
  └─ OnAbilityCustomizeSkillModuleEvent → CustomizeEventSkillModules
        ├─ UpdateTargetData:LaunchProjectile 读 TSC.CurrentTarget → 快照 HomingTarget
        │    (★ 追踪弹在此定型 —— 发生在 confirm 之后、montage 中途）
        └─ Execute：发射追踪弹
EndAbility
  └─ TSC->EndHolding()   // 退出 Holding 态；指示器在此隐藏

关键推论： Armed target 必须从 BeginHolding 一直存活到上面最后一次 UpdateTargetData 读取（追踪弹是 montage 中途，晚于 confirm）。任何“提前结束 Holding / 提前隐藏框”的优化都不能在读取前清掉 TSC.CurrentTarget，只能动指示器 可见性 —— 这正是 §4.2 末尾那个 1s 滞留优化的约束来源。

6. SelfCast modifier(Alt 强制自施法）

6.1 输入层

MatchPlayerController 在 SetupPlayerInputComponent 里给 GameInputData_Global.SelfCastModifierInputAction 绑 Started / Completed：

CurrentInputComponent->BindAction(InputDataAsset->SelfCastModifierInputAction,
    ETriggerEvent::Started,   this, &AMatchPlayerController::Input_SelfCastModifier, true);
CurrentInputComponent->BindAction(InputDataAsset->SelfCastModifierInputAction,
    ETriggerEvent::Completed, this, &AMatchPlayerController::Input_SelfCastModifier, false)；

Input_SelfCastModifier(bool） 维护 bSelfCastModifierActive，通过 IsSelfCastModifierActive(） 暴露。

6.2 消费层（两处）

早期消费（关键 fix） — HeroCharacter::Input_AbilityPressed， 在 GetAbilityInputActivateEventData(） 之前：

if (PC->IsLocalController() && PC->IsSelfCastModifierActive())
{
    if (UTargetSelectionComponent* TSC = PC->GetTargetSelectionComponent())
        TSC->BeginSelfCast();   // → CurrentTarget = owner pawn
}
// ↓ 这一行内部 snapshot TargetData 进 EventData
GetAbilityInputActivateEventData(...)；

兜底消费 — HeroAbility::ActivateAbility module loop 前：

if (PC->IsSelfCastModifierActive())
    TSC->BeginSelfCast();
else if (AbilityData->bHoldingAbility)
    TSC->BeginHolding(AbilityData->TargetSelectionProfile)；

6.3 状态化清理

HeroAbility::EndAbility 必须按当前状态分支调，不能两个都调：

if (TSC->GetState() == ETargetSelectionState::SelfCast)   TSC->EndSelfCast();
else if (TSC->GetState() == ETargetSelectionState::Holding) TSC->EndHolding()；

踩过的坑： 早期版本 SelfCast/Holding 两个都调，同帧切技能时会误关别人的 Holding indicator（详见 combat-skill-conventions.md L120 段最终方案）。

7. ResolvePolicy 语义（无候选时怎么办）

值 含义 Execute 行为 适用场景
NoCastOnEmpty（默认） 候选空 → TargetActors 保持空 Activate 已经跑过：Commit/Montage/Projectile 全执行，扣 CD/蓝、放动画；只是 ApplyBuff 没目标 = 空炮 大多数命中技能。无目标按下是策划上"允许的空炮"
AllowSelfFallback 候选空 → SkillModuleApplyBuffToTarget::UpdateTargetData 自动把 caster 喂进 TargetActors buff 落自己身上，框架原生支持（ADR-0001），GA / module 不用任何特判 增益类技能（自疗、自护盾、自 buff)；无目标按下应该"打自己"

重要： 这两个 enum 是已有的，不要为了实现“无目标→自施”自己写 CanActivateAbility gate；详见 §8 约束 2。

7.1 Holding 锁敌类的“无目标不放” = 确认门，不是 ResolvePolicy

ResolvePolicy 管的是 tap 类在 Execute 时候选为空怎么办（空炮 / 给自己）。Holding 锁敌类的“无目标不放、不进 CD”是另一套：确认门 CanConfirmHoldingSkillModule = FAbilityHasTSCTargetTrigger，在释放确认那一刻拦截 —— 本地端看 TSC.GetCurrentTarget(），空就不进确认链；非本地端信客户端 RPC 已带 TargetData （HeroAbilityTypes.cpp::FAbilityHasTSCTargetTrigger::CanDoAbilityTrigger）。

技能形态 “无目标”机制 配置位
tap / 自施法 ResolvePolicy（NoCastOnEmpty 空炮 / AllowSelfFallback 给自己） TSP.ResolvePolicy
Holding 锁敌 确认门 FAbilityHasTSCTargetTrigger（不确认 = 不放、不进 CD） HAD.CanConfirmHoldingSkillModule

8. 三条架构约束（踩坑红线）

完整版含历史方案对比、修复代码片段见 memory/combat-skill-conventions.md L90–186 "V2.1 单体目标选取 （TSC） — 关键架构约束"。下面是要点。

约束 1：CanActivateAbility 的 gate 不能依赖 TSC.CurrentTarget / TSC.State

会跟 TSC 评分死锁（TSC.Tick → CanActivate → 看 CurrentTarget → 永远 nil）。

当前防御：TSC 评分走 UHeroAbility::IsAvailableForIdleSoftTargeting(） （不是 CanActivateAbility）；该 helper 的注释里写明"禁止依赖 target"。 后续给 CanActivateAbility 加新 gate 时：

新 gate 类型 加到哪
与 target 无关（CD/消耗/沉默/等级/被动） CanActivateAbility 和 IsAvailableForIdleSoftTargeting 都加
依赖 target 只加 CanActivateAbility，绝不加 helper

约束 2:"无目标行为"优先 ResolvePolicy enum，不要新发明 gate

需求"无目标时不放空炮"或"无目标时给自己"已经在 enum 里。不要自己写 CanActivateAbility 里检查 TSC + 旁路 Alt 那一套 hack（已踩过坑，详见 memory L113–124）。如果 enum 现有 2 个值不够用，先讨论是否扩 enum，而不是写 hack。

约束 3：任何"修饰键改 target 选择"的逻辑必须在 GetAbilityInputActivateEventData 之前完成

EventData 一旦定型整条 GAS 管线（replication / buff target / projectile homing / prediction）都认那份快照。ActivateAbility 内再 BeginSelfCast 救不回来。

正确切入点：AHeroCharacter::Input_AbilityPressed，在 if (!AbilityInstance->IsActive()） 分支内、GetAbilityInputActivateEventData(） 之前。

9. 验收清单

任何新接入的单体技能必须跑全这套矩阵：

  • [ ] 有目标 → tap：打到 Soft target
  • [ ] 有目标 → Alt+tap：打到自己（不是原 Soft target）— 约束 3 高发区
  • [ ] 无目标 → tap：按 ResolvePolicy 表现（NoCastOnEmpty 空炮 / AllowSelfFallback 打自己）
  • [ ] 无目标 → Alt+tap：打到自己
  • [ ] 驱动源 ability 在 CD / 沉默 / 无蓝 → Soft 框不显示
  • [ ] Holding 类：Holding 中按 RMB 取消正常工作
  • [ ] 切换技能时：前一个技能的 Soft/Armed indicator 立刻消失，不残留
  • [ ] 队友看不到本机的 Soft 框（client-side only）

Holding 锁敌类（投掷弹 / 位移 / 冲刺）额外：

  • [ ] 面向敌方：按住 → 出敌方框；松手 → 结算命中锁定目标（突进到位 / 追踪弹命中）
  • [ ] bDrivesIdleSoftTarget=false：平时（没按）不常驻敌方框，按住才出
  • [ ] 无敌方目标按下：不放、不进 CD、不耗蓝（确认门 FAbilityHasTSCTargetTrigger）
  • [ ] 锁定目标移动：追踪弹飞行中持续跟踪（Instant homing）
  • [ ] 落点 AOE / 命中附带 buff 照常触发（与瞄准方式解耦）
  • [ ] 从 SAPDA 迁来的：老的地表区域 decal 不再出现

10. 调试与可视化

TSC 自带 DrawDebugVisualization(），由 CVar 控制（关键字搜 TargetSelectionComponent.cpp 里的 static TAutoConsoleVariable）。

打开后会在 CurrentTarget 上画 box + 名字，排查"为什么选错 / 为什么不选 / 为什么不切" 时很有用。
