1. 需求目标

1.1 背景

项目所有"落点类技能"(召唤、AOE、传送)的落点算法散落在 4 处,且服务端零校验:

模块现状

2. 设计哲学

一句话规则

玩家按下技能键,系统总是给出一个合法落点并把技能放出去。 落点合不合"理想",由系统调整(Clamp)。 玩家永远不会"按了键技能没出"。

2.1 Deadlock 派 vs Valorant 派

维度Deadlock 派(本系统采用)Valorant 派(明确不采用)
落点不合法时Clamp 到最近合法点红色预览,按键无效
资源/CD 消耗总是消耗(因为总是出技能)失败时不消耗
预览状态数2 态:合法 / 已 clamp3 态:合法 / 已修正 / 非法
LineOfSight由相机射线几何自然涌现(emergent),无显式校验显式 LOS 检测,隔墙拒绝
跨层处理不强求识别"同层",Clamp 到当前射线命中地面严格识别"同支撑面",跨层拒绝
巨大偏差时仍然出技能,玩家自己练瞄准失败,不消耗

2.2 为什么选 Deadlock 派

  • 与项目当前真实行为一致:所有 SkillModule 服务端都是"信任客户端坐标直接执行",SpawnCollisionHandlingOverride = AlwaysSpawn。本版只是把这个隐性行为显化为设计规范
  • 实现工作量小:不需要做"失败原因 → GAS Commit 时序拦截 → 客户端预测回滚"这条复杂链路
  • 手感更稳定:玩家不会被"明明能瞄到但不让放"惊讶
  • 对应项目调性:项目参照 Deadlock / Paragon 风格的 TPS MOBA,技能设计本身就避免了"必须精准放在某处"的范式

2.3 emergent LOS 已经过代码验证

grill 时验证了 MatchPlayerController.cpp:333: 真人玩家 GetPlayerViewPointForGameplay 返回相机位置(不是角色眼睛),旋转也是相机旋转。 这意味着 emergent LOS 的行为是 "相机能瞄到的地方就能放,角色被挡住不重要",正好对应玩家 "我准星指着哪 = 我能放在哪"的直觉。

未来如果出现"严格精准放置"技能(类似 Sage 墙、Symmetra TP), 再讨论 V2 是否加 Reject 分支。V1 不为不存在的需求做基础设施。

5. 落点流水线

所有接入技能按同一条流水线求值。客户端预览高频调用,服务端 Execute 调用一次, 两者代码完全相同。流水线永远输出 FinalPlacement,不返回失败。

步骤输入输出要求
1. 构建上下文Caster、AimRay、ProfilePlacementInput客户端预览和服务端 Execute 用同一类输入
2. 求 IntentPointAimRay、SurfaceChannel、MaxRangeIntentXY射线命中则取命中 XY;未命中则朝瞄准方向兜底到 MaxRange;最后 clamp 水平距离到 MaxRange
3. 落地IntentXY、UpwardTrace / DownwardTrace、MinSurfaceNormalDotUpFinalPlacement从 IntentXY 上方往下垂直射线,取第一个法线朝上的命中
原 V1.1 §4 的"Footprint Adapt"步骤已删除。V1 不做 Footprint 校验,召唤继续 AlwaysSpawn。V2 与 Footprint 字段一起加回。

6. 算法规范

本节给出 UPlacementResolverLibrary::ResolveGroundPlacement 的精确行为定义, 是策划、QA、程序对落点行为的唯一仲裁标准。

6.1 IntentPoint 算法(emergent LOS,修复 pick-further bug)

FVector2D ResolveIntentXY(AimRay, Caster, Profile):
    // 唯一一条射线:从 AimRay.Origin 沿 AimRay.Direction 打 SurfaceChannel
    HitResult = LineTrace(AimRay.Origin, AimRay.Direction, MaxTraceLength)

    if HitResult.bBlockingHit:
        IntentXY = HitResult.ImpactPoint.XY    // 撞哪算哪 → emergent LOS 自然生效
    else:
        // 射线完全没击中任何东西(比如瞄向天空)
        HorizontalDir = Normalize(AimRay.Direction.XY)
        IntentXY = Caster.XY + HorizontalDir * MaxRange

    // 水平距离 clamp
    Offset = IntentXY - Caster.XY
    if Offset.Size() > MaxRange:
        IntentXY = Caster.XY + Offset.Normalized() * MaxRange

    return IntentXY

关键性质:emergent LineOfSight

"墙能不能放过去"不需要额外做 caster → placement 的 LOS trace。 由相机射线几何自然涌现(已代码验证起点 = 相机位置):

  • 射线撞墙 → IntentXY 在墙这一侧 → 落点在墙这一侧
  • 玩家抬枪让射线越过墙 → IntentXY 在墙那一侧 → 落点在墙那一侧
  • 玩家想"隔墙放"必须能"瞄过墙",与 Deadlock 行为完全一致

6.2 落地算法

FVector ResolveGroundLocation(IntentXY, Caster, Profile):
    TraceStart = (IntentXY.X, IntentXY.Y, Caster.Z + Profile.UpwardTraceHeight)
    TraceEnd   = (IntentXY.X, IntentXY.Y, Caster.Z - Profile.DownwardTraceHeight)

    HitResult = LineTrace(TraceStart, TraceEnd, Profile.SurfaceChannel)

    if HitResult.bBlockingHit AND
       Dot(HitResult.ImpactNormal, UpVector) >= Profile.MinSurfaceNormalDotUp:
        return HitResult.Location
    else:
        // 完全找不到合法地面(罕见,通常是 IntentXY 在悬崖外 / 地图边界)
        return Caster.Location + AimRay.Direction.XY.Normalized() * 100
V1 没有 Footprint Adapt 步骤。Spawn 直接用 ResolveGroundLocation 输出,接受偶尔穿墙。V2 再加。

7. 服务端权威

客户端高频预览、客户端释放、服务端复检三处调用同一 Library。 服务端用 200cm 阈值制 决定最终坐标:差异小信任客户端(玩家手感无突变),差异大用服务端(挡 cheat)。

7.1 数据流

数据要求
客户端提交AimRay(Origin + Direction)、PlacementProfileId 或对应 Preview 引用、客户端预测 FinalPlacement 坐标
服务端复检用客户端 AimRay 和同一 Profile 调 ResolveGroundPlacement 得 ServerFinal
阈值判定计算 |ClientFinal - ServerFinal|。≤ 200cm 用 ClientFinal 出技能;> 200cm 用 ServerFinal 出技能并记录 audit log
失败路径无。技能必然出,资源/CD 必然消耗

7.2 AimRay 来源(V1 简化)

V1 服务端复算时使用客户端 TargetData 里发来的 AimRay。这意味着 cheat 工具同时伪造 AimRay + 坐标(让两者"自洽")可以绕过阈值——V1 接受这个漏洞,留待 V2 改用服务端 ControlRotation 或加 AimRay 合理性校验。

当前代码风险(V1 必修): SkillModuleSpawnMobaUnit.cpp:84、 SkillModuleAddSkillAreaWithBuffs.cpp:48、 SkillModuleTeleportToTarget.cpp(Execute) 现在都是直接 SpawnLocation = TargetData->XxxLocation, 服务端零校验。任何客户端工具可以 spawn 单位到地图任意角落。V1 必须修复——但按 Deadlock 派"修正不拒绝"方式, 走 §7.1 阈值流程。

8.2 Mode = None 的含义

Mode = None 表示这个技能不走 Library 接地,Library 不会被调用, 落点继续由 PreviewDataAsset 原有逻辑产生(对 SkillAreaPreviewDataAsset 来说就是 camera-rooted + AreaRelativeTransform 偏移)。

  • 项目现有所有 AOE 资产默认 Mode = None,接入零侵入,V1 上线无回归
  • 新接地 AOE 资产策划手动改 Mode = GroundAOE
  • 空中 AOE(玩家头顶云、悬空标记圈等)永久保留 Mode = None——这不是技术债,是真实设计需求

8.3 嵌入位置

9. 预览与反馈

Deadlock 派预览只有两态:合法 / 已 clamp。没有"非法"。 预览渲染必须显示 FinalPlacement(Library 输出),不显示原始 IntentPoint。

合法

玩家瞄准点本身合法,预览圈直接显示在该位置。

已 clamp

玩家瞄准点被 Library 夹取(超距 / 跨地形 downtrace 找到不同高度),预览显示最终落点。

不存在

无论玩家瞄什么,系统总能给出 FinalPlacement,所以没有"非法"展现态。

状态视觉释放结果
合法正常颜色预览圈 / 召唤轮廓 / 落点标记释放到 FinalPlacement
已 clamp预览圈显示在 FinalPlacement。可选:超过 MaxRange 时变橙色提示"已 clamp 到最大距离"释放到 FinalPlacement
抖动平滑拖拽预览时 Library 每帧重算即可,自然平滑释放时服务端再算一遍 + 阈值判定

13. 已知 bug 修复(顺手做)

13.1 SummonPreviewDataAsset 的 pick-further 穿墙 bug

现行 FSummonPreviewDataInstance::UpdatePreviewInternal(SummonPreviewDataAsset.cpp:53-104)的 IntentXY 算法: 

// Method A: 相机射线撞到什么,取 ImpactPoint XY
// Method B: 数学投影到角色 Z 平面(忽略所有碰撞)
// 取两者距离更远的(注释说:防止 "tilting up snap-back")

问题:水平瞄墙(墙距 500,MaxRange=1200):

  • Method A: 撞墙,DirectDist = 500
  • Method B: aim.Z ≥ 0 走 fallback,ProjectedDist = MaxRange = 1200
  • "取更远"→ 选 1200 → IntentXY 在墙后 1200 处
  • Downtrace 在 1200-XY 找到地面 → 落点穿墙到对面

这与 Deadlock emergent LOS 行为冲突,玩家明明瞄的是墙,结果落到墙背后。

13.2 修复方案

新 Library 算法砍掉 Method B 作为"远点候选",Method B 仅在射线完全没击中任何东西时兜底(见 §6.1)。

13.3 行为变化预警

修复后,原本能"水平瞄墙穿过去"放置的技能,会改为落在墙这一侧。 如果存在依赖该行为的技能,需要 QA 验收时确认是否仍可玩。 如果某个技能策划故意要"穿墙放",再讨论 V2 是否给 Profile 加 bAllowAimThroughObstacles(默认 false)。V1 不主动加。

13.4 注释里担心的 "tilting up snap-back" 不会发生

  • 抬枪指远方时,相机射线照样会撞到地面或远处物体(地面、山、建筑),IntentXY 仍然是远点
  • 只有完全瞄向天空(射线 100% 落空)才走 fallback,落到 Caster 前方 MaxRange 也合理
