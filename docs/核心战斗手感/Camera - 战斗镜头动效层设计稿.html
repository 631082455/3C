1. 背景与目标

旧的技能镜头(UAbilityCameraData / ACD_*)存在四个结构性问题:配不出任何旋转(大招俯角、命中踢无法表达);多效果互相覆盖(后触发的整体顶掉先前的,结果不可预期);被打断 / 死亡后可能残留(镜头卡在技能状态);偏移写进判定视图(镜头表现悄悄影响弹道)。

一句话目标

技能镜头效果做成"相对基础镜头的增量":可叠加、必然自动归零、永不触碰判定。策划以"选预设、填强度、画曲线"的粒度表达手感,清理与合法性由系统结构保证,不依赖任何人记得配"清除条件"。

本系统与旧 ACD 并存:新技能一律使用 CME;存量 ACD 技能不受影响,按需逐个迁移。

2. 设计铁律(所有配置与验收的最高准绳)

铁律含义业界依据
① 准星永不说谎一切镜头表现只作用于画面,永不写入控制朝向 / 判定视图。大招压着俯角开枪,子弹仍打准星指的位置。CS / Valorant:视觉后座与弹道彻底分离
② 玩家输入永远赢镜头可以建议构图,不能抢杆;全屏演出只允许出现在玩家本无操作权的窗口。Nesky《50 个镜头错误》第一条
③ 绕角色转,不原地转大幅旋转必须以角色为锚点做轨道运动,角色不滑出画面;原地转视角仅限小幅冲击反馈。Nesky:pivot around the avatar
④ 旋转预算与操作权成反比玩家操作权越完整,允许的旋转越小:可瞄准窗口仅 ≤3° 视线踢;施法锁定窗口允许中幅演绎(≤25°);全锁定窗口才可自由演出。演绎必须在操作权归还前混出完毕。对 Deadlock / 漫威争锋全部旋转案例逐一核实的共同规律
⑤ 震屏是预算不是调料冲击类效果有分带与全局上限,接近上限渐进压缩——团战刷屏叠不成晕。Predecessor 连续多个补丁砍震屏的教训

3. 设计模型:两层,各管各的

基础镜头(底座)

常态第三人称镜头的全部基础参数(臂长、FOV、俯仰限、碰撞避让)。全局一份模板,每英雄可覆盖枢轴偏移。它定义"原样"。

动效层(本设计)

技能期间叠加在底座之上的短时效果:旋转、推拉、FOV、冲击。全部是增量,自带包络必然衰减归零。拿掉全部动效,画面精确回到底座。

归属判据

一个镜头需求,问"效果结束时回到哪里":归零回原样 → 动效层(本系统,覆盖约 95% 的技能镜头需求);回到另一种持续模式(开镜、俯瞰操控、变身换机位)→ 状态镜头层,本项目已决策不做(见第 7 节)。

旋转的两种语义(动效层内)

绕枢轴轨道(Orbit)原地视线踢(ViewKick)
行为机位沿以角色为圆心的轨道移动,视线始终穿过角色机位不动,视线短暂偏移后回弹
用途 / 预算施法演绎、大招构图、蓄力(≤25°,限操作权受限窗口)命中踢、开火顿挫(≤3°,可瞄准窗口唯一合法旋转)
设计理由大幅旋转若不绕角色,角色滑出画面、玩家丢失参照(铁律③)小幅冲击要的就是"画面被撞了一下",绕圈反而无冲击感

4. 机制设计

4.1 叠加:增量求和,不是覆盖

  • 增量通道(旋转 / 位移 / FOV / 臂长):多个效果同帧求和——大招俯角 + 命中踢共存,踢叠在俯角之上,各自衰减各自归零。这是对旧系统 last-wins 的根本性替换。
  • 覆盖通道(跟随柔和度 Lag,数值相加无意义):高优先带胜出。
  • 优先带(Band):受击/命中 → 移动 → 技能 → 大招 → 系统。带用于冲突裁决与分带预算,不用于互斥。
  • 防叠爆:受击带旋转合计软钳 4°,全局旋转合计软钳 20°,FOV 合计 ±15。"软"指越接近上限压缩越强(而非硬截平)——团战最激烈时新的命中仍有递减反馈,不出现"越打越没反应"。

4.2 节奏:包络 × 形状 × 强度

  • 每个效果自带包络(混入 → 保持 → 混出),必然归零;持续时间可设"跟随技能生命周期"。
  • 形状与强度分离:形状曲线(内联可画,纵轴允许过冲与反向回弹)描述"这个效果长什么样";使用处的强度标量描述"多重"。重狙与冲锋枪共用同一形状、只差强度——保证全局镜头语言一致。
  • 重触发策略:同一效果连续触发时可选 重播 / 高强度胜出 / 忽略 / 续时。冲击类默认"高强度胜出",防止轻击的反馈吞掉随后重击的反馈。

4.3 清理:残留在结构上不可能

这是对旧系统最重要的设计承诺,不依赖配置、不依赖任何路径"记得清理":

  • 技能结束 / 被打断 / 预测回滚 → 该技能的全部动效自动淡出;
  • 持续时间到期 → 包络自然收尾;蒙太奇窗口结束 → 窗口效果自动释放(打断时由引擎保证送达);
  • 死亡 / 切换观战 → 全清(含震屏立停);
  • 兜底看门狗:归属技能实例已消亡的无限时长效果,下一帧强制剔除并报错——清理永不依赖"健康的归属者"(本项目 Lissandra 蒙太奇孤儿事件的结构化教训)。
竞品实证:Deadlock 存在一整类"大招被打断后镜头永久卡在拉远状态"的 bug(Haze / Mina / 滑索加速等),根因都是"进/出成对事件、出事件被打断吞掉"。本设计的效果没有"出事件",只有到期与死亡,两者都不依赖打断路径的良好行为——对这类事故结构性免疫。

5. 配置粒度设计

配置分两级,复杂度沉在资产、日常路径保持极简:

层级承载什么谁维护
CME 资产(共享库)效果的"形状":各通道幅度、包络、形状曲线、优先带、重触发策略预设库维护者(少数人)
使用位点(技能 / 蒙太奇窗口)引用哪个资产 + 强度标量(可挂多个叠加)配技能的每个人

预设三形态(一条黄金路径 + 两个后门)

形态定位解决什么
招牌预设配比调好的多通道成品(命中踢、大招俯角拉远等约 10 个)九成需求:选一个、填强度。镜头语言全局一致的保证
单通道原语 + 堆叠只动一个通道的原料,使用位点挂多个各调强度调配比 / 各通道不同节奏,不必新建资产
稀疏变体认父资产、只记录差异倍率;禁止套娃单英雄灰度试点,试完归并回父资产
复用纪律:形状是稀缺资产,强度是廉价参数。禁止在使用处自由改各通道(会让每处引用成为不可评审的孤本);新建 CME 前先确认三个后门都不适用。非法配置必须可见:越界值存盘即拒,可疑值报警告——"能配但没效果且不报错"列为 P0 缺陷类别。

7. 范围决策(已定,附触发再议条件)

决策理由再议触发
不做状态镜头层(开镜/俯瞰模式/变身档案)"技能期间改、结束回去"由动效层覆盖;产品无 ADS;升空俯瞰用动效即可;死亡观战现状够用出现操控型接管技能、或需要 per-hero 机位继承时
缓议:弹道命中回传(含方向性受击踢)脱手技能"命中那一刻给施法者反馈"缺触发位点,现无此类核心技能有投掷/远程脱手技能立项时
缓议:观战端动效重放动效为本地触发,观战镜头无演绎(体验缺失非 bug,清理已干净)观战/OB 体验变重要时
缓议:trauma 震屏预算 / Camera Bench 图形面板 / per-hero 距离缩放现阶段无对应痛点,预建即过度设计震屏内容增多 / 调参嫌命令烦 / 出现体型悬殊英雄

8. 验收标准(策划口径,逐条可打勾)

#验收项
1大招压俯角+拉远期间开枪,子弹打准星位置;强震屏下连续射击弹道不漂
2技能被打断 / 角色死亡,镜头在混出时长内回位,无任何残留(含震屏立停)
3大招俯角与命中踢同帧共存互不覆盖;结果可从调试面板逐项解释
4删除任意/全部镜头动效配置,游戏正常可玩(基础镜头永远兜底)
5贴墙释放拉远类效果,镜头不穿墙
6非法配置存盘被拒(混合时间为零 / 旋转超预算等);无 NaN、无瞬移
7演绎在操作权归还前混出完毕;配晚了有警告
8PIE 中修改资产字段/曲线,下一帧生效
9团战级叠加无晕动感(软钳生效,反馈递减不消失)
10单英雄灰度可走通(变体试点→归并→删除,不影响其他英雄);配比微调不需新建资产

以上 1–9 已在当前实现验证通过(编译 + PIE + Steel 大招真英雄链路);10 的机制已具备、待首次实际使用。
