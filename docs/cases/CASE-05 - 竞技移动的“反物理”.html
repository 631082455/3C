Combat 3C · Movement Case · Doc CASE-05

竞技移动的“反物理”：输入确认、非对称轨迹与战斗回流

这里的“反物理”不是绕开 CharacterMovement，而是削掉不服务操作的自然惯性。起步要尽早承认输入，高弧跳跃要在下坠时兑现威胁，翻越和落地要把控制权还回来；胶囊位置、判定与网络结果仍然一丝都不能骗。

课题真实加速会显得软，对称高弧会乱飘，完整播完 Stop / Land 又容易抢控制。问题不是物理对不对，而是它有没有服务输入、预警与战斗回流。
终案用四道门裁决每一次“反物理”：输入确认、轨迹造型、控制权归还、网络一致。真实位移交给 CMC / RMS，重量和提前读意图交给动画。
结果方向性移速、Sparrow 下降段缓降、Steel 非对称跳劈、Traversal 动量恢复已落地；Start / Pivot Boost 代码已成形，但网络移动合并缺口未过，当前关闭。

这篇不是从“游戏为什么需要反物理”讲起。这里的“反物理”只指削掉不服务操作的自然惯性，不是绕开 CharacterMovement，更不是不要网络预测。项目里已经有更具体的东西：起步爆发写进了 CMC，却因为网络移动合并没敢打开；Sparrow 蓄力时只把下降放慢；Steel 跳劈则反过来，过顶点后把重力乘到 4 倍。它们早就不是一条物理曲线。

我习惯用四个问题管这类需求：玩家的输入何时必须被承认，轨迹哪一段负责卖预警、哪一段负责兑现，控制权该由哪一层还回来，这个短窗能不能被客户端预测和服务器重放。这只是项目里的工作分法，不是行业分类。

目前已经落地的部分，本文会把代码、配置和边界写细；没有做完的也不补结论。Start / Pivot Boost 仍然关闭，普通 Jump 要不要加重下降、Jump Buffer 要不要立项，都留在建设清单里。

背景

移动系统现状与问题

项目是 UE5 多人竞技 MOBA / 射击。常态移动由 CharacterMovementComponent（CMC）结算真实速度，技能跳跃和冲刺会交给 Root Motion Source（RMS），Motion Matching 与动画蓝图负责 Start / Stop / Pivot、脚步和重量，GAS 与 Traversal 再切换移动配置。一个“起步有点软”，往往横跨四层。

真正难的是三个瞬间。静止起步时，自然加速度还没形成肉眼可读的速度，玩家已经按下了；高弧技能过顶点后按对称重力慢慢落，预警有了，威胁却砸不下来；Stop、Land、Traversal 如果坚持把动画播完整，胶囊体和控制权就被表现拖着走。

本文的红线

可以重排速度曲线，可以让动画提前读出意图，也可以把惯性留在上身；不能伪造胶囊位置、判定距离和联网结果。更准确地说：动画可以重排时间、缩放幅度、分配身体朝向，但不能伪造移动事件、方向符号和触地约束。

过程

最近的已知解

行业里处理这类问题的手艺并不神秘，大体是三件事：输入宽容让系统更早承认意图，分段曲线把上升和下降拆开，表现层承担重量但不改真实判定。项目真正多出来的一道约束，是这些短窗都要经过 CharacterMovement 的预测、合并和重放。

输入宽容

Boost、Buffer、迟滞窗都在缩短“我已经按了”到“系统承认了”的时间。它们不能顺手增加不可解释的总位移。

分段轨迹

上升负责轮廓和预警，下降负责操作窗口或威胁兑现。缓降和硬砸看起来相反，底层都是只改下降段。

表现与判定分离

胶囊体先停，上身与骨盆再把惯性演完；动画可以提前反打，但不能把判定位置提前搬过去。

多人保险丝

本地预测、服务器结算、远端表现必须是同一件事。单机很跟手、进 DS 就回弹，这个方案仍算没做。

这节目前只写项目能坐实的行业常用手段。外部检索工具在本轮不可用，没硬塞竞品名字；最相近的出货游戏先例待发布前补充调研。

方案排定：四个问题，问完再动手

每次讨论 Boost、重力、落地或 Traversal，先过四个问题：输入何时必须被承认；轨迹哪段卖预警、哪段做兑现；控制权由哪层还回来；这个短窗能不能被客户端预测和服务器重放。它们不是四套系统，是同一个改动的四个检查位。

被否掉的捷径

看起来省事的方案为什么不行最后怎么做
把 Start / Stop 动画拉长，靠表演卖重量输入已经发生，胶囊体却等动画；重量直接变成迟滞。真实位移先响应，动画只负责蹬地、制动脚和上身惯性。
统一把重力拉大，同时做高弧和硬砸上升、顶点和下降绑在一起；弧高与下坠互相打架。顶点前后分段，缓降与硬砸各自只改下降。
Traversal 结束速度归零玩家翻完重新起步，胶囊体停了，动画还在接跑，立刻滑步。按动作恢复部分动量，并在 Warp 完成后及时还控制。
先开 StartBoost，再补网络瞬时触发帧可能被 SavedMove 合并，客户端与服务器不是同一条曲线。默认关闭；补完 combine / replay 证据后再谈启用。

下面六节按机制平铺，每节只讲一件事：现象、实现、数字、当前状态与边界。

起步与折返：Start / Pivot Boost

已实现未启用

现象很老：静止起步时，自然加速度还没形成肉眼可读的速度，玩家已经按下了；大角度反打时，当前速度方向已经错了，系统却还在慢慢刹旧方向。两件事是一族问题，都是"输入发生了，系统还没承认"。

实现放在同一套 FStartBoostSetting 里：持续时间 1.3 秒、峰值上限 750cm/s（抬的是 MaxSpeed 上限，实际速度仍要靠加速度爬上去），Pivot 默认在速度方向与输入方向点积小于 -0.5 时触发（夹角约大于 120°）。UpdateStartBoost() 每帧检测"上一帧静止、本帧有输入"或"大角度反打"，重置计时；GetMaxSpeed() 再按剩余时间从峰值插值回基础速度。ABP 只能读 StartBoostAlpha 做表现，不能把速度写回来。方向性衰减排在 Boost 之后生效，后退起步同样打折。

当前关闭 FSavedMove 已保存和恢复 BoostTimer、上一帧运动状态，却没有 CanCombineWith() 比较。短触发帧可能被移动合并吞掉，客户端与服务器重放不同步。现行 Sparrow、Steel、Twinblast 等 MCD 实读均为 bOverrideStartBoost=false。

交叉复核又补了三条启用前置：Pivot 判定读的是 Velocity 与输入的点积，没有迟滞和边沿锁存，反向期间会逐帧把计时器续满；被击退等被动速度也可能误触发；BoostTimer 不复制，远端只会看到位置变快而没有对应的起步表现。这几条都在遗留表里挂成启用条件。

方向性移速衰减

已落地并配置

朝向开始收钱：正面盯着敌人后撤，信息保住了，速度要交一点；转身跑能拿满速，但把枪口和视野交出去。当前全英雄配置是前进 1.0、横移 0.9、后退 0.8。

它不是三个硬档，而是用角色面朝方向与 Acceleration 的二维点积连续插值：45° 斜前只吃约 29% 的侧移惩罚，斜后大约 0.83，摇杆画圆不会在区间边界跳速。系数排在属性、Buff、Boost 全部汇总之后整体相乘，任何来源的移速后退时都等比打折。

取 Acceleration 而不是 Velocity 是网络决定：Acceleration 进 SavedMove / ServerMove 的重放数据，客户端预测与服务器天然一致；Velocity 反过来决定自己的上限，会形成"速度算上限、上限再改速度"的反馈环。松开输入时 Acceleration 约等于零，系统返回 1.0，不碰制动滑行。边界也直说：代码与 20 个英雄的配置已落，多人盲测和观察期还没走完。

停止与制动：位置先停，重量晚半拍停

既有底座

角色底座当前使用 MaxAcceleration 2400、GroundFriction 8、BrakingFriction 6、BrakingDecelerationWalking 1400。拿同一组物理量预测停距、再按真实位移推进 Stop / Pivot 的 Distance Matching 代码在 UHeroAnimInstance 这条动画链里；现行共享 MM 父 ABP 走的是 GASP 的 MotionMatching / OrientationWarping / FootPlacement 节点。两条链要分开说：前者是把时间对齐真实距离的正确做法参考，不能直接写成当前 MM 链的上线事实。

这些常量不是这次新拍的，也没有一轮完整 A/B 可以冒领。它们能坐实的是结构：CMC 决定角色到底停在哪，动画负责制动脚、骨盆回收和上身惯性。不能为了把 Stop 播完，再把胶囊体拖出去一截。

一处口径冲突留档：量化标准和调参手册写过 Stride Warping 已配，但这次对 Content 与 Plugins 全库检索没有找到 StrideWarping 节点，只有旧动画链里的权重变量。进待验证，不在正文当既成事实。

Sparrow 蓄力缓降：只让下降慢下来

已落地并配置

Sparrow 蓄力配置把最大移动速度压到 300cm/s，并启用 DescentGravityScale=0.25。运行时只有 IsFalling() 且 Velocity.Z<0 才覆盖普通 GravityScale；上升段完全不动。

这解决的是操作窗，不是跳得更高。弓箭手已经离地并进入下降时，给她更长的蓄力和瞄准时间；如果把整个 GravityScale 都降下去，上升也会被拖长，技能节奏会一起变软。接口是通用的（任何 Override MCD 都能配下降段倍率），但目前坐实的使用案例就是 Sparrow 这一处，不写成普通跳跃已全面使用。

Steel 跳劈：上升卖预警，下降把威胁砸实

已落地并配置

Steel 的问题和 Sparrow 正好相反。AbilityJump 使用 RMS 的恒定向上速度与 CustomGravity；为了高弧调大向上分量，这股力在下降时仍然托着角色，对称曲线过顶点后发飘。项目新增 DescentGravityMultiplier：顶点前用原 CustomGravity，顶点后才乘下降倍率，倍率 1.0 时与旧逻辑逐帧一致。

Steel 现行落地段配置实读为 4.0。这个提交不只改一条曲线，还一起落了 NetSerialize、HAD 跳跃调参、砸地 Gameplay Cue、Camera Lag 和落地段 MCD。攻击方在上升阶段卖出轮廓，防守方能看清预警；过顶点后下坠变硬，威胁不会悬在空中等人。

一条链路边界要说清：普通空格跳走 CMC 重力，AbilityJump 走 RMS 自己的重力公式，两条链只共享落地环境。Steel 的 4.0 不代表所有跳跃技能都已非对称，更不代表普通跳跃该照抄。

落地与 Traversal 回流：动作可以占用控制，但必须有归还点

代码能力已落地

普通落地的动画权重按空中时长 0–0.4 秒映射到 0.1–1.0；蹲伏时再乘 0.3，落地仍有移动输入时乘 0.5。这里削的是 Land Recovery 表现，不是胶囊体控制权：玩家还在推杆，恢复动作就让位。

Traversal 走得更远。Hurdle / Vault 结束后恢复进入前水平速度的 80%，再切回 Falling 让物理接管落地；Mantle / Climb / Catch 直接回 Walking。EndAbilityTimingEarlyOffset 还能相对 Warp 窗口提前或延后还控制，计时会除以最终 PlayRate。0% 动量会重新加速并滑步，100% 又接近无成本跑酷，0.8 是当前默认折中。

两条边界一起记：恢复动量只缓存了进入前速度的标量，结束时沿角色 Forward 写回，横移、后退入场会被改造成前冲；翻越期间还临时设了忽略客户端纠正、接受客户端权威位置两个网络标志，EndAbility 会恢复，但竞技模式上线前必须补服务端的路径与终点合法性校验。

禁止上线条件

  • 本地与服务器速度曲线不同，或开启后出现位置修正。
  • 攻击方获得额外位移，但对手侧没有等价可读信号。
  • 动画已经反打，胶囊体仍长时间沿旧方向滑。
  • Stop / Land / Traversal 为了播完整动作继续锁玩家输入。
  • 普通 Jump 的曲线修改破坏落点预判或让空中暴露窗口大幅变化，却没有数值评审。

方法论沉淀

展开沉淀条目

候选 M：输入先确认，惯性后表现

位置和控制先服从输入，重量用骨盆、上身、制动脚和二级运动补。适用边界：普通 Locomotion；技能明确承诺窗内可以短时占用控制。

候选 M：上升卖预警，下降做兑现

用于有明确战斗承诺的跳跃技能。缓降也用同一分段接口，但目标是操作窗；不能把“下降加重”推广成所有 Jump 的默认答案。

候选 M：动画可以提前读意图，位置不能提前结算

Trajectory、Pivot Clip、Orientation / Stride Warping 可以更早表达反打；胶囊体、命中判定与网络权威仍由运动层决定。

候选 AP：单机先爽起来，网络以后补

短时 Boost、Buffer 和状态脉冲最容易被移动合并吃掉。没有 SavedMove、combine、replay、远端语义的证据，就不算完成。

候选 M：跨状态动量要同时回答收益和成本

0% 会断流，100% 可能无成本；继承多少必须和动作承诺、对手可读性、落地控制权一起拍。
