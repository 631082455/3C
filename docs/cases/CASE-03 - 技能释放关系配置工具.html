技能释放关系：谁挡谁、谁并行，策划一张表配完

A 技能放到一半，B 键按下来了——挡住、打断、还是一起放？这层规则原先写死在程序侧的技能蓝图里，策划改不了也看不全。这次把它抽成每英雄一张 H5 关系表：填表 → 存盘 → 约 1 秒局内生效，连 PIE 都不用停。演示里的 Skyfire 火焰期间并行放护盾、手雷，就是表上配出来的。

技能打架的规则（我放大招时你按闪现给不给走）是手感的一部分，但它原先住在程序侧：每个技能蓝图里写死一串「阻挡哪些 tag」。策划想改一条得开蓝图，想看全貌基本不可能。

做法是把这层规则抽出来：每个英雄一张表，一行就是一条「A 释放中，B 来了怎么办」。表存盘后编辑器里的监听脚本自动编译进数据表，运行时每次放技能都现读，所以改完约 1 秒就能在局内试。这是老套路（数据驱动），值钱的是配套语义：哪类技能进表、哪类不进，想清楚了表才不会越填越乱。

验收用了个有点极端的例子：Skyfire 火焰喷射期间要能同时放护盾和手雷，还得跳过它们的出手动作免得打断喷火——全程在表和资产配置里完成，没动火焰技能一行代码。

背景

课题

项目是 UE5 多人 MOBA 射击，战斗基于 GAS（UE 的技能框架）加自研扩展，英雄技能批量生产期。GAS 处理技能互斥的原生手段是每个技能身上配静态的「阻挡 tag 列表」——规则分散在几十个技能蓝图里，改一条要开蓝图找字段，两个技能的关系对不对得脑内拼图。

课题定性是改造型：机制本身没错，错在配置的形态。要的体验还是那句话，拿预设改 1-2 个数（这里是 1-2 行）级别的配置，加上一处能看全貌。

分工与工作方式

作者（战斗 3C 策划，不写代码）：定关系语义（03 节的边界整理是本案例真正的策划工作量）、按设计文档逐英雄填表、PIE 验收。AI agent（Claude）：执行全部实现——C++ 行结构与运行时钩子、编辑器监听、H5 编辑器，含自测。程序未参与。

过程

工具管线

五步，策划只碰第一步：

关系行长这样：A 技能 · A 窗口 · 来袭 B · 结果（结果三选一：拦下 B / 被 B 打断 / 并行）。「任意技能」行做兜底，具体技能行优先于它。运行时侧只加了一个钩子：技能激活前查这张表、把该拦的 tag 灌进 GAS 原生的阻挡机制——机制没换，换的是规则的存放处。

能做到约 1 秒生效、PIE 不用停，靠两件事：监听只改内存里的 DataTable（不落盘，所以运行时不受限制），运行时每次激活技能都现读表（没有任何烘焙缓存）。页面存盘后轮询回执，绿字「已生效」亮了才算数——工具给策划的每个承诺都有回执兜底，不靠信仰。

表的语义边界

管线是工具活，这节才是策划活：什么技能进表？第一版把 8 个英雄的技能全填了进去，越填越不对劲——Muriel 的护盾是个持续 buff，它「释放中」是哪段？填出来的行自己都说不清语义。整理后的边界：

进表：承诺出手 / 引导

按下后有一段「正在出手」的过程，期间来了别的输入需要裁决——普攻连段、蓄力射击、引导型大招、点选确认的短前摇技能（Gideon 屏障、Sevarog 领域这类，第一版错杀过，后来捞回）。

不进表：持续状态与位移

护盾、潜行、飞行这类挂上就完事的持续态，「释放」只是一瞬，释放期间的关系无从谈起——它们对别的技能的影响走状态 tag 判定，不占关系表。位移技（冲刺类）另有成套处理，表里只标「待处理」不硬塞。

一句话边界

关系表只回答「A 正在出手时 B 来了怎么办」；A 都没有「正在出手」这个阶段的，别进表。

窗口档位也做了分层：整段、蓄力段（Holding）立即可编译生效；更细的出手前 / 判定中 / 收招段（Startup / Active / Recovery）在下拉里预留，选了会提示需要先在动作上划窗口（ANS），不会假装生效。

结果

并行实例：Skyfire 火焰期间放护盾与手雷

需求原话：火焰（3 秒引导喷射）期间，要能放 2 技能护盾和 E 技能手雷，并且跳过这两个技能的出手动作——不然一出手就把喷火动作顶掉了。落下来分两层，都在配置侧：

  • 关系层：Skyfire 表上加两行——火焰 × 护盾 = 并行，火焰 × 手雷 = 并行。这两行把它们从「任意技能被拦」的兜底里挖出来。
  • 表现层：火焰技能激活期间挂一个状态 tag；护盾 / 手雷的动作模组配了「条件跳过」——检测到这个 tag 就不播出手动作、改走事件把该干的事干完（这套跳动作规则是现成机制，此前 Twinblast 冲刺时开枪已用过一次，这里是第二次复用）。

火焰技能本身的代码一行没动。顶部视频就是这条链路的实录。

遗留

展开清单
  • 细窗口未落地：Startup / Active / Recovery 三档需要动作上的 ANS 窗口配套，目前只是下拉预留，选了会提示不会生效。
  • 打断（CancelTags）走代码：表里「被 B 打断」的行目前编译进表的只有阻挡列；打断语义仍在代码侧，表行先当登记用。
  • 多人带延迟未测：运行时现读表在预测端和服务器端行为是否完全一致，缺一次带延迟实测。
  • 位移技悬置：冲刺类在表内标「待处理」，等位移套路统一后再收编。
  • 验收 n=1：目前只有需求方本人在用。

方法论沉淀

展开沉淀条目
  • 先定语义边界，再开放填表。数据驱动工具的失败模式不是管线不通，是表的语义含糊——第一版把持续态塞进关系表，填出一堆自己都说不清的行。「A 没有正在出手阶段就别进表」这条边界，比管线本身更值得沉淀。边界：适用于一切关系型配置表。
  • 内存生效与落盘入库分离。改内存 = 试错零成本且 PIE 不受限；落盘 = 有意识的入库动作。两档合一的工具要么不敢试要么进脏数据。（本条后来在技能数值配置管线上第二次成立。）
  • 工具的每个承诺都要有回执。「保存了」和「生效了」是两件事，中间隔着监听有没有跑、编辑器开没开。页面轮询回执、绿字亮了才算数，省掉的是「到底生效没有」的猜疑成本。边界：任何跨进程的配置工具。

工具坑清单（照抄可避）

  • 往 GameplayTag 源表（DT_HeroAbility 等）灌 CSV 会触发 tag 树重建，曾直接崩编辑器；要加 tag 优先走 ini 或挂调试器操作。
  • init_unreal.py 只在编辑器启动时加载，改监听必须重启编辑器。
  • DataTable 的空 tag 容器要写 （GameplayTags=），空串会报 Missing opening parenthesis。
  • UGS 开着时 pending CL 可能被自动提交（经历过一次"CL 消失"，实为被 UGS 提交并重编号）；整理 CL 期间留意。
  • MD 的 md5 口径要两端一致（BOM 去掉再算），否则回执永远「待同步」。
