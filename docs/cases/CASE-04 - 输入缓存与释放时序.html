Combat 3C · Case Study · Doc CASE-04

玩家已经按了，技能什么时候算数？

一套 UE5 多人战斗输入，需要同时回答三件事：技能暂时放不出来时要不要记住这次按键，玩家持续按住时何时补触发，以及蓄力技能延迟出手时究竟按哪个时间算伤害。本文以 Sparrow（项目中的弓箭手英雄）蓄力普攻为主案例，复盘从输入缓存、秒松、长按续触到网络预测竞态的完整释放时序。

课题玩家按下、松开和服务端允许激活并不发生在同一帧；系统既不能吞输入，也不能把长按做得比完美手点更强。
终案短时缓存、物理按住状态、蓄力释放时间分开管理。AttackInterval gate 结束时只消费“仍按住”的事实，新一轮蓄力不回溯。
结果通用缓存、最短释放时间、Sparrow 蓄力参数、自动续触与 50 ms 服务端相位保护均完成代码／资产链路核验；未找到正式用户测试样本。
分工作者负责 Sparrow 体验目标、时序取舍、资产调优与验收；AI agent 协助当时的代码／ADR 落地及本次历史取证；项目既有 GAS、输入和技能模块提供运行框架。

玩家已经按了，技能却不一定能在这一帧放出来。可能是上一发的攻击间隔没结束，也可能是表现上必须先把弓拉到最低出手点。只用一个“输入缓存”去解释这些情况，很快就会乱。

Sparrow 的蓄力普攻把问题摊得很开：按下开始拉弓，松开决定蓄力值，动画和发弹可以稍后执行；射完立刻再按，新的输入又可能撞上攻击间隔。我们最后没有把所有东西塞进一个队列，而是把短时重试、持续按住和蓄力时间拆开，各自在正确的时机被消费。

真正难排的是多人网络。客户端认为 gate 已经结束，立刻预测下一发；服务端的计数还差一点才清掉，于是拒绝这次激活，玩家看到的就是偶尔丢一箭。问题只差几十毫秒，手感上却像输入不靠谱。

背景

体验目标与课题

项目是 UE5 实现的多人英雄射击游戏，技能激活建立在部分 GAS 与自研模块之上。输入要先经过本地预测，再与服务端权威状态对上；对普攻来说，还叠着攻击间隔、动画、弹道和按键生命周期。

Sparrow 的普攻不是点一下立刻生成固定箭矢。按下后进入拉弓，松开时的蓄力进度会影响伤害倍率、弹速、重力和准星；射出后又有 1 秒攻击间隔。玩家如果在 gate 期间再次按下并保持，系统既要记得他还想继续，又不能把等待时间偷算进下一箭的蓄力。

别吞输入

上一段状态刚结束时按下，玩家期待下一段自然接上，而不是必须猜中那一帧再按一次。

别送强度

长按可以减轻手部负担，但不能回溯被 gate 挡住的时间，否则长按 DPS 会高于完美手点。

核心课题：一次输入究竟在什么时候成立？物理按键发生、技能允许激活、表现真正出手和服务端接受请求必须各自有清楚的时间源。

分工与工作方式

作者主导 Sparrow 3C 改造，确定蓄力、攻击间隔、长按不增益和松手可取消等体验边界，并持续调整动画、弹道、准星、空中缓降与反馈资产。AI agent 协助实现自动续触、记录决策和排查网络竞态；本文再次由 AI agent 按指定账号与开发客户端交叉核对 P4、源码、DataAsset、Montage 与设计文档。正式玩家测试记录没有找到，不补写样本。

过程

业界调研

这类问题的已知解并不少，但它们解决的是不同层。动作游戏常用有限输入缓存，让过早一点的按键在状态开放后补执行；蓄力武器通常把按下、保持、松开拆成不同阶段；多人战斗再加一层客户端预测与服务端裁决。危险在于把三者都叫“缓存”，然后互相借时间。

常见模式解决的问题不该承担的职责本项目对应
有限输入缓存技能暂时失败时，短时间内自动重试。不代表玩家现在仍按着，也不应无限等待。按能力选择开启，超时清除，并排除冷却、消耗、网络等失败原因。
按住状态表示物理输入仍持续，适合长按和持续瞄准。不保存历史按键时刻，也不自动等于蓄力进度。Input Press Gameplay Tag 随按下增加、随松开移除。
蓄力时间戳计算本次出手的伤害、弹速、曲线和表现。不负责下一次技能能否激活。由技能激活与释放报告计算，供 projectile、buff 和 UI 消费。
网络预测客户端先响应，服务端随后确认。不能假定两端 Timer 在同一帧结束。续触仍走玩家手点入口，不新增另一条 RPC。

本节是机制谱系，不是完整竞品测评。哪些游戏把长按做到“辅助操作但不增伤”，仍需作者在入库前补一次行业实例复核。

方案排定

关键决定不是“缓存几毫秒”，而是先承认这里有三种状态。通用失败缓存只保存一次待重试的能力；长按续触看当前是否还按着；Sparrow 的蓄力值只认本轮激活到释放的时间。

采用：gate 结束时续触

AttackInterval 真正结束的瞬间，若输入仍按着，就重新走一次玩家按下入口。它不会提前承诺，也不会错过开放帧。

否决：在 EndAbility 判断

能力可能在 0.2 秒就结束，攻击间隔却要到 1 秒。玩家在中间松手，系统仍会按旧状态错误续触。

否决：复用 ContinuousFire

持续吐弹模块有自己的 FireInterval，会绕开 Montage、combo 和 AttackInterval，也表达不了“按下蓄力、松开射击”。

否决：补偿被吞时刻

如果把 gate 期间的按键时间算进下一轮蓄力，Sparrow 的 DPS 会从“攻击间隔与蓄力共同限制”退化成只看攻击间隔。

为什么还要走玩家手点入口

直接调用 GAS 激活看起来更短，但会跳过 Press Tag、同帧双触保护、输入事件与失败处理。自动续触最终调用同一个 Input_AbilityPressed，只用 cached 标记避免重复增加按住 Tag，行为才不会分叉。

Sparrow 当前 HAD 的通用失败缓存开关是关闭的。它不是先把普攻塞进 0.3 秒缓存，再等机会重试；AttackInterval 到期时消费的是“LMB 现在仍按着”。这正是两套机制不能混用的证据。

设计保险丝：长按 DPS 不得高于完美手点 DPS。Sparrow 的新一轮蓄力起点等于 gate 结束时刻，不回溯到更早的按键时刻。

实验与落地实录

这轮工作不是一次提交完成的。先有通用输入缓存，再把 Sparrow 普攻从动画到弹道重新搭起来，随后才遇到秒松、续触和网络丢发。

第一轮：给过早输入一个短窗口

通用输入层增加可配置缓存时间。能力激活失败时，只有显式开启缓存的能力才进入重试；冷却、消耗、输入被禁和网络失败被排除。缓存每帧尝试一次，能力成功激活或超过窗口就清掉。它解决的是“差一点按早了”，不是“玩家一直按着”。

第二轮：Sparrow 的蓄力不只是一条计时器

普攻激活后进入约 0.71 秒拉弓 Montage；当前资产的满蓄口径是 0.7 秒，伤害、发射速度和重力都从释放报告取曲线。准星用自己的 Tick 对齐同一段 0.7 秒设计口径，从 2.5 倍外圈收至中心；蓄力时移动速度压到 300 cm/s，空中下降重力另做减缓。它们看起来是动画、UI、弹道和移动问题，实际都在问同一件事：这次蓄了多久。

第三轮：秒松与延迟释放

Sparrow 设置了最短释放时间，玩家过早松键时，释放模块会等到最低出手点再执行。问题随即出现：物理松手时刻、真正执行释放模块的时刻，以及最终蓄力效果读取的时刻不再相同。历史修复先调整释放时间写入，再补资产曲线取值，说明“延迟表现”与“蓄力语义”必须分别定义，不能默认共用当前帧。

第四轮：gate 期间重按不能被吞

射出一箭后，如果玩家在 AttackInterval 尚未结束时再次按下并保持，这次激活会失败，但 Press Tag 仍表示 LMB 处于按下状态。于是 AttackInterval 模块增加 opt-in 开关：Timer 到期先移除 gate，再检查 Press Tag，仍按着就模拟一次玩家手点。Sparrow 开始新一轮蓄力，但起点就是此刻。

第五轮：客户端先到，服务端还差一点

自动续触上线后出现偶发“丢一发”。客户端 Timer 先结束并预测下一次激活，服务端的 AttackInterval 计数还没清零，于是服务端拒绝请求。修复让所有权威端的 gate 最多提前 50 ms 结束，客户端视觉节奏不变；当攻击间隔极短时，最多只减原时长的 50%，避免把服务端 gate 减成零或负数。

50 ms 是针对当时竞态窗口的工程保护，不是通用网络常数。它依赖 Timer、RTT 与预测链路，换项目不能照抄。

结果

终案分层

终案的核心不是一个万能缓存，而是三个时间源只在自己的边界内生效。下图用 Sparrow 射出一箭后立即重按为例。

短时缓存层

保存“刚才想放这个技能”，有超时、有失败原因过滤，适合瞬时意图。

持续输入层

Press Tag 只回答“现在是否还按着”，松开立即移除，适合续触判断。

释放语义层

本轮激活与释放时间供伤害、弹速和重力取值；UI 另按同一设计时长对齐。

网络相位层

服务端 gate 略早收口，避免客户端预测撞上仍未清零的权威状态。

方法论沉淀

展开沉淀条目

输入缓存不等于按住状态

适用于所有带 gate、蓄力或持续输入的动作系统。瞬时意图需要超时，持续状态需要跟随松手；把两者共用一个缓存，很容易出现松手后还执行。

延迟表现前先定义语义时间

适用于最低前摇、动画对点、网络延迟执行等场景。先决定数值读取物理输入时刻还是表现执行时刻，再安排 Timer。顺序反过来，伤害、弹道和 UI 会各取一个时间。

辅助操作不能偷偷提高上限

适用于长按连发、自动续招和连点辅助。降低操作负担可以，回溯 gate 内时间则会改变 DPS。除非它本来就是平衡目标，否则应守住“长按 ≤ 完美手点”。

预测竞态先画两端状态，不先加重试

适用于客户端预测与服务端 Timer 同时参与的技能。偶发丢发不一定是输入没送到，也可能是客户端先清 gate、服务端晚一帧。先对齐两端计数和时间，再决定提前、延迟或重试。

验收矩阵按生命周期展开

至少覆盖按下、秒松、半蓄、满蓄、gate 内重按、重按后松手、极端攻速、Listen Server、Dedicated Server 与不同 RTT。只测“按住能连”不够。
