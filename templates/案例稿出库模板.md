# 出库单 v4.1：工作会话直接产 H5 案例稿，维护会话守入库关

> 用途：任何一个 AI 工作会话做完有方法论价值的 3C 工作后，把下面的「H5 案例稿指令」
> 原样贴给它（把文件名占位改成实际日期和主题），让它**以空白母版为基底填空产出
> H5 案例稿**，然后你和它在 H5 上迭代（改内容、出图、定制模块）——迭代发生在
> 有一手上下文的地方。
> 知识库维护会话不参与逐轮迭代，只守**入库关**：主题合规、术语、M/AP 反哺、
> CASE 编号分配、目录、发布。
>
> **三重定位**（缺一不可）：
> 1. **策划作品集**——展示资深 3C 管线负责人的决策力：体验出发、业界视野、方案排定；
> 2. **工程落地复盘**——展示真实落地能力：对齐、实现、实验、数据。可信度的来源；
> 3. **3C 管线知识库**——沉淀可复用规律，反哺元规则/反模式，供未来的工作与 agent 检索。
>
> **结构理念（v4.1）**：故事与资料分离。前三组（背景→过程→结果）是有因果链的
> 叙事，必须顺序读；资料组（参数/遗留/沉淀/附录）是查表内容，可折叠。
> 封面 TL;DR 卡服务只有 3 分钟的读者。母版已把这套结构预置好。
> 分组标签用中性工程词（背景/过程/结果/资料），不用"第X幕"等戏剧化命名。

## H5 案例稿指令（原样粘贴给工作会话的 agent）

```
请把本次工作写成一篇 H5 案例稿。

【第一步：复制空白母版】
把 C:\Users\User\3C\templates\案例稿空白母版.html 复制为：
C:\Users\User\3C\knowledge-local\inbox\<今天日期>-<主题短名>.html
母版已预置：精装手册主题、封面 TL;DR 卡与图签、背景/过程/结果/资料四组结构
（两级目录、分组分隔条、资料区折叠）、亮暗切换。
你的工作是【填空】：替换所有〖【占位】〗文本与标题占位，按每个 <section> 内
HTML 注释的指引写正文。不许修改 <style>、脚本与四组分组结构；组内章节可按
案例实际取舍（如没做业界调研就删该节），章节内可自由增设 h3 小节与组件。
图签的文档编号保持 CASE-XX 不动（入库时统一分配）。

【第二步：写正文】
定位：战斗 3C 管线负责人的案例复盘。三类读者：评审同行（看决策过程）、
合作程序（看工程落地）、未来检索知识库的人和 agent（看可复用规律）。
结构母版已定：封面 TL;DR（课题/终案/结果/分工 四行）→ 背景（01 体验目标与
课题）→ 过程（02 业界调研 → 03 方案排定 → 04 实验与落地实录）→ 结果
（05 终案分层 → 06 验收实录）→ 资料（07 关键参数 → 08 遗留与保险丝 →
09 方法论沉淀 → 10 附录，正文包在 details.fold 里）。
各章要求以母版内注释为准，重点：
- 01 开头给 1-2 句项目背景（游戏类型/引擎/系统阶段），别让外部读者猜；
  课题如实定性——矛盾型/建设型(0→1)/打磨型/排查型，没有矛盾不要硬造矛盾
  （硬造戏剧冲突是 AI 味的一种）；本节末尾设「分工与工作方式」小节；
- 03 写决策过程（候选、取舍标准、每步为什么、回退条款），不要只给结论；
- 04 被否掉的方案保留完整篇幅——"为什么它看起来对、实际错在哪"；
- 05 是实验尘埃落定后的静态全貌，不要在它里面剧透或重讲实验；
- 09 每条规律写清适用边界；只收 3C 方法论，工具/流程坑放附录。

【署功与可信度硬规则】（作品集的生命线，违反=返工）
- 分工如实：TL;DR 分工行 + 01 的「分工与工作方式」小节，写清 作者（策划）/
  AI agent / 程序 各自做了什么。代码、审查、检索由 AI 执行的就写 AI 执行、
  作者指挥与验收——工作方式本身是加分项，遮掩才是减分项；
- 验收如实：注明验收人身份与样本量（如"验收人=需求方本人，n=1"）；样本局限
  写进 08 遗留（如"多人盲测未做"）；
- 内部编号（CL 号等）只出现在附录；
- 首次出现的项目专有名词给一句括注解释。

【视觉与出图硬规则】
- 只用母版已有 CSS 变量与组件 class（.rule/.note/.grid/.box/table/tr.group/
  .ok/.no/.later/.tldr/.part/details.fold 等），需要新版式用现有变量组合；
- 禁止新增色值、渐变文字；禁止加载外部资源（字体/脚本/图片 CDN）——单文件
  自包含。指向参考资料的普通超链接允许；
- 示意图优先内联 SVG（class="diagram"），颜色一律取 CSS 变量
  （var(--accent)/var(--brass)/var(--line)/var(--muted)/var(--ink)），
  保证亮暗双主题都成立；几何/分层/时序类内容能画图就画图，别用大段文字硬扛；
- 截图/录屏用本地相对路径，GIF 建议转 MP4 用 <video autoplay loop muted playsinline>。

【文风纪律】（防 AI 味）
- **同一论点全篇只完整陈述一次**：引用原话、形式化表述、举例解释——选一个位置
  落地，其余地方只引用不重述。写完每节自查:"这段有没有在重讲上文已立住的论点";
- 每个决策写"为什么"而不只"是什么"；忠实优先于好看；证伪与失败必须保留；
- 全篇自造概念/金句不超过 2 个，其余用平实表述；加粗每节不超过 3 处；
- 禁营销腔（"我们成功地/完美地"）与整齐排比；结论用数据和原话支撑；
- 记不清的标注"存疑"，不要脑补。

【交稿前自查】（逐条确认后才算完成）
1. 全文搜索「占位」「〖」——必须零残留；除 CASE-XX 外无母版遗留示例文本；
2. TL;DR 四行齐且分工行如实；01 有项目背景与「分工与工作方式」小节；
3. 两级目录锚点与正文 section id 一一对应；资料区 details 折叠可开合；
4. 全文无新增 16 进制色值、无外部资源加载；SVG 填色/描边全用 var(--*)；
5. 验收节写了验收人身份与样本量；CL 等内部编号只在附录。
```

## 入库关（知识库维护会话执行，迭代定稿后）

1. 主题合规检查：无新色值/外部资源加载/禁项；图签、两级目录、折叠、亮暗切换完好；
2. 分配 CASE 编号（替换 CASE-XX，按 docs/ 内既有案例递增）；
3. 术语与既有文档对齐（只改形式不改事实）；抽查署功与验收表述是否如实；
4. 反哺：对照 09「方法论沉淀」，新增/并入 `references/` 的 M/AP 条目，
   案例稿与元规则建双向引用；
5. 归档（**位置由入库关按死规则定，产出方永远只写 inbox，不做目录选择**）：
   - 案例复盘 → `docs/cases/`（CASE 编号即文件名前缀）；
   - 成套专题 → `docs/<专题名>/`（整套一个目录，如 核心战斗手感/）；
   - 单篇散稿 → 按主模块 `docs/camera|skill|character|feedback/`；
   - 「镜头/输入/动作」等模块分类只存在于首页导航层（一篇文档可同时挂多个
     模块入口），不决定物理目录——物理层求 URL 稳定，导航层求好找；
6. 首页/总览加条目 → commit → push → Pages 生效；inbox 迭代稿留档（不入 git）。
