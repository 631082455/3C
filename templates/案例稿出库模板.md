# 出库单 v2.2：工作会话 → 案例稿初稿

> 用途：任何一个 AI 工作会话（不限工具）做完一件有方法论价值的 3C 工作后，
> 把下面的「初稿指令」原样贴给它，让它**直接写出案例稿初稿**。
> 分工按"内容 vs 形式"切：源会话是唯一的一手作者，负责叙事正文（内容零压缩）；
> 知识库维护会话负责编辑装帧（术语统一、M/AP 交叉引用、精装手册 H5、发布），
> 编辑只改形式不改事实。
>
> **三重定位**（缺一不可）：
> 1. **策划作品集**——展示资深 3C 管线负责人的决策力：体验出发、业界视野、方案排定；
> 2. **工程落地复盘**——展示真实落地能力：对齐、实现、实验、数据。这部分是可信度
>    的来源，防止整篇读起来像吹嘘；
> 3. **3C 管线知识库**——沉淀可复用规律，反哺元规则/反模式，供未来的工作与 agent 检索。

---

## 初稿指令（原样粘贴给工作会话的 agent）

```
请把本次工作直接写成一篇「案例稿初稿」，纯 Markdown（不要 HTML/排版），写入：
C:\Users\User\3C\knowledge-local\inbox\<今天日期>-<主题短名>-draft.md

定位：这是一位战斗 3C 管线负责人的案例复盘，三类读者：评审同行（看决策过程）、
合作程序（看工程落地）、未来检索这个知识库的人和 agent（看可复用规律）。
全文要同时回答：基于哪些体验？想解决什么问题？业界怎么做？方案怎么排定？
落地时真实发生了什么？沉淀出了什么可复用的东西？
你是一手作者，按下面的叙事骨架写全文，允许写长：

# 标题（用一句话矛盾或最终原则命名，例：准星永不说谎）

1.【体验目标】我们要玩家感受到什么（用玩家体验语言写，不用技术语言开头），
   以及现状哪里背离了这个体验——最好有玩家/测试者的原始反馈
2.【问题定义】把体验落差拆成明确矛盾：哪两个正确目标在打架、为什么不能全都要、
   解不开会付出什么代价
3.【业界调研】同类问题业界怎么做：逐个案例写"它的做法 → 我们借了什么 /
   为什么排除"。有实机核实的注明核实方式
4.【方案排定】候选路径有哪些、用什么标准取舍（体验优先级/实现成本/风险）、
   每一步为什么这么定。写决策过程，不要只给结论
5.【终案分层】最终方案的分层拆解，每层附一句"它守住了哪条体验"
6.【工程落地实录】真实的落地过程：与程序对齐的接口与约束、实现中暴露的
   新问题及应对、实验与证伪（按时间序：假设 → 验证方式 → 数据/现象 → 结论）。
   被否掉的方案必须保留完整篇幅——"为什么它看起来对、实际错在哪"
7.【关键参数】所有拍板的数字、阈值、曲线设定，逐条附体验依据
8.【验收实录】实测结论与用户/玩家原话（注明是原话）
9.【遗留与保险丝】还挂着的需求、兜底设计、已知妥协
10.【方法论沉淀】从这个案例中提炼的可复用规律：候选的元规则/反模式/检查清单
   条目，每条写清适用边界（什么情况下成立、什么情况下不适用）
11.【附录：未入正文的素材】凡没写进上文但可能有用的细节（数据、对话记录、
   资产/代码路径、中间结论）全部倒进这里，宁滥勿缺。

写作纪律：
- 开篇必须站在玩家体验，不许用系统实现开场；每个决策写"为什么"而不只"是什么"；
- 忠实优先于好看；证伪与失败必须保留，不许只写成功路径；
- 不要营销腔，不要"我们成功地/完美地"，结论用数据和原话支撑；
- 不确定的记忆就标注"存疑"，不要脑补。
```

## 编辑侧流程（知识库维护会话执行）

1. 读 `knowledge-local/inbox/` 中的初稿；
2. 编辑（只改形式不改事实）：术语与既有文档对齐、结构微调、废话收敛；
3. 反哺：对照初稿第 10 节「方法论沉淀」，决定新增/并入 `references/` 的 M/AP 条目，
   并在案例稿与元规则之间建立双向引用；
4. 装帧：套精装手册主题出 H5 进 `docs/`，Markdown 版进 `output/examples/`；
5. 首页/总览目录加条目 → commit → push → Pages 生效；
6. 初稿原件留 inbox 归档（不入 git），作为原始素材可回溯。
