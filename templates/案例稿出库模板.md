# 出库单 v4.2：工作会话直接产 H5 案例稿，维护会话守入库关

> 用途：任何一个 AI 工作会话做完有方法论价值的 3C 工作后，把下面的「H5 案例稿指令」
> 原样贴给它（把文件名占位改成实际日期和主题），让它**以空白母版为基底填空产出
> H5 案例稿**，然后你和它在 H5 上迭代（改内容、出图、定制模块）——迭代发生在
> 有一手上下文的地方。
> 知识库维护会话不参与逐轮迭代，只守**入库关**：主题合规、术语、M/AP 反哺、
> CASE 编号分配、目录、发布。
>
> **三重定位**（缺一不可）：
> 1. **策划作品集**——展示资深 3C 管线负责人的决策力：体验出发、业界视野、方案排定；
> 2. **工程落地复盘**——展示真实落地能力：对齐、实现、实验、数据。可信度的来源；
> 3. **3C 管线知识库**——沉淀可复用规律，反哺元规则/反模式，供未来的工作与 agent 检索。
>
> **结构理念（v4.2）**：故事与资料分离。前三组（背景→过程→结果）是有因果链的
> 叙事，必须顺序读；资料组（参数/遗留/沉淀/附录）是查表内容，可折叠。
> 封面 TL;DR 卡服务只有 3 分钟的读者。母版已把这套结构预置好。
> 分组标签用中性工程词（背景/过程/结果/资料），不用"第X幕"等戏剧化命名。

## H5 案例稿指令（原样粘贴给工作会话的 agent）

```
请把本次工作写成一篇 H5 案例稿。

【第一步：复制空白母版】
把 C:\Users\User\3C\templates\案例稿空白母版.html 复制为：
C:\Users\User\3C\knowledge-local\inbox\<今天日期>-<主题短名>.html
母版已预置：精装手册主题、封面 TL;DR 卡与图签、背景/过程/结果/资料四组结构
（两级目录、分组分隔条、资料区折叠）、亮暗切换。
你的工作是【填空】：替换所有〖【占位】〗文本与标题占位，按每个 <section> 内
HTML 注释的指引写正文。不许修改 <style>、脚本与四组分组结构；组内章节可按
案例实际取舍（如没做业界调研就删该节），章节内可自由增设 h3 小节与组件。
图签的文档编号保持 CASE-XX 不动（入库时统一分配）。

【第二步：写正文】
定位：战斗 3C 管线负责人的案例复盘。三类读者：评审同行（看决策过程）、
合作程序（看工程落地）、未来检索知识库的人和 agent（看可复用规律）。
结构母版已定：封面 TL;DR（课题/终案/结果/分工 四行）→ **导读**（封面下方
独立框：帮助读者快速理解课题、处理方式、结果或使用方法与证据边界；按文档
类型组织，不强制固定段数，不替作者编感受）→ 背景（01 体验目标与课题）→ 过程（02 业界调研 → 03 方案排定 →
04 实验与落地实录）→ 结果（05 终案分层 → 06 验收实录）→ 资料（07 关键
参数 → 08 遗留与保险丝 → 09 方法论沉淀 → 10 附录，正文包在 details.fold 里）。
各章要求以母版内注释为准，重点：
- 01 开头给 1-2 句项目背景（游戏类型/引擎/系统阶段），别让外部读者猜；
  课题如实定性——矛盾型/建设型(0→1)/打磨型/排查型，没有矛盾不要硬造矛盾
  （硬造戏剧冲突是 AI 味的一种）；本节末尾设「分工与工作方式」小节；
- 02 业界调研的**必答题**：业界处理最相近问题的、最普遍的做法是什么？
  我们与它的差异是什么？先锚定最近的已知解，再收集边缘案例。注意检索的
  路灯效应——检索只能找到被写下来的东西，行业常识（如切镜头保瞄准点）
  往往没人写，这类盲区要靠作者补，所以调研谱系成稿后必须请作者过目；
- 终案叙事的调门：如果解法与业界已知模式同源，直接说明同源点、迁移条件和
  项目差异，不写成发明，也不使用“老手艺”“这不是发明”等防御性措辞；
- 03 写决策过程（候选、取舍标准、每步为什么、回退条款），不要只给结论；
- 04 被否掉的方案保留完整篇幅——"为什么它看起来对、实际错在哪"；
- 05 是实验尘埃落定后的静态全貌，不要在它里面剧透或重讲实验；
- 09 每条规律写清适用边界；只收 3C 方法论，工具/流程坑放附录。

【导读生成规范】
导读不是作者自白、故事开场、功劳说明或正文摘要，而是正文入口。读者看完后应能
迅速判断：这篇处理什么问题，为什么值得处理，采用了什么方式，目前做到什么程度，
结论在哪个验证范围内成立。
- 按文档类型组织，不套固定三段式：
  - 案例复盘：体验问题与冲突 → 关键决策/处理方式 → 结果、失败证据与验证边界；
  - 手册/配置说明：适用对象与使用入口 → 推荐的排查/配置顺序 → 证据标签、版本差异与待验证项；
  - 规划/生产规格：规划对象与统计口径 → 目标状态和取舍原则 → 当前成熟度、已完成与未完成；
  - 调试实录：系统原状与故障表现 → 关键修改及其原因 → 实测变化与遗留限制。
- 段数由信息决定，通常 2-4 段；每段只承担一个任务。不得为了形式凑成三段，也不要
  把正文目录压缩复述一遍。
- 第一段直接说问题，不卖关子；第二段只保留决定结果的处理方式；最后一段同时写
  结果与边界。没有独立实验或失败方案时，不要硬凑第二段。
- 不替作者编造心理活动、亲历争论、验收原话或“当时意识到”等故事。只有一手材料
  明确记录的内容，才可写成亲历叙事。
- 不写“这不是发明”“只是老手艺”“最值钱的是”“真正重要的是”等防御性、自评式
  语句。若与业界方案同源，直接说明同源点、迁移条件与项目差异。
- 不以失败做人设或戏剧开场。失败只作为证据：说明为什么看起来可行、实测错在哪里、
  因此排除了什么。
- 导读里的数字必须带口径：对象、单位、统计层级和前后状态。PoseSearch Entry、源 Clip、
  数据库数量、计分点等不得混写；百分比要有分母，耗时要说明测量条件。
- 强结论必须收回到实际验证范围。明确验收人、样本量、环境和未覆盖项；单人 PIE、n=1、
  未做多人/网络/盲测时，不得写成系统已经普遍成立。
- 项目事实、源码/编辑器事实、实测结果与推断分开表达。存疑或口径冲突必须显式写出，
  不得为了让导读顺滑而替正文消除矛盾。

【全文语料生成规范】
- 先列事实台账再写正文，至少分成：已确认事实、实测结果、作者判断、推断、待确认。
  只能润色表达，不能把后四类自动升级为已确认事实。
- 先写普通准确的工程中文，再做语感调整。语料档案用于控制词汇域，不用于机械模仿
  作者聊天口吻，更不能凭几个习惯词虚构作者经历。
- 体验问题用身体动词和对局情境落地；实现问题写清状态、时序、边界和回退条件。
  抽象判断后必须跟项目实例或数字，但不要为了“人味”强塞轶事、吐槽和括号旁白。
- 结论强度跟证据等级一致：配置/源码可证明“机制存在”，单机实测可证明“在该条件下
  成立”，多人、普适性与体验偏好需要相应测试，不能互相代替。
- 数字生成后逐项回到原文、资产或日志核对；同一数字存在两种统计层级时先命名层级，
  不能自行选一个写成总数。工程保护值不得包装为通用常数。
- 业界先例用于定位方案谱系和差异，不用于借权威抬高结论。没有可靠来源时写“待补调研”，
  不用熟悉感补齐游戏名、GDC 观点或行业共识。
- 删除防御性解释、自我评价、格言式结尾和“证明作者厉害”的句子。可信度来自决策链、
  失败证据、数字口径、署功和验证边界。

【署功与可信度硬规则】（作品集的生命线，违反=返工）
- 分工如实：TL;DR 分工行 + 01 的「分工与工作方式」小节，写清 作者（策划）/
  AI agent / 程序 各自做了什么。代码、审查、检索由 AI 执行的就写 AI 执行、
  作者指挥与验收——工作方式本身是加分项，遮掩才是减分项；
- 验收如实：注明验收人身份与样本量（如"验收人=需求方本人，n=1"）；样本局限
  写进 08 遗留（如"多人盲测未做"）；
- 内部编号（CL 号等）只出现在附录；
- 首次出现的项目专有名词给一句括注解释。

【视觉与出图硬规则】
- 只用母版已有 CSS 变量与组件 class（.rule/.note/.aside/.grid/.box/table/
  tr.group/.ok/.no/.later/.tldr/.part/details.fold 等），需要新版式用现有变量组合；
- **并列信息必须卡片化陈列**：目标/约束、两侧代价、方案对比这类并列事项用
  `.grid+.box`（每卡一个 h3 短标题），禁止写成连排长段落；公式、关键数字、
  核心参数上封面（TL;DR 行或 chips），别埋在正文里；
- 母版排版是 v4 节卡片式（每节独立卡浮于暖纸渐变背景、篇章浮标签、28px 节标题），
  这些由母版样式自动提供，agent 不要自行改动；
- 衍生思考用 `.aside`（◇ 题外话旁注）就近安置——主线不被污染，也不必全塞附录；
- 禁止新增色值、渐变文字；禁止加载外部资源（字体/脚本/图片 CDN）——单文件
  自包含。指向参考资料的普通超链接允许；
- 示意图优先内联 SVG（class="diagram"），颜色一律取 CSS 变量
  （var(--accent)/var(--brass)/var(--line)/var(--muted)/var(--ink)），
  保证亮暗双主题都成立；几何/分层/时序类内容能画图就画图，别用大段文字硬扛；
- 截图/录屏用本地相对路径，GIF 建议转 MP4 用 <video autoplay loop muted playsinline>。

【文风与语料纪律】
- **写作前必须加载两份规范**：`conventions/作者语料档案.md`（限定词汇域，不是模仿
  聊天口吻）与 `conventions/行文规范.md`（符号层/句式层/结构层的完整规则）。若两份
  旧规范与本出库单冲突，以本出库单的事实、证据和导读规则为准，并在入库时回修旧规范；
- **事实台账先于成文**：动笔前列出已确认事实、实测结果、作者判断、推断、待确认，
  导读、TL;DR、正文和结论都只能使用对应强度的表述；
- 内容层硬规则（防冗余，编辑实战沉淀）：**同一论点全篇只完整陈述一次**；
  引语不翻译（引完不许白话复述）；图表不重讲；跨节不重讲（机制只在一节
  完整出现，其余指路）；事实说完就停（禁抒情收尾与自我评价从句）；
- 每个决策写“为什么”而不只“是什么”；忠实优先于好看；证伪与失败必须保留；
  记不清的标注“存疑”，不要脑补；
- 交稿自查除行文规范的“数、读、比”外，增加“核、分、收”：核对所有数字来源与
  统计口径；分开事实/实测/判断/推断；把结论收回到实际样本和测试环境。

【交稿前自查】（逐条确认后才算完成）
1. 全文搜索「占位」「〖」——必须零残留；除 CASE-XX 外无母版遗留示例文本；
2. TL;DR 四行齐且分工行如实；01 有项目背景与「分工与工作方式」小节；
3. 导读名称正确，读完能回答“问题/方式/进度/边界”；按文档类型组织而非固定三段；
   无虚构心理活动、事故故事、防御性自贬、自我评价或格言式结尾；
4. 建立并复核事实台账；项目事实、源码/编辑器事实、实测结果、作者判断、推断和待确认
   没有混写；存疑内容未被润色成定论；
5. 逐项核对数字的对象、单位、统计层级、前后状态与来源；Entry/Clip/数据库/计分点等
   名词没有混用；百分比有分母，耗时有测试条件，保护值未写成通用常数；
6. 验收节写了验收人身份、样本量、测试环境与未覆盖项；结论强度未超过证据等级，
   CL 等内部编号只在附录；
7. 两级目录锚点与正文 section id 一一对应；资料区 details 折叠可开合；
8. 全文无新增 16 进制色值、无外部资源加载；SVG 填色/描边全用 var(--*)；
9. **冗余终查**（逐节过一遍）：引语后无白话复述；图表内容正文无重讲；
   同一机制只在一节完整出现；无抒情收尾与自我评价从句；
10. 执行“数、读、比、核、分、收”：数符号禁项，读关键段落，比段落节奏，核数字
    来源，分证据类型，收结论边界。
```

## 入库关（知识库维护会话执行，迭代定稿后）

0. **作者两道人工关**（发布前置条件）：①亲笔位——提醒作者花 10 分钟亲笔
   改写标题、开篇首段、各节首句、结论（读者对"是不是 AI 写的"的判断八成
   形成于这些位置）；②内行嗅觉关——调研谱系逐行过作者：最普遍的同类解
   漏没漏？（CASE-01 的教训：检索找不到行业常识，漏掉的恰是最近的先例）；
1. 主题合规检查：无新色值/外部资源加载/禁项；图签、两级目录、折叠、亮暗切换完好；
2. 分配 CASE 编号（替换 CASE-XX，按 docs/ 内既有案例递增）；
3. 术语与既有文档对齐（只改形式不改事实）；抽查署功与验收表述是否如实；
4. 反哺：对照 09「方法论沉淀」，新增/并入 `references/` 的 M/AP 条目，
   案例稿与元规则建双向引用；
5. 归档（**位置由入库关按死规则定，产出方永远只写 inbox，不做目录选择**）：
   - docs/ 顶层锁死两个夹子，永不新增：`cases/`（类型）与 `topics/`（主题）；
   - 案例复盘 → `docs/cases/`（CASE 编号即文件名前缀）；
   - 其余一切 → `docs/topics/<专题名>/`（整套一个目录；单篇散稿归入最相关的
     专题夹，没有就为它开新专题夹——一篇也算专题的开端）；
   - 「镜头/输入/动作」等模块分类只存在于首页导航层（一篇文档可同时挂多个
     模块入口），不决定物理目录——物理层求 URL 稳定，导航层求好找；
6. **修订记录**：在附录末尾生成/更新「修订记录」表（日期 + 版本 + 变更摘要，
   从 git log 与迭代过程提炼）——活文档的证据，每次后续修订都要追加；
7. 首页/总览加条目 → commit → push → Pages 生效；
8. **收尾清场**：迭代稿与全部过程材料（素材目录、修订指令等）移入
   `knowledge-local/archive/<日期-主题>/` 一批一夹留底。**inbox 只住"迭代中"
   的稿**——里面每一项都代表一件没做完的事，做完就得搬走。
