# 操控与输入（Control）

覆盖：输入响应链、缓冲与取消、施法方式、预览/执行分离、目标选择、多端适配。

## 1. 输入响应链：延迟预算

从按键到画面反馈的每一环都有延迟，先建预算表再谈手感：

```
硬件输入 → 引擎输入采样 → 游戏逻辑判定（含缓冲/取消规则）→ （网络预测启动）→ 动画/表现首帧
```

- 竞技向目标：按键到**可感知反馈首帧** ≤ 100ms（本地）；其中游戏逻辑层自己引入的延迟应 ≤ 1~2 帧。
- **反馈首帧 ≠ 判定帧**：反馈可以先行（起手光效、音效），判定按设计的前摇走。玩家感知的"跟手"主要来自反馈首帧。

## 2. 输入缓冲与取消窗口

三件套决定连招手感，必须全局统一规则、逐技能可覆盖：

| 机制 | 定义 | 设计要点 |
|---|---|---|
| 输入缓冲 | 当前动作未结束时提前按下一动作，系统记住并在可执行时立即执行 | 全局统一时长（常见 150~300ms）；队列深度通常 1（记最后一个）；缓冲的输入要带方向快照还是执行时重采样？——明确定义，位移技能尤其敏感 |
| 后摇取消 | 动作的收招段可被哪些动作提前打断 | 用取消表（动作 × 可取消它的动作类别）而不是逐对配置；移动取消后摇是"跟手感"的最大来源 |
| 前摇锁定 | 判定生效前的不可取消段 | 竞技公平性参数，改动即平衡改动，参数表中标红 |

## 3. 施法方式与预览/执行

带目标/落点的技能推荐统一为**预览态 → 执行态**两段式框架：

- **预览态**：显示指示器（范围/落点/轨迹），可取消，不消耗资源。
- **执行态**：锁定参数快照，进入前后摇，走判定。
- 两态**共用同一份参数源**（M8）：射程、宽度、落点合法性规则完全一致。指示器显示 8m 而实际判定 7.5m 的方案直接打回。
- 快速施法 = 跳过预览态的显示但**不跳过其参数计算**；智能施法的方向来源（指针位置）在按下瞬间快照。
- 预览期间的移动/转向规则要显式定义（能移动吗？角色是否随指示器转向？）。

## 4. 目标选择

- **锁定 vs 非锁定**先定框架；混合框架（软锁定辅助 + 手动瞄准）要定义辅助的介入曲线（角度容差、距离权重、切换粘滞）。
- 目标切换的粘滞（hysteresis）：候选目标优于当前目标一定阈值才切换，否则边界抖动。
- 目标死亡/离开视野的降级规则：清空 / 自动换最近 / 保持最后方向。**必须定义**，这是高频"打空气"投诉源。
- 辅助瞄准强度手柄 > 键鼠，参数按设备分开配（见 §6）。

## 5. 控制权剥夺与归还

击飞/眩晕/强制位移期间的输入处理，逐条定义（M2）：

- 被控期间的输入是**丢弃**还是**缓冲**？（推荐：缓冲最后一个方向输入 + 最后一个技能输入，控制结束立即执行——受控结束瞬间的响应感非常值钱）
- 被控打断施法的规则表：哪些控制打断哪些施法段（前摇/引导/后摇分开定义）。
- **自我施加的控制不得打断自身来源技能**（M5）——白名单机制，新技能 checklist 项。

## 6. 多端适配

- 参数按设备档分开：缓冲时长、辅助瞄准强度、死区、灵敏度曲线，键鼠/手柄/触屏各一套，共享机制逻辑。
- 手柄摇杆：内死区（防漂移）+ 外死区（保证能到满速）+ 响应曲线（推荐可选线性/平方）；八向意图明显的操作加角度吸附。
- 方案评审必答题："这套操作在手柄上同样成立吗？"（来自约定的评审关注点时优先准备）。

## 7. 常见操控问题 → 排查方向

| 症状 | 优先排查 |
|---|---|
| "不跟手" | 反馈首帧延迟预算超标；过渡动画阻塞输入；缓冲时长过短或队列被清 |
| "吞技能" | 缓冲队列深度/清空时机；被控期间输入被丢弃；取消表漏配 |
| 指示器与实际不符 | 预览/执行参数源分裂（M8） |
| 偶现方向不对 | 方向快照时机不一致（按下时 vs 执行时）；缓冲输入未带方向快照 |
| 受控结束后卡一下 | 控制归还协议缺失（M2）：输入缓冲在受控期间被清空 |
