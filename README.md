# Keke 的战斗 3C 工作站（Keke's Combat 3C Workstation）

> ### 📖 [在线阅读文档站 → 631082455.github.io/3C](https://631082455.github.io/3C/)
> HTML 文档请从上面的文档站打开（渲染版）。在 GitHub 里直接点击 `.html` 文件看到的是源码，不是网页。

这不是一套输入需求就能自动生成优秀 3C 方案的万能模板，而是一个服务真实 3C 团队与 AI 工作流的知识工作站。它保存项目上下文、判断标准、方案收敛过程、验证证据与失败记录，让下一次任务不必从零开始。

3C 最终呈现的是输入、移动、镜头与动画共同形成的体验。Agent 可以帮助检索、组织上下文、搭建设计框架和执行检查，但方案是否符合战斗 Pillar、是否能让玩家保持心流，最终仍需要人上手裁决。

因此，这里不只保留答案，也记录答案从哪里来、在哪些条件下成立，以及哪些尝试没有奏效。

> **具体方案与参数通常不能原样迁移；真正可复用的是建立上下文、沉淀判断并持续验证的方法。**

## 这个工作站保存什么

- **项目上下文与约定**：视角、输入、网络模型、引擎形态和体验基调，先让 Agent 知道问题发生在哪里。
- **设计稿与案例复盘**：不只给终案，也保留问题、选型、调试过程、失败路径、验收结果和证据边界。
- **元规则与检查框架**：把反复验证过的判断压缩成可检索、可复用的规则，用于生命周期、冲突仲裁、参数化和质量审计。
- **Agent 索引与执行入口**：让 Agent 按任务读取必要上下文，而不是把整个知识库一次性塞进 Prompt。
- **团队记忆**：同时写给当前执行任务的人、相关上下游，以及半年后已经忘记细节的自己。

## AI 与人的分工

| Agent 更适合做 | 人必须负责 |
|---|---|
| 检索相关规则、案例与历史证据 | 判断是否符合战斗 Pillar |
| 组织上下文，补齐设计结构和检查清单 | 上手感受节奏、心流与操作负担 |
| 参数化方案，检查生命周期和系统冲突 | 在多个可行方案之间做体验取舍 |
| 执行批量整理、复查和文档维护 | 对最终体验与上线结果负责 |

这里使用 AI，不是为了把体验判断外包，而是为了减少上下文重建、遗漏检查和重复劳动。

## 哪些东西值得复用

### 可以直接复用

- 怎么为团队和自己搭建一套可持续积累的 3C 工作站。
- 怎么组织项目约定、索引层级、元规则、设计模板和验收证据。
- 怎么让 Agent 只读取当前任务真正需要的上下文。
- 怎么记录一个具体问题在项目中逐步收敛的过程。

### 可以借鉴，但需要重新判断

- 半身分层、动画叠加、Overlay、Motion Matching 等项目实践。
- 输入缓存、释放时序、目标选择、镜头动效与冲突仲裁等系统架构。
- 案例中的排查路径、验证方法和失败经验。

### 不应直接照搬

- 具体参数、阈值和预设档位。
- 依赖特定英雄、动画资产、网络模型或引擎实现的方案。
- 脱离原项目上下文后的“最佳实践”结论。

换一个项目，原来的平衡点通常就不存在了。行业里并不缺成熟方法和公开案例，真正稀缺的是这些方法如何在一个具体项目中被取舍、验证和修正。

## 怎么使用

### 作为读者

直接进入[在线文档站](https://631082455.github.io/3C/)。每篇文档开头会先说明问题、处理方式、当前进度和验证边界；不需要从头通读整个站，对哪部分感兴趣，再顺着证据和实录往下看。

### 作为 Agent

1. 先读 [`SKILL.md`](SKILL.md)，确认任务流程和知识索引。
2. 再读 `conventions/` 中当前项目的约定，避免脱离视角、输入、网络与引擎语境。
3. 按 `SKILL.md` 的路由只加载本次任务相关的 `references/`、模板、设计稿和案例。
4. Agent 负责检索、结构化、执行与复查；体验目标、方案取舍和最终验收仍由人决定。

不同 AI 工具的接入方式见 [`USAGE.md`](USAGE.md)。六阶段设计流程仍然有效，但它是工作站中的执行框架，不是“一句话生成好手感”的承诺。

## 目录结构

```text
3C/
├─ index.html                  # 在线文档站首页与总目录
├─ docs/
│  ├─ topics/                 # 按 Movement、Control、Camera、Animation 等主题整理的设计文档
│  └─ cases/                  # 从问题到验证边界的完整案例复盘
├─ SKILL.md                   # Agent 执行入口：流程、约束与知识路由
├─ USAGE.md                   # 在不同 AI 工具中接入本工作站的方法
├─ conventions/               # 项目上下文与 3C 约定
├─ references/                # 元规则、反模式、质量标尺与领域参考
├─ templates/                 # 设计稿、参数表和案例稿模板
├─ output/                    # 工作流产出与样板
└─ knowledge-local/           # 本地事实底稿与私有资料，不进入公开仓库
```

## 公开边界

本站公开内容均经过脱敏，只保留个人方法论、设计框架、案例复盘和必要证据，不包含项目资产原件、未公开玩法与数值。`knowledge-local/` 已被 Git 忽略，用于保存不能公开的事实底稿；内部资料不要复制到公开目录。
